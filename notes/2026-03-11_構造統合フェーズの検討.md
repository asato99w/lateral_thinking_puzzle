# 構造統合フェーズの検討

## 背景

turtle_soup 05 のブラッシュアップ第2反復で、C3/C4 の統合（分岐合流構造）を行った。この作業の中で、現行アルゴリズムに統合的な構造検討を行うフェーズが欠けていることが示唆された。

## 現行アルゴリズムの構造的な制約

### 1. T 方向のみの連鎖

Phase 3（戦術連鎖）は S から T への経路に沿って連鎖を構築する。各分割命題の真側（T 側）の道しか設計されず、偽側の道は存在しない。

turtle_soup 05 の分割階層:

```
P1=真 {T,M1,M2,M3,M4,M8,M12}  ← 既存の連鎖あり（C1→C2→C3）
  P2=真 {T,M1,M4}
    P3=真 {T,M1}
      P4=真 {T}
      P4=偽 {M1}
    P3=偽 {M4}
  P2=偽 {M2,M3,M8,M12}         ← 連鎖なし・分割未展開
P1=偽 {M5,M6,M7,M9,M10,M11}   ← 連鎖なし・分割未展開
```

### 2. 導出経路の単一性

Phase 3 が各命題に対して1つの連鎖（1つの導出経路）を設計するため、全命題の fc が単一の AND グループになる。fc の OR 構造（代替導出経路）は意図的に Phase 3 時点では使用しない（統合アルゴリズムに明記済み）。

## 次のフェーズで必要な作業

### A. 偽側の枝の構築（連鎖の外部）

各分割命題の偽側に、分岐点から棄却命題へ至る連鎖を構築する。

- **P1=偽側**: {M5,M6,M7,M9,M10,M11} の方向を探索する枝。内部の詳細な分割は必須ではないが、この方向を探索して行き止まりに至り、P1=真に収束する道が必要
- **P2=偽側**: {M2,M3,M8,M12} の方向を探索する枝。同様

枝の長さ・深さは難易度設計のパラメータ。短ければすぐに T 方向に収束し簡単になり、長く深ければ多くの探索を経験する。アルゴリズムとしては枝を構築する仕組みを用意し、長さは設計判断として残す。

### B. 既存連鎖の内部の分岐

既存の連鎖（C1, C2, C3）の各ステップで、反対の回答に進む枝を設計する。既存のノードから分岐するため、新たな連鎖の構築は不要。

### C. 統合による合流

A, B の設計過程で、異なる枝が同じ命題に合流する構造を意識する。

- 偽側の枝の中間命題が真側の既存命題と合流できる可能性がある
- 合流が生じれば、その命題の fc に自然に複数の AND グループ（代替導出経路）が生まれる
- 偽側の枝の設計と代替導出経路の設計が同時に進む

## 早いフェーズの分析の位置付け

### 棄却分析（Phase 3）の未活用

現行の Phase 3（棄却分析）は以下のデータを生成する:

- `rejection_map`: 各モデルの偽命題と最小棄却条件
- `rejection_power`: 各命題の棄却力（棄却するモデル数）
- `rejection_progression`: 棄却の進行順序

しかし、これらは後続の Phase 4〜6（戦術連鎖・質問生成・ブラッシュアップ）で**一切参照されていない**。Phase 5（検証）で rejection_progression を表示するが、連鎖構築へのフィードバックはない。

棄却分析が実際に意味を持つのは、「偽側の枝をどこまで深くするか」「どの分岐点でどのモデルが棄却されるか」を設計するとき、つまり分岐連鎖構築フェーズである。T 方向の連鎖構築には不要な分析を早いフェーズで行っている状態。

### 命題レベルの棄却条件

命題の rejection_conditions も同様。T 方向のみの構築段階では偽側の命題が存在しないため、棄却条件は使いようがない。偽側の命題が導入される分岐連鎖構築フェーズで初めて必要になる。

### 示唆

棄却分析全体（モデルレベル・命題レベル）を分岐連鎖構築フェーズに移動する方が、アルゴリズムの各フェーズの役割が明確になる可能性がある。早いフェーズ（Phase 1〜3）は T の構造分解とモデル分割に専念し、棄却の設計は分岐構築と一体で行う。

## 設計上の注意

- 代替導出経路（fc の OR 構造）の設計は、偽側の枝の構築と合わせて行う方が自然。独立したフェーズとしてさらに後に回す必要はない
- 連鎖の分岐合流の判例は `precedents/戦術連鎖の構造の判例.md` として作成済み
