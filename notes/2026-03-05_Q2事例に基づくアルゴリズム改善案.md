# Q-2 事例に基づくアルゴリズム改善案

## 日付
2026-03-05

## 経緯

turtle_soup 03 の Q-2（「男は過去に危機的な体験をしましたか？」）の再設計過程で、以下の構造的問題が発見された:

1. Q-2 の recall を自然にするにはキーピース方向の知識が必要
2. キーピース方向の知識がある段階ではQ-2自体が不要
3. D-16（危機的体験）は D-6（遭難）から導出可能であり、独立質問が不要
4. D-15 への到達パスの再構成で T の不足が判明（船乗り vs 乗客）

これらは個別の修正で対処したが、同種の問題を早期に検出・対処するためのアルゴリズム改善案を以下にまとめる。

## 改善案

### 改善案 1: 検査（Phase 7）に recall のキーピース近接性検査を追加

#### 問題
Q-2 の recall を自然にするには X-3（「自殺するほどショックを受ける事実を男が知った」）のようなキーピース周辺の記述素が前提として必要だった。これは recall_conditions の調整では解決できない構造的問題であり、質問の存在自体に問題があることを示す。

#### 提案
検査 Step 5（仮称）として以下の構造的チェックを追加する:

> 各質問について、recall_conditions を認知的に自然にするために追加が必要な記述素を仮定し、その記述素がキーピースの members またはその周辺に位置するかを検査する。

具体的な判定基準:
- recall の記述素から質問の方向（reveals の記述素が属するピース）へ自然に至るために、**キーピースの members または depends_on 先のピースの members** が必要になる場合、構造的近接性の疑いとして報告する

この検査は Step 3（方向性漏洩の構造的チェック）と同様に、認知的判断に依存しない構造的チェックとして機能する。

#### 期待される効果
Q-2 のようなケース（recall がキーピース方向の知識を前提とする質問）を、認知的自然さの判断なしに構造的に検出できる。

#### 課題
- 「recall を自然にするために追加が必要な記述素を仮定し」の部分は厳密に形式化しにくい。LLM による認知的判断の限界（別ノート参照）との兼ね合いで、どこまで構造的に定式化できるかは検討が必要
- 代替案として、より機械的な近似指標を検討する余地がある（例: recall の記述素と reveals の記述素の間のピース距離）

---

### 改善案 2: 調整適用（Phase 8）にパターン 6 を追加 — 質問の破棄と経路再構成

#### 問題
現行のパターン 1〜5 は全て「質問を維持したまま修正する」方向の調整であり、「質問自体を破棄する」パターンが明示されていない。Q-2 の事例では、recall_conditions の改善（パターン 5）では解決できず、質問の破棄と到達パスの再構成が必要だった。

#### 提案
パターン 6: **質問の破棄と経路再構成**

適用条件:
- 検査で構造的問題（改善案 1 の検査等）が報告された質問
- パターン 5 の対応 A・B のいずれでも認知的経路を確保できない場合

手順:
1. 破棄する質問が reveals する記述素 D について、他の記述素からの formation_conditions による導出が可能かを検討する
2. 導出可能な場合、D に formation_conditions を追加し、D を末端記述素から除外する
3. D に依存していた戦術連鎖の起点・経路を再構成する
4. 再構成により新たな末端記述素が必要になる場合は追加する

Q-2 での実例:
```
破棄: Q-2 (reveals: D-16)
導出化: D-16 ← formation [["D-6"]]（遭難 → 危機的体験）
経路再構成: D-15 への 3 段階連鎖（職業 → 乗り物の職業 → 船乗り）
新規記述素: D-23, D-24
```

#### 期待される効果
「修正不能な質問」に対する正式な対処パターンが確立される。

---

### 改善案 3: Phase 3（仮説連鎖）に導出可能性の事前検査を追加

#### 問題
D-16（危機的体験）は D-6（遭難）から自明に導出可能だったが、Phase 3 の時点では検出されなかった。Phase 3 はトリガー記述素の横展開に焦点を当てており、「既存の記述素が他の記述素から導出可能か」の逆方向の検査を行っていない。

#### 提案
Phase 3 の Step 2（末端記述素の確認）の後に、以下の検査を追加する:

> 末端記述素のうち、ピースメンバーの記述素から自明に導出可能なものがないかを検査する。導出可能な場合、formation_conditions の追加を検討する。

判定基準:
- ピースメンバー（特にトリガー近傍）の記述素を前提として、末端記述素が論理的帰結として導かれるか
- 「自明」の基準: 追加の仮定や推論なしに、前提の記述素から直接導かれること

Q-2 での実例:
- D-6（遭難した）を知っていれば D-16（危機的体験があった）は自明
- Phase 3 時点で D-16 に formation_conditions [["D-6"]] を設定し、末端記述素から除外できた可能性がある

#### 期待される効果
不要な末端記述素（=不要な戦術連鎖・質問）の生成を予防する。

#### 注意
- Phase 3 時点ではピース内部の連鎖（Phase 4）がまだ存在しないため、ピースメンバーの到達順は不明。あくまで「到達しうる記述素からの導出可能性」の検査であり、到達順序の考慮は Phase 4 以降に委ねる
- 過剰な導出化は質問数の減少によりゲーム体験を損なうため、自明な場合に限定する

---

### 改善案 4: Phase 4（戦術連鎖）に T の十分性検査を追加

#### 問題
D-15 への新パス（人物像の探索 → 段階的絞り込み → 船乗り）を設計した際、T に「男は船乗りで」が明示されていないことが判明した。T に「乗っていた船」という記述はあったが、職業としての船乗りか乗客かは不明で、「職業は関係しますか？」に客観的に Yes と回答できなかった。

#### 提案
Phase 4 の連鎖構成時に、以下を検査する:

> 戦術連鎖の各ステップで生成・到達する記述素が、T から客観的に Yes/No で回答可能かを確認する。曖昧性がある場合、T の補足が必要なケースとしてフラグを立てる。

Q-2 での実例:
- 「船乗りか？」→ T は「乗っていた船」としか述べておらず、職業かどうか不明
- T に「男は船乗りで」を追加して解決

#### 期待される効果
戦術連鎖の設計段階で T の不足を検出し、Phase 5（質問生成）での客観性問題を予防する。

#### 注意
- T の変更は例外的措置であり、自動的に行うものではない。検査はフラグを立てるに留め、T の変更判断は人間に委ねる
- Phase 5 の「質問の客観性」規則が既にこの問題をカバーするが、Phase 4 の時点で検出する方が手戻りが少ない

---

## 改善案間の関係

```
Phase 3（改善案 3: 導出可能性の事前検査）
  → 不要な末端記述素を早期に除去
  → 不要な戦術連鎖・質問の生成を予防

Phase 4（改善案 4: Tの十分性検査）
  → 連鎖設計時にTの不足を検出
  → 質問の客観性問題を早期に発見

Phase 7（改善案 1: recall のキーピース近接性検査）
  → 構造的に問題のある質問を検出
  → 改善案 2 の適用トリガーとなる

Phase 8（改善案 2: 質問の破棄と経路再構成）
  → 改善案 1 で検出された問題への対処パターン
```

改善案 3 は予防的（問題の発生を防ぐ）、改善案 1・2 は検出・対処的（発生した問題を見つけて直す）。両方が揃うことで、Phase 3 で見逃した場合でも Phase 7-8 で検出・対処できる。

## 適用の優先度

1. **改善案 2**（調整パターン追加）: 現時点で最も実用的。Q-2 事例で実際に使用したパターンの形式化
2. **改善案 1**（検査の追加）: 構造的チェックとして定式化できれば有用だが、判定基準の厳密化が課題
3. **改善案 3**（導出可能性の検査）: Phase 3 への追加は影響範囲が狭く安全
4. **改善案 4**（Tの十分性検査）: 有用だが発生頻度は低い。記録に留めて必要時に適用でもよい
