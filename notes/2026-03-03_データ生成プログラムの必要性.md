# データ生成プログラムの必要性

## 発見された問題

turtle_soup 02 のデータ生成過程で、戦術連鎖（Phase 5）から質問生成（Phase 6）への変換時にズレが生じた。

具体的には:
- 戦術連鎖では `{Pt-4, Pt-5, Ps-3} → Pt-7` のように、異なるピースの事実を組み合わせて次の事実に到達する接続が記述されていた
- しかし質問生成の段階で、この接続が想起条件として形式化されず、代わりにトリガー仮説 H-forced-soup が Q-13〜Q-16 の想起条件として直接使われた
- その結果、クリア条件に関わる質問（Q-14, Q-15）の想起条件が、最終ピース P-realization に属さない仮説に依存する構造になった

## 根本原因

自然言語で記述されたアルゴリズム（理論）を、人間（またはAI）が解釈して適用する過程で、意図が正しく反映されない。

戦術連鎖の段階では事実間の接続を記述しているが、それを質問の recall_conditions に変換する明確な規則がない。この変換が暗黙的であるため、解釈のブレが発生する。

## 戦術連鎖における派生的事実の問題

現在の戦術連鎖は「既知の事実/仮説 → 戦術 → 仮説」の形で記述されているが、連鎖の中間で生成される事実の扱いが形式化されていない。

- T の事実の中には、異なるピースの事実群を結びつける役割を持つものがある（例: Pt-7「味が記憶と違った」は P-deception の事実と S の事実を繋ぐ）
- こうした事実の位置づけや、連鎖における役割が明文化されていない

## 提案: データ生成用プログラムの構築

エンジン（プレイヤー向け）とは別に、データ生成過程を形式化したプログラムを構築する。

### 目的

- 戦術連鎖・仮説連鎖のロジックをコードとして実装し、解釈のブレを排除する
- 生成過程の各ステップを検証可能にする
- 「戦術連鎖の出力 → 質問の recall_conditions」の変換を明示的な規則として実装する

### エンジンとの違い

| | エンジン | データ生成プログラム |
|---|---------|-----------------|
| ユーザー | プレイヤー | パズル設計者（またはAI） |
| 入力 | プレイヤーの質問選択 | S, T, 理論 |
| 出力 | ゲーム状態の遷移 | data.json |
| 関心 | 状態管理・判定 | 連鎖の形式的構成・整合性保証 |

### 形式化すべき対象

- 戦術連鎖: 事実集合 → 戦術 → 事実集合 の適用と、中間事実の管理
- 仮説連鎖: 仮説の形成条件の構成と、ピース間の依存関係の検証
- 質問生成: 戦術連鎖の接続から recall_conditions への変換規則

## 未決事項

- 派生的事実の扱い: T に含まれる事実を連鎖の中で配置するのか、T 外の新しい事実を導入するのか
- プログラムの粒度: 完全自動生成か、対話的な生成支援か
- 既存の eval スクリプトとの関係: 生成プログラムが整合性を保証するなら eval の役割が変わる
