# 仮説によるオープン制御の検討

## 背景

v2 POC の初版では、質問のオープン制御を **事実ベースの preconditions 充足判定** で行っている。

```
質問がオープン ⟺ preconditions の全事実が観測済み
```

これにより動く骨格はできたが、以下の問題がある：

- 戦略が質問に貼られたラベルに過ぎず、エンジンロジックに関与していない
- 事実の充足だけでは、質問の開放タイミングが不自然になるケースがある
  - 例: `q_a01: 男の過去に危険な出来事があった？` の preconditions が `[Ps-1]`（男がレストランに入った）のみ → ゲーム開始直後からオープンになり唐突

## 議論の経緯

### 戦略の位置づけ

戦略（汎用戦略）とは：
- 任意のパズルで適用可能なもの
- 質問の集合自体は所与（パズルデータとして事前定義）

検討の結果、**戦略は質問構成を設計する際に用いるものであり、オープン制御とは別の関心事** という認識に至った。

### 事実ベースの限界

理想的なゲーム展開では、キーとなる質問（例: 過去の危険な出来事）はいきなりオープンにならず、**探索の結果として現れてほしい**。

しかし事実ベースだけでは：
- 排除質問（「毒？」→ No、「病気？」→ No）が reveals=[] で空振りになる
- 排除の蓄積が次の質問を開放する仕組みがない
- 「常識では説明できない → 何か特殊な背景があるはず」という推論の結果を表現できない

### 仮説の導入

事実と質問の間に **仮説** という中間層を導入する。

```
事実 → 仮説の形成 → 質問のオープン
```

仮説はピース（T を構成する真実の断片）とは全く別の概念：
- **ピース**: 解くべき対象。T の構造
- **仮説**: プレイヤーの推論状態。探索を駆動するもの

仮説は最終的に事実集合に帰着するが、中間層として置くことで：
- 同じ仮説に複数の経路（異なる事実の組み合わせ）から到達できることを自然に表現できる
- 質問のオープン条件が仮説 ID で書けて、事実→仮説のマッピングを別に持てる
- パズル設計時の思考の道具としても機能する

現時点では便宜的な道具という認識だが、ドメインモデルには組み込む。

## 結論: 採用する構成

```
事実集合の族 → 仮説 → 質問群
(複数の経路)    (1つ)   (複数の質問)
```

- 仮説 h に対して、それを形成する事実集合が複数ありうる（族 = OR of AND）
- どの経路でも h が成立すれば、h に紐づく質問群がオープンになる
- 1つの仮説が複数の質問を開放しうる

### ドメインモデル

```python
@dataclass
class Hypothesis:
    id: str
    label: str
    formation_conditions: list[list[str]]  # 事実IDの集合の族（OR of AND）
    opens: list[str]                        # この仮説がオープンする質問ID群
```

### 海亀での例

```
h4「この男には特殊な背景がある」
  formation_conditions:
    - [排除事実1, 排除事実2, 排除事実3]   # 経路A: 排除を3つ
    - [排除事実1, 排除事実2]              # 経路B: 2つでも十分
    - ...
  opens: [q_a01, ...]
```

## ネクストステップ

この構成で、海亀パズルの**完全に合理的なパス**を構成する。
- 全ての質問に対して、それがオープンになる仮説を定義する
- 全ての仮説に対して、形成条件（事実集合の族）を定義する
- 複数の経路で T に到達できることを確認する
