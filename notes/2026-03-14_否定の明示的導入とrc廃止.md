# 否定の明示的導入と recall_conditions の廃止

## 背景

### recall_conditions の問題

recall_conditions（rc）は質問の利用可能条件として定義されていたが、これは reveals 先の命題の formation_conditions と常に一致するものとして運用されていた。しかし rc を独立概念として扱った結果、以下の問題が発生した:

- Phase 8 で Q4a の rc に合わせて P2.fc を操作（依存方向の逆転: 命題→質問が正しいのに、質問→命題の方向で fc を変更）
- eval スクリプト（check_recall_fc）を修正して辻褄を合わせる事態

根本原因: 質問は reveals によって形式化される。rc は独立した概念ではない。

### 否定の欠如

現状、N 系命題（N1a, NF1a 等）は意味的には他の仮説の否定だが、「何の否定か」という構造が暗黙的。特に「いいえ」で答える質問が何を問うているかが構造上不明確。

## 設計

### 命題間の否定関係

命題に `negation_of` フィールド（nullable、命題 ID）を追加する。

- P と ¬P が互いの否定である場合、双方が `negation_of` で相手を指す
- 否定関係は T における真偽とは独立な論理的関係
- 全命題が原理上否定を持つが、データ上で相手が不要なら null

```
Proposition:
  id: str
  label: str
  negation_of: str | null       # ← 新規
  formation_conditions: ...
  entailment_conditions: ...
  rejection_conditions: ...
```

### 質問の利用可能条件

質問は reveals のみで形式化される（この原則は変わらない）。

- 「はい」で答える質問: reveals 先の命題 P を confirm。利用可能条件 = P.fc
- 「いいえ」で答える質問: reveals 先の命題 ¬P を confirm。利用可能条件 = P.fc（¬P.negation_of が指す命題の fc）

「いいえ」の質問は P について問い、答えが「いいえ」なので ¬P が確立される。質問がいつ問えるかは P がいつ仮説として形成可能かで決まる。

**¬P.fc と P.fc は独立した値であり、一致する必然性はない。** ¬P.fc は ¬P 自身が仮説として形成される条件であり、P.fc とは別の概念。

### recall_conditions の廃止

rc は不要になる。質問の利用可能条件は:
- reveals → 命題 → （negation_of があれば）否定対象の命題 → fc

という経路で導出される。質問に独立した利用可能条件フィールドは不要。

## 影響範囲

### データ構造 (models.py)

- `Proposition` に `negation_of: str | None` を追加
- `Question` の docstring から rc 言及を除去（既に recall_conditions フィールドは存在しない）

### エンジン (engine.py)

質問の利用可能条件を導出する関数の修正。判別条件は **質問の回答** であり、命題が `negation_of` を持つかどうかではない。`negation_of` は対称的な関係（P も ¬P も互いに持ちうる）であるため、フィールドの有無では判別できない。

```python
def _question_availability_conditions(q, puzzle):
    """質問の利用可能条件を返す。

    - 回答が「はい」→ reveals 先の命題の fc
    - 回答が「いいえ」→ reveals 先の命題の negation_of が指す命題の fc
    """
    prop = puzzle.propositions.get(q.reveals[0])
    if prop is None:
        return None
    if q.answer == "いいえ" and prop.negation_of is not None:
        target = puzzle.propositions.get(prop.negation_of)
        if target is not None:
            return target.formation_conditions
    return prop.formation_conditions
```

`load_puzzle` にも `negation_of` の読み込みを追加:

```python
p = Proposition(
    id=item["id"],
    label=item["label"],
    negation_of=item.get("negation_of"),  # 追加
    formation_conditions=item.get("formation_conditions"),
    entailment_conditions=item.get("entailment_conditions"),
    rejection_conditions=item.get("rejection_conditions"),
)
```

### eval スクリプト

- `check_recall_fc.py`: 廃止（rc 概念がなくなるため存在意義消滅）
- `run_all.py`: check_recall_fc の参照除去
- `check_integrity.py`: rc 参照チェック除去
- `check_reachability.py`: 利用可能条件の導出を修正
- `check_chain_consistency.py`: 同上
- `visualize.py`: rc 整合性チェック除去
- 新規チェックの検討: `negation_of` の対称性検証（A.negation_of = B なら B.negation_of = A）

eval スクリプトでも、利用可能条件の導出はエンジンと同じ原則に従う。判別は **質問の回答**（はい/いいえ）で行う:

```python
def _get_availability_fc(q, descriptor_map):
    """質問の利用可能条件（reveals 先の fc または negation_of 先の fc）を返す。"""
    reveals = _get_reveals(q)
    if not reveals:
        return None
    d = descriptor_map.get(reveals[0])
    if d is None:
        return None
    if q.get("answer") == "いいえ":
        neg = d.get("negation_of")
        if neg is not None:
            target = descriptor_map.get(neg)
            if target is not None:
                return target.get("formation_conditions")
    return d.get("formation_conditions")
```

### アルゴリズム (統合アルゴリズムv3.md)

- 「維持する原則」の `recall_conditions = reveals される命題の formation_conditions` を廃止
- Phase 4: 質問生成で `recall_conditions` の設定を廃止。代わりに N 命題生成時に否定対象の仮説命題を同時に生成し、`negation_of` を設定
- Phase 7: 検証 Step 4「recall = fc」を廃止。代わりに `negation_of` の整合性検証に置換
- Phase 8: 跳躍レビューの対象を「reveals 先の fc（または negation_of 先の fc）」に変更
- Phase 9: NF 命題生成時に否定対象を明示
- 出力データ構造の全箇所から `recall_conditions` を除去
- 工程2 の説明から rc 用語を除去

### データ (data_src.json)

- 全質問から `recall_conditions` フィールドを除去
- N 系命題に `negation_of` を追加し、否定対象の仮説命題を新規作成
- NF 系命題にも同様に `negation_of` を追加
- P2.fc の不正な第2分岐 `["P1", "S10"]` を除去し `[["S10"]]` に復元
