# 統合アルゴリズム v4 設計

## 背景

v3 サンプル生成（バーの男）で以下の問題が判明した:

1. 導出グラフの頂点が T（非形式的なストーリー全体）のままで、形式的な命題になっていない
2. Phase 5（T 事実の統合）と Phase 6（クリア条件とユニーク要素）が導出グラフの命題体系と別の枠組みで動いている
3. 共通具象元の導入条件が厳しすぎ、既存命題と同一分割の中間命題が不採用になる
4. 抽象化フェーズ（Step 3e）がオプションのため形骸化する

## v3 からの主要な変更

### 1. 最終命題（FP）の導入

- T の代わりに、導出グラフの頂点となる形式的な命題を定義する
- FP から全分割命題への演繹経路が存在すること
- clear_conditions = [["FP"]]（単一要素の集合族、単一要素の集合）

### 2. Phase 構成の再編

- Phase 2 を分割: 分割命題の抽出（Step 1-2）と導出構造の構築（Step 3 + FP）を別 Phase に
- Phase 5（T 事実の統合）を廃止: FP に吸収
- Phase 6（クリア条件とユニーク要素）を廃止: FP のクリア適切性検証に置き換え
- 新 Phase 3 として「導出構造と最終命題」を新設

### 3. entailment_conditions の廃止

- ゲームの探索は抽象→具象の方向に進むため、具象命題の確認時に抽象命題は既に確認済み
- 導出グラフの具象→抽象方向の entailment は実質的に発動しない
- v3 で entailment が使われていたのは T 事実の自動導出だが、T 事実の仕組み自体を FP に置き換えるため不要
- formation_conditions と導出グラフだけで構造を表現する

### 4. 共通具象元の導入条件緩和（v3 で対応済み）

- 共通具象元の分割が既存命題と同一でも、演繹的分岐があれば導入可
- Step 3e（抽象化による発見）を必須ステップに

## 新 Phase 3 の詳細: 導出構造と最終命題

### Step 1: T から分割命題への演繹構造の整理

v3 Phase 2 Step 3a と同一。

### Step 2: 共通具象元の発見

v3 Phase 2 Step 3b と同一。分割が既存命題と同一でも演繹的分岐があれば導入可。

### Step 3: 中間命題の検証

v3 Phase 2 Step 3c と同一（導入条件緩和済み）。

### Step 4: 再帰的適用

v3 Phase 2 Step 3d と同一。

### Step 5: 抽象化による発見

v3 Phase 2 Step 3e と同一（必須化済み）。

### Step 6: 最終命題（FP）の定義

1. 導出グラフの T 直接ノード（T からの直接演繹にとどまる命題群）を列挙する
2. それら全てを演繹できる単一の命題を FP として定義する
3. FP の内容は T の情報から必要十分な範囲を抽出する
4. FP を導出グラフの頂点に据え、T を除去する
5. FP → 各 T 直接ノードへの演繹関係と理由を記録する
6. FP の true_models を判定する（{G1} のみでも可。最終命題に非自明分割は要求しない）

FP の設計基準:
- パズルの核心（プレイヤーの「わかった」に対応する情報）を含むこと
- 全 T 直接ノードを演繹できること
- 独立性・客観性を満たすこと
- 1 つの命題に統合すること

### Step 7: FP のクリア適切性検証

FP が確認された場合に、S の全緊張が解消されるかを検証する。

検証方法:
1. FP から演繹される命題群（導出グラフ全体）と initial_confirmed を合わせる
2. Phase 1 で特定した各緊張に対して、解消メカニズムが説明できるかを確認する
3. 全緊張が解消されれば FP は適切。clear_conditions = [["FP"]]

### Step 8: 不足がある場合の FP' 導入

Step 7 で解消されない緊張がある場合:

1. 不足している情報を特定する
2. FP と不足要素の両方を導出する FP' を定義する
   - FP' → FP
   - FP' → 不足要素
3. FP' を導出グラフの新しい頂点に据える
4. Step 7 を FP' に対して再実施する
5. clear_conditions = [["FP'"]]

再帰的に適用可能だが、実際上は FP' で十分なケースがほとんど。

### Step 9: fc の仮置き

導出グラフに基づき fc を仮置きする。

- P が {B1, B2, ...} を導出 → P.fc = [[B1, B2, ...]]
- 複合命題の fc は構成要素から
- entailment_conditions は設定しない（v4 で廃止）

### Step 10: 棄却不可能モデルの確認

v3 Phase 2 Step 4 と同一。

## データ構造の変更

### Proposition

```python
class Proposition:
    id: str
    label: str
    negation_of: str | None = None
    formation_conditions: list[list[str]] | None = None
    # entailment_conditions: 廃止
    rejection_conditions: list[list[str]] | None = None
```

### data_src.json

- `clear_conditions`: `[["FP"]]` 形式に変更
- `_phase5`（T 事実メタデータ）: 廃止
- `_phase6`（クリア条件メタデータ）: 廃止。Phase 3 の出力に統合
- 導出グラフの頂点が FP（T ではない）

### data.json（エクスポート）

- `entailment_conditions` フィールドを除去
- `clear_conditions` は `[["FP"]]`

## エンジンへの影響

- entailment_conditions の不動点計算を除去
- サイクルが `f(confirmed) → derived` と `g(question, answer) → confirmed` の 2 関数に簡略化
- クリア判定: clear_conditions の命題が全て confirmed であるかの単純チェック

## 維持する原則

- 二段階導出のうち formation_conditions（confirmed→derived 1回パス）は維持
- 質問の利用可能条件は命題構造から導出（質問に独立した条件フィールドを持たない）
- 否定関係（negation_of）は維持
- 1 段階推論の原則は維持
- 命題の独立性・客観性は維持
- T 忠実性は維持
