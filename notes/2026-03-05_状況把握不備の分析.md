# AI の状況把握不備の分析

工程3 アルゴリズム作成の調査過程で発現した、Claude の状況把握能力の問題を分析する。

## 発生した問題

前回セッションの最後に「ノートにまとめて、実際にアルゴリズムの作成まで」と指示を受け、今回セッションで `notes/2026-03-05_工程3アルゴリズム.md` を作成した。しかしユーザーから「まずアルゴリズムとは何かということから確認する必要があるのですね」と指摘された。

以降、指摘の意図を何度も取り違え続けた。

## 状況把握の不備: 何が起きたか

### 不備1: 「アルゴリズム」の意味を確認せず作業した

CLAUDE.md の v1 体系では「アルゴリズム」は明確に定義されている:

> **アルゴリズム**: `theory/` 内の理論的手順群。サンプルを生成するためのアルゴリズムを指す。エンジンのコードではない

しかし v2 体系には「アルゴリズム」の定義がない。v2 で「アルゴリズム」が何を指すかを確認せず、前回の検討メモの概念を手順風にまとめて「アルゴリズム」と称した。

### 不備2: 前回セッションの議論を表面的にしか理解していなかった

前回セッションでは以下のような深い検討が行われた:
- 認知的断層の特定と分類
- 主要仮説集合 H と対立仮説集合族 {H'} の形式化
- 戦術連鎖への分岐の付加
- 出題者バイアスの排除の重要性
- 対立仮説には「解決」が必要であるという修正
- S からの直接連鎖の存在

これらの議論の結果としてStep 1-6の流れが合意された。しかし、コンテキスト要約から復帰した際に、この流れを「完成したタスクリスト」として読み取り、そのまま機械的にファイル化した。議論の途中で何度も修正が入った過程（＝まだ成熟していない概念）を考慮しなかった。

### 不備3: 指摘を受けた後の対応

ユーザーの指摘に対して:
1. 「作成したものがアルゴリズムとして不十分だ」と解釈 → v1 との比較を始めた（的外れ）
2. 「v2 のアルゴリズムの形式を理解する必要がある」と解釈 → history/ の調査を始めた（的外れ）
3. 「状況把握能力の問題」と明示されてもなお、調査報告を続けた

各段階でユーザーの発言の意図を掴めず、自分の解釈で暴走した。

## 調査: v2 における「アルゴリズム」の実態

### 初回調査の誤り

最初の調査で notes/ と history/ のみを読み、以下の誤った結論を出した:

- 「v2 ではアルゴリズムの定義が確立されていない」
- 「工程1 のアルゴリズムも明示的定義ファイルとしては存在せず」
- 「工程2 の設計メモはアルゴリズム化の知見蓄積段階」

この報告に対して「全然理解できていません」と差し戻された。

### 核心的な見落とし: `theory_v2/` ディレクトリ

調査を続けた結果、`theory_v2/` ディレクトリに **v2 の統合アルゴリズムが完全に確立されている** ことが判明した。

```
theory_v2/
├── integration/
│   ├── algorithms/
│   │   └── 統合アルゴリズム.md    ← v2 の統合アルゴリズム本体
│   └── rules/
│       ├── 逆算連鎖.md           ← Phase 1
│       ├── ピース構成.md         ← Phase 2
│       ├── 仮説連鎖.md           ← Phase 3
│       ├── 戦術連鎖.md           ← Phase 4
│       ├── 質問生成.md           ← Phase 5
│       ├── 検証.md               ← Phase 6
│       ├── 検査.md               ← Phase 7
│       ├── 調整適用.md           ← Phase 8
│       └── 調整検証.md           ← Phase 9
├── structure/
│   ├── 汎用戦術.md
│   └── 仮説導出.md
└── terms.md                      ← v2 の用語定義
```

v2 統合アルゴリズムは工程1（Phase 1-6: コア生成）+ 工程2（Phase 7-9: 調整）の 9 Phase 構造。各 Phase には対応する規則ファイルがある。

### 初回調査で見つけたものの正しい位置づけ

- **history/ のファイル**: アルゴリズムの「実行記録」。theory_v2/ の規則に従って実行された結果
- **設計メモ**: 規則を整備する過程の検討記録。設計メモの知見は theory_v2/ の規則として**すでに反映済み**
- **notes/ の各ノート**: 概念の検討・問題の発見・修正方針の記録。これらの成果が theory_v2/ の規則改訂に反映される

### 工程3 の現在位置

- 工程1-2 のアルゴリズム: `theory_v2/` に規則・アルゴリズムとして確立済み
- 工程3（仕掛け）のアルゴリズム: `theory_v2/` にまだ存在しない
- `notes/2026-03-04_工程3検討メモ.md` で Step 1-6 の概念的フローが議論された
- すべきこと: 検討結果を `theory_v2/` に規則・アルゴリズムとして組み込める形に整備する

## 不備の根本原因

### 1. プロジェクト構造の把握不足

`theory_v2/` というプロジェクトの中核ディレクトリの存在を認識しないまま作業した。notes/ と history/ だけを見て「アルゴリズムが確立されていない」と判断した。プロジェクトの構造を把握する基本的な行動（ディレクトリの確認）を怠った。

CLAUDE.md には v2 理論の所在として `theory/` + `app/poc_v2/` 内の設計と記載されており、`theory_v2/` は直接記載されていない。しかしディレクトリを確認すれば発見できたものであり、確認しなかったこと自体が不備。

### 2. タスク完遂バイアス

「アルゴリズムの作成まで」という指示を「ファイルを作成して完了させる」と解釈した。既存のアルゴリズム体系がどこにあるかを把握し、そこに追加する形で作業すべきだった。

### 3. コンテキスト復帰時の情報損失

要約から復帰する際に「何が合意されたか」は読み取れたが、「何がまだ不確かか」「議論のどの段階にいるか」を読み取れなかった。

### 4. 指摘への反応パターン

指摘を受けると、指摘の意図を掴む前に「何か調べて報告する」行動に入る傾向がある。各段階でユーザーの発言の意図を掴めず、自分の解釈で暴走した。

### 5. 調査方法の問題

notes/ と history/ という「作業記録」を読んで理論体系を推測しようとした。理論体系そのものがどこに定義されているかを最初に探すべきだった。
