# 新アルゴリズム設計メモ

砂漠の男を対象に、現行 v2 アルゴリズムの流れに沿いつつ改善点を記録する。

## Phase 1: 逆算連鎖

### S 抽出の分離
- 現行は Phase 1 で S の記述素も抽出するが、不要
- Phase 1 は T の因果分解に専念すべき
- S の記述素抽出は S → T への接続を設計する時点（戦術連鎖相当）で初めて必要

## clear_conditions

### 設計原則
- 最小集合ではなく、T の記述素のうち S で既知のもの以外を原則全て含める
- 「大きくても害がなく、小さいと弊害がある」集合
- 単一の AND グループで十分。族（OR of AND）の活用は理論上は「一部省略しても成立するパターン」を表現できるが、小さいグループが大きいグループを無効化するため実質的に意味がない。また省略バリアントの設計はLLM能力の限界を考えると誤りの元になるのでやらない方が良い

## 記述素の品質

### 客観性の徹底
- 行為の目的・解釈を記述素に混入させない（例: 「捨てて軽量化した」→「捨てた」）
- 程度の評価（「不十分だった」）や必要性の判断（「必要があった」）は客観的事実に置き換える

### 独立性と複合文
- 独立であれば複合的な文でよい。「荷物を全て捨て服を脱いでも熱気球の高度低下は止まらなかった」は他の記述素と内容が重複するが、自己完結しているので独立性を満たす
- 「止まらなかった」のように他の記述素の状態を暗黙に前提とする表現は独立性に反する

### 独立性における暗黙の依存パターン
- 時系列への依存: 「下がり続けた」「後も下がった」は過去からの継続・先行状態を前提としており独立性に反する
- 対処: 依存先の内容を文中に含めて自己完結させる（例: 「高度低下への対処が行われた後も高度が下がっていた」）。短くしようとして依存を省くのではなく、複合文にしてでも独立性を保つ

## トリガー設計

### S を直接参照しない
- 現行アルゴリズムではトリガーが S の記述素を直接参照しがち（例: P-BAL のトリガーに D-S1「砂漠にいた」を使う）
- トリガーは S から導出される中間記述素であるべき。例: 「上空から転落して砂漠に落ちた」→ P-BAL、「トラブルが発生した」→ P-WGT

### S の記述素の扱い
- S の記述素を単独でトリガーにするのは問題（S を見ただけでピースが開く）
- S の記述素は AND グループの一部として組み合わせる分には問題ない
- 例: P-BAL trigger [["D-11"], ["D-12", "D-S1"]] — D-S1 は D-12 との組み合わせで使用

### Phase 2/3 の統合の示唆
- トリガー設計には中間記述素の生成とその導出条件（formation_conditions）の設定が不可分
- 現行は Phase 2（ピース構成・トリガー設計）と Phase 3（仮説連鎖・導出条件設定）を分けているが、統合した方が自然かもしれない

## ピースの領域

### 考え方
- ピースを起点に、そこから広がる探索領域（中間記述素群）を定義する
- T の事実だけでなく、プレイヤーが持ちうる仮説方向の記述素も含まれる

### P-BAL の領域
- 男は乗り物に乗っていた
- 男は空を移動していた
- 男は一人ではなかった
- 乗り物の高度が低下した
- 乗り物の機材にトラブルがあった
- 燃料が不足した

後の2つは T の事実ではなくプレイヤーの仮説方向。高度低下の原因として自然に推測されるもの

### P-WGT の領域
- 乗っている人たちが荷物を捨てた
- 乗っている人たちが身につけているものを脱いだ
- 高度低下への対処が行われた
- 高度低下への対処が行われた後も高度が下がっていた

### P-LOT の領域
- 乗っている人のうち一人が犠牲になることになった
- 犠牲になる人をくじ引きで決めた
- マッチ棒がくじ引きの道具として使われた
- 男がくじ引きに負けた
- 男は乗り物から落下した

### 仮説方向の記述素
- ピースの領域に仮説方向の記述素（マッチで火をつけた等）は不要

## Phase 3 への移行で生じた誤り

### Phase 2/3 統合についての誤解
- 「フェーズを一つにまとめた理由も考えるように」は、現行アルゴリズムで既に行われた統合についての話だった
- 将来の新アルゴリズムで統合すべきという提案と誤解した

### formation_conditions の対象
- T の記述素であってもトリガーであれば formation_conditions の対象
- D-5 は T 由来だが P-LOT のトリガーなので対象外ではない
- 「T の記述素だから対象外」という判断は規則に根拠がない

## Phase 3: 仮説連鎖の導出設計

### 導出チェーンと領域の区別
- 「男は乗り物に乗っていた」は領域の記述素ではなく、D-11 への導出チェーンの中間ステップ
- 領域は探索の広がり、導出チェーンはトリガーへの到達経路。混同しない

### トリガー並列 OR グループ間の導出禁止
- D-11 と D-12 は P-BAL トリガーの並列 OR グループ
- 一方から他方を導出可能にすると、片方の OR グループが常に吸収されて死ぬ
- 並列グループは独立した経路であるべき

### 導出の考え方
- 仮説演繹的に考える。「何が分かればこの結論に至るか」を下方向に展開する
- 末端記述素は質問で直接確認されるもの（例: D-28「男の死因は転落死によるものだった」）
- 不必要に中間ステップを増やさない。D-28 → D-12 のように直接的でよい

## Phase 4: 戦術連鎖

### 誤り: 領域記述素の無視
- P-BAL の戦術連鎖を提案した際、Phase 2 で定義した領域記述素（D-15〜D-17）を無視し、トリガー D-11 から直接 T メンバー D-1, D-2 へ至る連鎖を提案した
  - 誤った提案: `D-11 →[段階的絞り込み]→ D-1` `D-1 →[観測事実の深掘り]→ D-2`
  - 正しくは D-15「空を移動していた」, D-16「一人ではなかった」, D-17「高度が低下した」を経由すべき
- 原因: Phase 2 の成果物（領域記述素）を Phase 4 の検討時に参照しなかった。データには `piece_domain` として適切に登録されていた

### 誤り: データへの反映漏れ
- 戦術連鎖の検討結果を履歴ファイル（history/04_戦術連鎖.md）にのみ記録し、data_src.json への `_phase4` セクション追加を忘れた
- 議論の結果は必ずデータに反映する。履歴ファイルはあくまで記録であり、データが正

### 戦術連鎖の設計方針
- トリガーから次のピースのトリガー（とその導出仮説）への到達パスを主軸にする
- 主軸に直接関与しない記述素（例: D-16「一人ではなかった」）は独立鎖とし、深さを抑える
- 複合的な T 記述素は分解して導出仮説を用意する（例: D-1「数人で熱気球に乗っていた」← D-34「熱気球に乗っていた」+ D-16「一人ではなかった」）
- 抽象度の異なる記述素ペア（乗り物 vs 熱気球）は、抽象版 + 具体化記述素の組み合わせで導出する（例: D-2 ← D-17 + D-34、D-13 ← D-35 + D-34）

### P-WGT トリガーの変更
- D-13「熱気球にトラブルが発生した」→ D-2「熱気球の高度が下がり始めた」に変更
- D-13 は D-35「乗り物にトラブルが発生した」+ D-34「熱気球に乗っていた」から導出される中間記述素に
- D-17「乗り物の高度が低下した」は D-35 から導出、D-2 は D-17 + D-34 から導出
- D-34 が乗り物→熱気球の具体化に繰り返し使われる構造

### P-BAL グループ1 の戦術連鎖
```
D-11「上空にいた」
  ├─[段階的絞り込み]→ D-15「空を移動していた」
  │    └─[段階的絞り込み]→ D-34「熱気球に乗っていた」
  ├─[登場人物の探索]→ D-16「一人ではなかった」
  │  ※ D-34 + D-16 → D-1「数人で熱気球に乗っていた」導出
  └─ D-1 から:
       ├─[時間軸の拡張]→ D-32「死ぬ間際に男は焦っていた」

```

## Phase 5: 質問生成

### 誤り: 戦術構造の完全な無視
- Phase 4 で設計した戦術連鎖（戦術の種類・入れ子構造・段階的プロセス）を質問に変換する際、戦術の構造を完全に無視し、結論の記述素を直接尋ねる質問のみを生成した
  - 例: 戦術「常識的仮説の排除 > 段階的絞り込み」で {D-S1, D-S4} → D-14 に対し、「男は乗り物に乗っていましたか？」という単一の質問を提案
  - 問題: 戦術が示す探索プロセス（ありがちな説明を段階的に排除していく過程）が質問に反映されていない。排除ステップも絞り込みの段階もなく、結論だけを直接聞いている
- 戦術の種類と構造が質問の形・数・順序を決定するべき。戦術→質問の変換規則が現行アルゴリズムでは不十分であり、大幅な見直しが必要

### 戦術連鎖の再検討: 末端チェーンの重複

- D-14「乗り物に乗っていた」への連鎖を再検討した際、最もシンプルな経路はD-S1「砂漠の真ん中にいた」から観測事実（足跡がない、タイヤの跡がない等）を段階的に積み上げて到達する形だった
- しかし、この観測事実群（D-29, D-30, D-31）はD-27「地上の乗り物ではない」の導出チェーンと完全に重複する
- D-14 と D-27 は同じ「乗り物」に関する記述素であり、同じ観測事実から同時に導かれるのは自然
- 異なる末端チェーンが同じ観測事実を共有するケースは他にも発生しうる。重複を整理・統合するステップをアルゴリズムに設けるのが妥当

### 汎用戦術の不足: how（手段）を問う戦術がない

- 既存の戦術は why（因果の逆転、動機の探索）、what（観測事実の深掘り）、who（登場人物の探索）、when（時間軸の拡張）をカバーするが、how（手段・方法）を明示的に問う戦術がない
- 因果の逆転は「なぜ」を問い、手段の探索は「どうやって」を問う。同じ事実に対しても探索方向と到達する記述素が異なる
  - 因果の逆転: {砂漠の真ん中にいた} → {気球が飛んでいたから}（原因）
  - 手段の探索: {砂漠の真ん中にいた} → {乗り物で移動していた}（手段）
- 新戦術「手段の探索」の案:
  - 入力: 記述素集合（状態・結果）
  - 出力: 手段・方法に関する記述素集合
  - 説明: 状態や結果が「どのように」実現されたかを問う
  - 適用例: {砂漠の真ん中にいた} → {乗り物で移動していた}
  - 適用例: {死んでいた} → {転落死だった}

### D-14 への連鎖の再設計

- 手段の探索を上位戦術とし、その中で常識的仮説の排除を適用:
  ```
  {D-S1「砂漠の真ん中にいた」}
    → [手段の探索 > 常識的仮説の排除]
      → 「男は歩いてここに来たのではない」
    → D-14「乗り物に乗っていた」
  ```
- 手段の探索が「どうやってここに来たか」という問いの枠を設定するため、「歩いてない」→「乗り物」の遷移が自然になる
- 以前の設計（常識的仮説の排除 > 段階的絞り込み）では手段という枠がなく、排除結果から乗り物への接続が唐突だった

### 誤り: D-32 連鎖の問題を見逃した

- {D-1「数人で熱気球に乗っていた」} -[時間軸の拡張]-> D-32「死ぬ間際に男は焦っていた」を「素直な適用」と判断した
- 実際の問題:
  1. 連想が不自然: 「熱気球に乗っていた」からプレイヤーが「死ぬ間際に焦っていたか」を問う動機がない
  2. 結論から逆算した設計: D-32 + D-33 → D-35 の導出に必要だから作っただけで、プレイヤーの自然な思考の流れではない
  3. 客観性の問題: 「焦っていた」は内面の状態であり、客観的に確認しにくい
  4. 前提の問題: 「死ぬ間際に」は気球と死の時間的連続を前提とするが、プレイヤーがそれを知っているとは限らない
- D-14, D-28 では跳躍の大きさに気づいて中間記述素を導入できたが、D-32 では同じ批判的な視点を適用できなかった。全ての連鎖に対して一貫して「プレイヤーの自然な思考か」を検証すべき

### D-32 連鎖の再検討

- 起点を D-28「転落死だった」に変更し、時間軸の拡張が外枠
- 内部戦術の検討:
  - 常識的仮説の排除「自殺ではなかった」→ 有効
  - 因果の逆転、観測事実の深掘り → 「焦っていた」への接続が不自然
- 「自殺ではない」から「焦っていた」への接続が自然に埋まらない → **保留**とする判断が妥当

### アルゴリズムにおける「保留」の導入

- 戦術連鎖の設計中、自然な連鎖が見つからない場合に無理に不適切な戦術を当てはめるべきではない
- 保留にしておくことで:
  - そのパスが不要と判明して破棄される可能性
  - 他の経路との自然な合流が後から見つかる可能性
  - いずれも無理な当てはめより良い結果になる
- 中間データ（data_src.json）上で保留状態が明確にわかる構成が必要（例: `"status": "pending"` 等のマーカー）
- アルゴリズムのステップとして「保留判断」を明示的に組み込むべき

### 誤り: 安易な保留判断

- D-33 の循環問題を指摘された際、検討もせずに「保留が妥当でしょうか？」と提案した
- 保留は十分に検討した上で自然な連鎖が見つからない場合の判断であり、考えてもいないものを保留とすべきではない
- D-32 では複数の内部戦術を検討した上で保留とした。D-33 にも同等の検討が必要

### 誤り: 段階的絞り込みの具体例を省略

- 段階的絞り込みを戦術として使う際、中間記述素の具体例を挙げずに「段階的絞り込み → 終点」と一足飛びに書いた
- 段階的絞り込みの本質は中間ステップの存在であり、少なくとも1つの具体的な中間記述素を示さなければ戦術として機能していない
- これは D-14 で指摘されたのと同じ問題の繰り返し

### D-33 への連鎖の再設計

- 現行の起点 {D-1, D-2} は循環（D-2 は D-33 → D-35 → D-17 → D-2 の出力側）
- 起点を D-28「転落死だった」に変更。因果の逆転が外枠
- 再設計:
  ```
  {D-28「転落死だった」}
    → [因果の逆転 > 常識的仮説の排除]
      → 「事故や自殺による転落ではなかった」
      → 「男はやむを得ず自ら飛び降りた」
    → [因果の逆転 > 矛盾の指摘]
      → D-33「転落しなくても死ぬ可能性があった」
  ```
- 矛盾: 自ら死を選ぶ行為を「やむを得ず」行った ←→ 自己保存本能。解消として「飛び降りなくても死ぬ状況だった」

### 前提条件と想起条件の分離

#### 誤り1: 前提条件の意味を取り違えた

- 質問に「前提条件」フィールドを追加する提案に対し、「構造的・論理的な必須条件」（例: ピースが開いている、記述素が確定済み）と解釈した
- 正しくは「質問文に含意された前提条件（presupposition）」である
  - 例: Q「マッチ棒でくじ引きをしましたか？」→ 前提条件: くじ引きが行われた（D-23）。「マッチ棒**で**」と手段を問う以上、くじ引き自体が前提
  - 例: Q「男は転落して死にましたか？」→ 前提条件: 男は死んだ（D-S4）。質問文が死を前提としている
- 現実的には確定していないものを前提にする質問もありうるが、区別自体は重要

#### 誤り2: 前提と想起の関係を逆に理解した

- 「前提は導出で満たされうるが、想起は導出では満たされない」と主張した → **逆**
- 正しい関係:
  - **想起条件は仮説の導出である**: formation_conditions と同じ仕組み。確定した仮説の集合から「この質問が思い浮かぶ」を導出する。導出された記述素を入力にしても問題ない
  - **前提条件は導出ではない**: 質問文の言語的前提であり、プレイヤーと出題者の間の対話上で明示的に確立された事実でなければならない。エンジン内部の導出だけでは対話上の前提として成立しない
- 導出の意味を正確に理解していなかった。本システムにおける「導出」は仮説の導出であり、formation_conditions による記述素の自動導出と、想起条件による質問の想起はどちらも同じ「仮説の導出」の枠組みに属する

#### 体系の整理

- **仮説**: 本システムの基本単位。観測された事実（S の記述素）は「確定された仮説」であり、仮説の一種
- **導出**: 確定した仮説の集合から新たな仮説を導く操作
  - formation_conditions: 仮説 → 仮説（記述素の自動導出）
  - recall_conditions: 仮説 → 質問の想起（質問が思い浮かぶ）
  - どちらも「仮説に基づく導出」という同じ枠組み
- **前提条件**: 質問文が含意する前提。導出とは別の概念
  - 前提は対話上で確立されている必要がある（initial_confirmed または質問の reveals）
  - 導出（formation_conditions）のみで得られた記述素は前提を満たさない

#### 実際の問題: D-2 が前提条件として不適格

- Q-15「高度低下に対して何か対処をしましたか？」の前提条件に D-2「熱気球の高度が下がり始めた」を設定した
- D-2 はどの質問の reveals にも含まれず、formation_conditions [["D-17", "D-34"]] でのみ導出される

## Phase 6: 検証で発見された構造的欠陥

### D-21 スキップ問題

手動パストレースの実施中に、clear_conditions の設計を検討する過程で発見された。

#### 背景: clear_conditions における derived の問題

- clear_conditions は `check_complete` で `state.confirmed` に対してチェックされる
- 当初の clear_conditions `[["D-1", "D-2", ..., "D-9"]]` には D-1, D-5, D-7, D-9 など導出でしか得られない記述素が含まれていた
- これらは confirmed に入らないため、クリア判定が成立しない構造的な問題があった
- clear_conditions を revealed される記述素のみで構成する必要があり、意味的な妥当性を検証していく過程で D-21 の問題が浮上した

#### 発見された問題

D-5「荷物を全て捨て服を脱いでも熱気球の高度低下は止まらなかった」の formation_conditions が `[["D-2", "D-3", "D-4"]]` であった。D-21「対処後も高度が下がっていた」が含まれていない。

これにより以下のパスが成立する:

1. D-2 confirmed（Q-25 回答）
2. D-3 confirmed（Q-16 回答）
3. D-4 confirmed（Q-17 回答）
4. D-5 が導出される（D-2 + D-3 + D-4）
5. Q-19「誰かが犠牲に〜」が開く（recall: [[D-5]]）
6. P-LOT の質問群が順次開く

この間、Q-18「それでも高度は下がり続けましたか？」（D-21 を reveal する唯一の質問）は一度も回答される必要がない。プレイヤーは「対処が失敗した」事実を確認せずに「犠牲が必要」の段階に進める。

#### 意味的な分析

D-5 のラベルには「止まらなかった」が含まれている。これは「対処後もなお高度が下がっていた」（D-21）を前提としている。しかし formation_conditions は D-2（高度低下）+ D-3（荷物を捨てた）+ D-4（服を脱いだ）のみで、「対処したが足りなかった」という情報が欠落していた。

D-2, D-3, D-4 が揃うだけでは「対処をした」ことは分かるが「対処が失敗した」ことは分からない。D-21 がなければ D-5 への導出は意味的に不完全である。

#### 修正

D-5 の formation_conditions を `[["D-2", "D-3", "D-4", "D-21"]]` に変更する。

これにより:
- D-21 は Q-18 の reveal でのみ得られるため、Q-18 の回答が必須になる
- Q-18 は prerequisites [D-2, D-20] を持ち、対処の事実確認後にのみ開く
- D-5 の導出に「対処の失敗」の確認が含まれ、P-LOT への遷移が意味的にも正当化される

#### 教訓

- formation_conditions の設計時に、ラベルの意味と条件の意味の一致を検証すべき
- 「足りず」「止まらなかった」のような否定的結果を含む記述素は、その否定的結果に対応する記述素が条件に含まれているか確認が必要
- 手動パストレースによる検証は、機械的な整合性チェックでは発見できない意味的な欠陥を発見できる
