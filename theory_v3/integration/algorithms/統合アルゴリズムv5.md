# 統合アルゴリズム v5

## 概要

モデル集合の分割構造を起点に、導出構造と最終命題（FP）を構築し、戦術連鎖・質問生成を経て、真側連鎖の分岐分析を通じて棄却側を構築するアルゴリズム。工程（真側構築→ブラッシュアップ→棄却側構築）による段階的構築を行う。

### v4 からの主要な変更

- **YES 進行原則の導入**: 「はい」回答のみがゲーム進行の駆動力。「いいえ」回答は排除（消去法）。fc に N*/NF* を含めない
- **ブラッシュアップの拡張**: 跳躍レビューに加え、fc 内の N*/NF* の「はい」命題への置き換えを実施
- **全ステップ分岐の必須化**: 連鎖内分岐分析で各連鎖の全ステップに最低 1 つの分岐を要求
- **質問文の含意検証の追加**: 検証フェーズで質問文の前提が利用可能時点で保証されているかを判例に基づき検証
- **否定テストの拡張**: 情報漏洩チェック（NF ラベルが正解情報を含まないか、文脈上正解が自明にならないか）を追加

### 維持する原則

- **仮説導出**: formation_conditions（confirmed→derived 1回パス）。`structure/仮説導出.md` 参照
- **質問の利用可能条件は命題構造から導出**: 質問に独立した条件フィールドを持たない。回答が「はい」なら reveals 先命題の fc、「いいえ」なら reveals 先命題の negation_of が指す命題の fc が利用可能条件となる
- **否定関係（negation_of）**: 命題間の対称的な否定関係。P.negation_of = Q ⟺ Q.negation_of = P。nullable（全命題に必須ではない）
- **仮説連鎖の廃止**: formation_conditions は confirmed のみ参照。derived を参照しない
- **1 段階推論の原則**: 各導出条件は単一の推論ステップを表現する
- **命題の独立性・客観性**: 全命題は自己完結的で客観的に検証可能
- **T 忠実性**: 生成する命題は T の情報から逸脱しない
- **YES 進行原則**: 「はい」回答のみが進行を駆動する。fc は「はい」で confirm される命題（P*, I*, S*）のみを含む。ゲームは「いいえ」質問を一度も聞かずにクリア可能であること

## 入力

- S: 問題文
- T: 真相

## 出力

- data_src.json: 全フェーズの中間生成物を含む
- data.json: ゲームデータ

## 大域目標

**複数の質問選択順において合理的なパスで T に到達できること**

合理的なパスとは、各質問の開放が想起（確認済み命題または導出命題）により説明でき、唐突な飛躍がないパスを指す。

## 工程構成

Phase 1-8 を 3 つの工程に分け、段階的に構築する。

### 工程1: 真側の構築

| Phase | 内容 |
|-------|------|
| Phase 1 | 緊張構造分析 + モデル列挙 |
| Phase 2 | モデル具象命題 + 分割命題 |
| Phase 3 | 導出構造 + 最終命題 + クリア適切性検証 |
| Phase 4 | S 細分化 + 戦術連鎖 |
| Phase 5 | 質問生成 |
| Phase 6 | 検証（真側） |

真側だけで完結する。棄却側は扱わない。検証は真側の範囲で実施する（棄却の完全性チェックは除く）。

### 工程2: ブラッシュアップ

| Phase | 内容 |
|-------|------|
| Phase 7 | 戦術連鎖のブラッシュアップ |

工程1の真側を磨き上げる。各質問の利用可能条件（reveals 先命題の fc）に跳躍がないかをレビューし、跳躍がある場合は戦術連鎖を修正して解消する。反復的に実施する。

### 工程3: 棄却側の構築

| Phase | 内容 |
|-------|------|
| Phase 8 | 連鎖内分岐分析 |
| Phase 6 | 検証（全体） |

真側連鎖の分岐を分析し、棄却側の質問を生成する。反復的に実施する（導出と検証を別ファイルに分離し、問題があれば修正→再検証のサイクルを回す）。検証は全体（真側 + 棄却側）の範囲で実施する。

### 工程4: 調整

| Phase | 内容 |
|-------|------|
| Phase 10 | 調整（Step A: 間引き、Step B: 演繹関係の特定） |

Step A で冗長命題を間引き、Step B で論理的結合の命題に entailment_conditions を設定する。両ステップとも質問数の削減に寄与する。

### 工程5: 質問分類

| Phase | 内容 |
|-------|------|
| Phase 9 | 質問分類 |

全質問にプレイヤー向けのトピックカテゴリを付与する。戦術ベースの固定カテゴリセットを使用し、情報漏洩検証とバランス確認を行う。ゲームロジックには影響しない後処理である。

## Phase 構成

### Phase 1: 緊張構造分析とモデル列挙

規則: `rules/緊張構造分析_v4.md`、`rules/モデル列挙_v2.md`

#### 目的

S に含まれる緊張構造を特定・分解し、T および競合モデルを列挙した上で、緩和パターンによるバリアント識別と抽象化を行う。

#### 入力

S, T

#### 手順

1. S の緊張構造を抽出する
   - 予測の失敗: S の事象から常識的に予測される結果と実際の乖離
   - 前提の崩壊: S が暗黙に想定させる前提の不成立
   - 説明不能な点: S の事象間で因果関係が見えない箇所
2. 緊張の分解: 各緊張に分解原則（程度の分離・前提の抽出・独立検証）を適用し、細粒度の緊張を導出する
3. T がそれらをどう緩和するか確認する
4. 緊張の部分集合ごとに、S と整合する競合モデルを列挙する
   - 全緊張を緩和するものから、一部のみ緩和するものまで
   - 各モデルは T とは異なる具象的なストーリーを持つ
5. 各モデルの緩和パターン（各緊張をどのメカニズムで緩和するか）を記録する
6. 緩和パターンが同一のモデルをバリアントとして識別し、省く
7. バリアント群の共通抽象を特定し、抽象モデル（abstract_models）を導出する

#### 出力データ構造

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "_phase1": {
    "tensions": [
      {"id": "α1", "label": "緊張の記述", "parent": "α"},
      {"id": "α2", "label": "緊張の記述", "parent": "α"}
    ],
    "models": [
      {
        "id": "T",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α1", "α2", "β1", "β2", "γ1", "γ2"]
      },
      {
        "id": "M1",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α1", "α2", "β1", "β2", "γ1", "γ2"]
      }
    ],
    "relaxation_patterns": {
      "T": {"α1": "メカニズム記述", "α2": "メカニズム記述"},
      "M1": {"α1": "メカニズム記述", "α2": "メカニズム記述"}
    },
    "variant_groups": [
      {
        "group_id": "G1",
        "label": "共通抽象の記述",
        "representative": "T",
        "members": ["T", "M1", "M2"],
        "shared_pattern": {"α1": "共通メカニズム", "α2": "共通メカニズム"}
      }
    ],
    "abstract_models": [
      {
        "id": "G1",
        "label": "共通抽象の記述",
        "representative": "T",
        "members": ["T", "M1", "M2"],
        "tension_subset": ["α1", "α2", "β1", "β2", "γ1", "γ2"]
      },
      {
        "id": "M5",
        "label": "モデル名",
        "members": ["M5"],
        "tension_subset": ["α1", "α2", "β1", "β2", "γ1", "γ2"]
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `title`: パズル名（文字列）
- `S`: 問題文（文字列）
- `T`: 真相（文字列）
- `_phase1`: Phase 固有メタデータ
  - `tensions`: 緊張の配列。各要素は `{id, label, parent}`
    - `id`: 緊張 ID（分解後のサブ ID。例: α1, α2）
    - `label`: 緊張の記述
    - `parent`: 分解元の緊張 ID（分解されていない場合は null）
  - `models`: 全モデルの配列（バリアントを含む完全なリスト）。各要素は `{id, label, summary, tension_subset}`
  - `relaxation_patterns`: 各モデルの緩和パターン。モデル ID → {緊張 ID → メカニズム記述} の辞書
  - `variant_groups`: バリアント群の配列。各要素は `{group_id, label, representative, members, shared_pattern}`
  - `abstract_models`: 抽象モデルの配列。Phase 2 への入力として使用される
    - バリアント群: `{id（=group_id）, label, representative, members, tension_subset}`
    - 単独モデル: `{id（=モデルID）, label, members（=[自身]）, tension_subset}`

#### 規則

- 競合モデルは最低 10 以上列挙する
- 各モデルは S の全事実と整合すること
- 緊張の部分集合の分類により、モデルの網羅性を担保する
- T も models に含め、他のモデルと同列に扱う
- バリアント判定は緩和パターンの構造的同一性に基づくこと
- Phase 2 は abstract_models を入力として使用する

### Phase 2: モデル具象命題と分割命題

規則: `rules/モデル具象命題と分割構造_v4.md`

#### 目的

各モデルの具象命題を記述し、モデル横断で共通命題を抽出して、分割命題を導出する。

#### 入力

Phase 1 の出力（`abstract_models` を使用する。個別モデルではなく抽象モデルを分割の単位とする）

#### 手順

1. 各抽象モデルについて、具象命題を列挙する
   - バリアント群の場合は、群が共有する因果構造を命題として明示する
   - 単独モデルの場合は従来通りモデルの因果構造を記述する
2. 抽象モデル間で共通する命題を抽出する
   - 同じ事実を共有する抽象モデル群を特定する
   - カテゴリは「人物の状態」「他者の認識と行動」「解消メカニズム」等、パズルの構造に応じて自然に定まる
   - 同一の分割を達成する命題が複数ある場合、より抽象的な表現を優先する

#### 出力データ構造

```json
{
  "_phase2": {
    "model_propositions": {
      "G1": ["過去に他者から偽りの情報を受けた", "極限状況で飲食経験があった", "..."],
      "M5": ["男は料理人として働いていた", "..."]
    },
    "division_propositions": [
      {
        "id": "P1",
        "label": "男は何らかの症状・状態に困っていた",
        "truth_value_in_T": true,
        "true_models": ["G1", "M5", "M8"],
        "false_models": ["M9", "M10", "M11"]
      }
    ]
  }
}
```

**注**: `model_propositions`, `division_propositions` のモデル ID は `_phase1.abstract_models` の ID を使用する（バリアント群の場合は group_id、単独モデルの場合はモデル ID）。

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `_phase2`: Phase 固有メタデータ
  - `model_propositions`: 抽象モデル ID → 具象命題ラベルのリスト（辞書）
  - `division_propositions`: 分割命題の配列。各要素:
    - `id`: 分割命題 ID（P*）
    - `label`: 分割命題のラベル
    - `truth_value_in_T`: T における真偽（必ず true）
    - `true_models`: 真である抽象モデル ID リスト
    - `false_models`: 偽である抽象モデル ID リスト

#### 規則

- 分割命題は T において全て真であること
- 具象命題はモデルの因果構造を反映すること（単なる事実の羅列ではなく因果関係を含む）
- 分割命題はモデルの具象命題から抽象化してよいが、T の事実に忠実であること
- 同一の分割を達成する命題が複数ある場合、より抽象的な表現を優先すること
- **導入時検証では `precedents/命題の適切さの判例.md` を参照し、各命題を個別に検証すること**

### Phase 3: 導出構造と最終命題

規則: `rules/導出構造と最終命題.md`

#### 目的

Phase 2 で生成した分割命題間の導出関係を分析し、導出グラフを構築する。導出グラフの頂点に最終命題（FP）を据え、FP のクリア適切性を検証する。fc の仮置きと棄却不可能モデルの確認も行う。

#### 入力

Phase 1〜2 の出力（分割命題群を使用する）

#### 手順

##### Step 1: T から分割命題への演繹構造の整理

T から全分割命題への演繹関係を整理する。全分割命題は T において真であるため、T から直接演繹できる。この過程で命題間の演繹関係も自然に現れる。

手順:
1. T → 各分割命題の直接演繹を描く
2. 命題間の演繹関係を特定する（A → B: A の概念的内容から B が論理的に導かれる場合）
3. T → B の直接辺が T → A → B の経路に包含される場合、冗長として除去する

判定基準:
- 導出関係は命題の意味に基づいて判定する（モデル集合の包含関係に依存しない）
- 各導出について理由を記録する

##### Step 2: 共通具象元の発見

演繹構造の中に、複数の命題を同時に演繹する**共通具象元**を見出す。T と分割命題の間に限らず、構造内の任意の位置に存在しうる。

共通具象元とは:
- 2 つ以上の命題より具象的（情報が多い）な命題
- 異なる情報を落とす（= 抽象化する）ことで、それぞれの命題が演繹される
- 一本道（1 つの命題のみを演繹する中間）ではなく、**分岐**を生む命題

発見の手順:
1. T の事実の中から、複数の分割命題に跨る具象的事実を特定する
2. その事実から異なる情報を落とすことで、対応する分割命題が得られることを確認する

##### Step 3: 中間命題の検証

Step 2 で見出した共通具象元が、中間命題としての要件を満たすか検証する。

**中間命題の二重要件**:
1. **演繹的分岐**: 複数の分割命題を同時に演繹する（具象→抽象の情報削減で）
2. **モデル非自明分割**: true_models が {T} のみでなく、モデル集合を非自明に分割する

**分割の同一性に関する許容**: 共通具象元の true_models が既存の分割命題と同一であっても、演繹的分岐を満たす限り中間命題として導入してよい。導出グラフに深さを生むことが目的であり、分割の一意性は要求しない。

検証手順:
- Step 1 のモデル具象命題と照合し、各モデルにおける真偽を判定する
- true_models と false_models を明示する
- 意味確定性チェックを適用する

要件を満たさない場合:
- true_models = {T} のみの場合、命題が具象的すぎる。モデル分割が成立するレベルまで抽象化を検討する
- 抽象化しても分割が成立しない場合、その共通具象元は中間命題としては不適格

##### Step 4: 再帰的適用

Step 1-3 を再帰的に適用する。見出した中間命題を分割命題群に追加し、T からの演繹構造を再整理する。新たな共通具象元が見つからなくなるまで繰り返す。

結果として、T から分割命題群までの演繹 DAG が構築される。DAG に循環がないことを確認する。

##### Step 5: 抽象化による発見

既存の分割命題に抽象化を適用することで、一見無関係に見える命題間の共通構造が浮かび上がり、Step 2 では直接見出せなかった共通具象元の発見を導くことがある。

**本ステップは必須である。** Step 2-4 で構築した導出グラフに対して、以下を実施する:

1. T から直接演繹にとどまる分割命題（グラフ上で T の直下にあるノード）を列挙する
2. それらの命題間に、抽象化を通じた共通構造がないか検討する
3. 共通構造が見つかった場合、共通具象元の候補として Step 3 の検証を適用する
4. 検証を通過した場合、Step 4 に従い再帰的にグラフを更新する

グラフが十分な深さを持つ場合（T から最も抽象的な命題までの最長経路が 2 以上）でも、本ステップは省略しない。発見的な操作であるため新たな命題が見つからない場合もあるが、探索自体は実施すること。判例（`precedents/導出関係の判例.md`）で具体パターンを参照する。

##### Step 6: 最終命題（FP）の定義

1. 導出グラフの T 直接ノード（T からの直接演繹にとどまる命題群）を列挙する
2. それら全てを演繹できる単一の命題を FP として定義する
3. FP の内容は T の情報から必要十分な範囲を抽出する
4. FP を導出グラフの頂点に据え、T を除去する
5. FP → 各 T 直接ノードへの演繹関係と理由を記録する
6. FP の true_models を判定する（{G1} のみでも可。最終命題に非自明分割は要求しない）

FP の設計基準:
- パズルの核心（プレイヤーの「わかった」に対応する情報）を含むこと
- 全 T 直接ノードを演繹できること
- 独立性・客観性を満たすこと
- 1 つの命題に統合すること

##### Step 7: FP のクリア適切性検証

FP が確認された場合に、S の全緊張が解消されるかを検証する。

検証方法:
1. FP から演繹される命題群（導出グラフ全体）と initial_confirmed を合わせる
2. Phase 1 で特定した各緊張に対して、解消メカニズムが説明できるかを確認する
3. 全緊張が解消されれば FP は適切。clear_conditions = [["FP"]]

##### Step 8: 不足がある場合の FP' 導入

Step 7 で解消されない緊張がある場合:

1. 不足している情報を特定する
2. FP と不足要素の両方を導出する FP' を定義する
   - FP' → FP
   - FP' → 不足要素
3. FP' を導出グラフの新しい頂点に据える
4. Step 7 を FP' に対して再実施する
5. clear_conditions = [["FP'"]]

再帰的に適用可能だが、実際上は FP' で十分なケースがほとんど。

##### Step 9: fc の仮置き

導出グラフに基づき fc を仮置きする。

- P が {B1, B2, ...} を導出 → P.fc = [[B1, B2, ...]]
- 複合命題の fc は構成要素から

**この fc はあくまで骨格であり、Phase 4 で戦術連鎖を設計する際に確定・修正される。**

##### Step 10: 棄却不可能モデルの確認

導出グラフにおいて、T の位置づけと棄却不可能モデルを確認する。

確認事項:
- 導出グラフの最も具象的なノード（FP に最も近い中間命題）において、T と区別できないモデル群（全分割命題を共有し、棄却不可能なモデル群）が存在するか
- その群内での具象的な差異はパズルの探索としての重要度が低いか
- この確認は棄却分析の unrejectable_models の根拠となる

注意:
- 粗い抽象を安易に T と等価としないこと。等価の範囲は慎重に判断する
- 導出グラフがあれば、どの段階を境界とするかは後から選べる

#### 出力データ構造


- FP のクリア適切性検証で全緊張が解消されること（解消されない場合は FP' を導入）
- 棄却不可能モデル群の根拠を明示すること
- 粗い抽象を安易に T と等価としないこと
- fc の仮置きはあくまで骨格であり、Phase 4 の戦術連鎖設計で確定されること

### Phase 4: S 細分化と戦術連鎖

規則: `rules/S細分化.md`、`rules/戦術連鎖_v5.md`

#### 目的

S を原子命題に分解し、S 命題から分割命題への戦術連鎖を構成する。Phase 3 の仮置き fc を出発点として連鎖を設計し、fc を確定する。

#### 入力

Phase 1〜3 の出力（導出グラフ・仮置き fc を含む）、`structure/汎用戦術.md`

#### 手順

##### Step 1: S 細分化

S を分割不能な原子命題に分解する。論理的含意による導出命題も列挙する。

- 原子命題: S の文から直接取り出せる最小の事実
- 導出命題: 原子命題から論理的に含意されるもののみ。推測・解釈を含めない
- 自明な含意（「バーがあった」等）も見落とさないこと

##### Step 2: S → 根命題への戦術連鎖

S の原子命題を起点とし、根命題（最も抽象的な分割命題）に至る戦術連鎖を構成する。各連鎖は中間命題を導入しながら根命題に至る（量的最低基準: 各連鎖に最低 1 つの中間命題）。

- 各独立軸に対応する根命題ごとに、独立した連鎖を設計する
- 各連鎖は `structure/汎用戦術.md` の戦術を適用する
- 各ステップの出力を中間命題として形式化する（命題 ID を付与）
- S の情報構造から自然に生まれるものであり、T からの逆算ではない

##### Step 3: 具象方向への戦術連鎖

根命題の確認を起点とし、導出グラフに沿ってより具象的な分割命題に至る戦術連鎖を構成する。各連鎖は中間命題を導入しなければならない。

- 仮置き fc が連鎖の起点を示す
- 各ステップの出力を中間命題として形式化する（命題 ID を付与）
- 連鎖設計の結果として fc を確定する（仮置きの修正を含む）

本 Phase では各命題に対して 1 つの導出経路（fc の単一 AND グループ）を設計する。fc の OR 構造（複数の代替導出経路）はこの時点では使用しない。

#### 出力データ構造

```json
{
  "_phase4": {
    "s_propositions": [
      {"id": "S1", "label": "ある男がいた", "type": "atomic"},
      {"id": "S10", "label": "男はバーの外から来た", "type": "derived", "derived_from": ["S3"]}
    ],
    "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"],
    "tactical_chains": [
      {
        "id": "C1",
        "type": "S_to_root",
        "source": ["S5"],
        "target": "P1",
        "intermediates": ["I1"],
        "tactic": "動機の探索 → 段階的絞り込み",
        "description": "S5 →[動機の探索]→ I1(水を求めた理由がある) →[段階的絞り込み]→ P1"
      },
      {
        "id": "C5",
        "type": "toward_concrete",
        "source": ["P1", "S5"],
        "target": "P2",
        "intermediates": ["I5"],
        "tactic": "観測事実の深掘り → 段階的絞り込み",
        "description": "P1+S5 →[観測事実の深掘り]→ I5(困りごとと水の関係) →[段階的絞り込み]→ P2",

```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `initial_confirmed`: ゲーム開始時の確認済み命題 ID リスト（S 原子命題 + 導出命題）
- `_phase4`: Phase 固有メタデータ
  - `s_propositions`: S 命題の配列。各要素:
    - `id`: S 命題 ID（S*）
    - `label`: 命題のラベル
    - `type`: `"atomic"` | `"derived"`
    - `derived_from`: 導出元の命題 ID リスト（derived の場合のみ）
  - `initial_confirmed`: initial_confirmed のコピー（トップレベルと同一）
  - `tactical_chains`: 戦術連鎖の配列。各要素:
    - `id`: 連鎖 ID（C*）
    - `type`: `"S_to_root"` | `"toward_concrete"`
    - `source`: 起点命題 ID リスト
    - `target`: 到達目標の分割命題 ID
    - `intermediates`: 連鎖内で導入された中間命題 ID リスト（**最低 1 つ**）
    - `tactic`: 適用する汎用戦術名
    - `description`: 連鎖の記述（各ステップの出力は命題 ID で記述）
    - `fc_source`: 起点の由来（toward_concrete の場合）
  - `generated_intermediates`: 本 Phase で導入された全中間命題の ID リスト

#### 規則

- S の細分化は分割不能なレベルまで行う
- 導出は論理的含意のみ。「不満なく帰った」のような推測を含めない
- 戦術連鎖は S の情報構造から自然に生まれるものであり、T からの逆算ではない
- 各連鎖ステップは `structure/汎用戦術.md` の戦術を使用する
- **各戦術ステップにつき最低 1 つの中間命題を導入すること（中間命題数 ≥ 戦術ステップ数。target は中間命題に含めない）**
- 各戦術ステップの出力は命題 ID であること（非形式テキストは不可）
- 各連鎖の起点の由来を明示すること
- **fc は連鎖設計の結果として確定すること（仮置きのまま残さない）**
- **fc は連鎖内の直前ステップを参照すること**: target の fc は最後の intermediate を、各 intermediate の fc は直前のステップ（source または前の intermediate）を参照する。source を直接参照し intermediate をスキップする fc は不正（検証: `check_chain_consistency.py`）

### Phase 5: 質問生成（連鎖ベース）

規則: `rules/質問生成_v4.md`

#### 目的

戦術連鎖に沿って質問を生成する。各質問の fc（形成条件）は前段の reveals から自然に定まる。

#### 入力

Phase 1〜4 の全出力

#### 手順

##### Step 1: 連鎖ごとの質問展開

各戦術連鎖について、連鎖のステップを質問に展開する。

- 排除質問（回答: いいえ）: 常識的な仮説を排除し、探索方向を示す
- 確認質問（回答: はい）: 分割命題を確認する

##### Step 2: fc の設定

各質問の fc を、連鎖内の位置に基づいて設定する。fc は段階的オープンを実現するための順序制約であり、各質問は直前のステップの reveals を fc に含めなければならない。

- **連鎖の入口質問**（各連鎖の最初の質問のみ）:
  - S_to_root 連鎖: fc = {起点の S 命題群}（常にオープン）
  - toward_concrete 連鎖: fc = {前段の連鎖で reveals された P 命題群}
- **連鎖内の後続質問**（入口以外の全質問）: fc に直前の質問で reveals された中間命題を含める。連鎖 source → I1 → target の場合:
  - I1 の質問: fc = {source}（入口）
  - target の質問: fc に I1 を含める（I1 確認後にオープン）

**target の fc が source のみを参照し intermediates をスキップしてはならない。** fc が intermediate を参照しないと段階的オープンが機能しない（全質問が初手からオープンになる）。

##### Step 3: reveals の設定

- 分割命題を確認する質問: reveals = [確認される P 命題]
- 中間ステップの質問で名前付き命題を reveal しない場合: 中間命題を明示的に導入するか、後続質問の prerequisites で処理する

##### Step 4: 分割命題の formation_conditions 確定

各分割命題の formation_conditions を、連鎖内の直前ステップから確定する。

- 連鎖 source → I1 → ... → In → target において:
  - target.fc = [[In]]（最後の intermediate）
  - Ik.fc = [[Ik-1]]（直前の intermediate。k=1 の場合は source）
- **fc は連鎖の source ではなく、直前のステップを参照すること。** source を直接参照すると intermediate がゲートとして機能しない

##### Step 5: 否定命題の導入

回答が「いいえ」の質問について、reveals される命題の否定対象（negation_of が指す命題）を定義する。

- 「いいえ」質問が reveal する命題（N*）に対して、その肯定版の仮説命題（H*）を作成する
- N*.negation_of = H*.id, H*.negation_of = N*.id（対称関係）
- H* の formation_conditions を設定する（質問の利用可能条件として使用される）

##### Step 5.5: prerequisites の設定

全質問について、質問文が言語的に前提とする情報を特定し、prerequisites フィールドに設定する。initial_confirmed に含まれない前提概念に対応する命題を prerequisites に追加する。判定基準は `precedents/質問文の含意の判例.md` に従う。

##### Step 6: 探索パスの構成

複数の探索パス（異なる入口から開始）を構成し、合理性を確認する。

- 最低 3 パスを構成する
- 各パスが合理的であること（各質問の開放が想起により説明でき、唐突な飛躍がない）

#### 出力データ構造

```json
{
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "_truth_value": true,
      "formation_conditions": [["S5"]],
      "rejection_conditions": null
    },
    {
      "id": "N1a",
      "label": "男が水を求めたのは喉の渇きからではない",
      "_truth_value": true,
      "negation_of": "H1a",
      "formation_conditions": [["S5"]],
      "rejection_conditions": null
    },
    {
      "id": "H1a",
      "label": "男が水を求めたのは喉の渇きからだった",
      "_truth_value": false,
      "negation_of": "N1a",
      "formation_conditions": [["S5"]],
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1a",
      "text": "男が水を求めたのは喉が渇いたからですか？",
      "answer": "いいえ",
      "reveals": ["N1a"],
      "mechanism": "observation",
      "prerequisites": []
    },
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",

      "prerequisites": []
    }
  ],
  "_phase5": {
    "chain_mapping": {
      "C1": ["Q1a", "Q1b"]
    },

      {
        "name": "パス A: 水の動機から入る",
        "sequence": "Q1a(いいえ) → Q1b(はい:P1) → ..."
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `propositions`: 分割命題 + 中間命題 + 否定対象命題 + FP の配列。各要素は `Proposition` 型（`models.py` 参照）
  - `id`: 命題 ID（P* は分割命題、N* は中間命題、H* は否定対象命題、FP は最終命題）
  - `label`: 命題のラベル（文字列）
  - `negation_of`: `str | null`。否定関係にある命題の ID（対称的）
  - `formation_conditions`: `list[list[str]]`（OR of AND）。Phase 4 の連鎖構造から確定
  - `rejection_conditions`: `list[list[str]] | null`。Phase 8 で設定するため、本 Phase では null
  - `_truth_value`: T における真偽（エクスポート時に除去される）
- `questions`: 質問の配列。各要素は `Question` 型（`models.py` 参照）
  - `id`: 質問 ID
  - `text`: 質問文
  - `answer`: 「はい」or「いいえ」
  - `reveals`: `list[str]`。回答で明らかになる命題 ID 群
  - `mechanism`: `"observation"` | `"link"` | `"anomaly"`
  - `prerequisites`: `list[str]`。質問文の言語的前提となる命題 ID 群
- `_phase5`: Phase 固有メタデータ
  - `chain_mapping`: 各連鎖 ID → 対応する質問 ID リスト
  - `exploration_paths`: 探索パスの記録

#### 規則

- 回答は T の事実と矛盾しないこと
- 各質問は単一の推論ステップに対応すること
- **全質問が reveals を持つこと**（reveals: [] は原則あり得ない）
- reveals のない中間ステップは、中間命題を導入して reveals を持たせる
- 探索パスは最低 3 つ検証する
- `reveals` は `list[str]` 型であること（文字列ではなくリスト）

### Phase 6: 検証

規則: 工程1では `rules/検証_工程1_v3.md`、工程3では `rules/検証_工程3_v3.md`

#### 目的

大域目標の達成と各種整合性を検証する。工程1では真側のみ、工程3では棄却側を含む全体を検証する。

#### 入力

工程1: Phase 1〜5 の全出力
工程3: Phase 1〜5, 7, 8 の全出力

#### 手順

##### Step 1: 構造的整合性

- 全命題の ID が一意であること
- 全質問の reveals が存在する命題 ID を参照していること
- formation_conditions / negation_of が存在する命題 ID を参照していること
- initial_confirmed の命題が全て定義されていること

##### Step 2: 利用可能条件の到達可能性

- 全質問の利用可能条件が初期確認済み命題（initial_confirmed）から到達可能であること
  - 利用可能条件は reveals 先命題の fc（「いいえ」回答の場合は negation_of 先命題の fc）
  - fc に含まれる P 命題が、先行する質問の reveals に含まれること
  - fc に含まれる S 命題が initial_confirmed に含まれること
- 循環依存がないこと

##### Step 3: reveals の網羅性

- 全分割命題が少なくとも 1 つの質問の reveals に含まれること
- 棄却可能な全モデルが、対応する分割命題の質問回答により棄却されること

##### Step 4: 否定関係の整合性

- 「いいえ」回答の質問が reveal する命題に negation_of が設定されていること
- negation_of が指す命題が存在し、formation_conditions を持つこと
- negation_of 関係が対称的であること（P.negation_of = Q ⇒ Q.negation_of = P）

##### Step 5: 回答の T 整合性

- 全質問の回答（はい/いいえ）が T の事実と矛盾しないこと
- 分割命題の truth_value が T の内容と整合すること

##### Step 6: 導出条件の 1 段階推論原則

- 全命題の formation_conditions について、各 AND グループが単一の推論ステップとして妥当であること
- 結論から逆算した不自然な条件がないこと

##### Step 7: パストレース

複数の質問選択順でゲームをシミュレーションする。

1. confirmed = initial_confirmed から開始
2. 仮説導出（formation_conditions 1回パス）→ derived 計算
3. known = confirmed ∪ derived
4. 利用可能条件が known で満たされる質問を列挙（オープン質問）
5. オープン質問から 1 つ選択 → reveals を confirmed に追加
6. 2〜5 を繰り返し、全分割命題が confirmed になるまで

確認事項:
- **到達可能性**: いかなる選択順でも T 近傍に到達可能であること
- **行き詰まり**: オープン質問がなくなり、かつ分割命題が未確認の状態が発生しないこと
- **想起の合理性**: 各質問の開放が唐突な飛躍を含まないこと

最低 3 パスを異なる入口から検証する。

##### Step 8: 棄却の完全性

- 棄却可能な全モデルが、探索パスの終了時に棄却されていること
- 残存するモデルが棄却不可能モデルのみであること

##### Step 8.5: 連鎖構造と fc の整合性

全連鎖について、target および intermediates の fc が連鎖内の直前ステップを参照しているかを検証する。

- 連鎖 source → I1 → ... → In → target において:
  - target.fc が In を参照していること
  - 各 Ik.fc が Ik-1（k=1 の場合は source）を参照していること
- target.fc が source のみを参照し intermediates をスキップしている場合は FAIL

検証: `check_chain_consistency.py` の「連鎖構造と fc の整合性」チェック

#### 検証条件

全ステップが PASS であること。FAIL の場合、エラー内容に応じて原因フェーズに差し戻す。

| 検証項目 | FAIL 時の差し戻し先 |
|---|---|
| 構造的整合性 | 該当フェーズ |
| 利用可能条件の到達可能性 | Phase 4 または 5 |
| reveals の網羅性 | Phase 5 または Phase 8 |
| 否定関係の整合性 | Phase 5 または Phase 8 |
| 回答の T 整合性 | Phase 5 または Phase 8 |
| 1 段階推論原則 | Phase 4 または 5 |
| 連鎖構造と fc の整合性 | Phase 4 または 5 |
| パストレース | Phase 4 または 5 |
| 棄却の完全性 | Phase 2 または Phase 8 |

### Phase 7: ブラッシュアップ

規則: `rules/戦術連鎖のブラッシュアップ_v2.md`

#### 目的

工程1の真側を磨き上げる。(1) 各質問の利用可能条件に跳躍がないかレビューし解消する。(2) fc に含まれる N*/NF* を「はい」命題に置き換える。

#### 入力

工程1（Phase 1〜6）の全出力

#### 手順

反復的に実施する。各反復で:

1. 全質問の利用可能条件を確認し、跳躍（想起の不自然さ）がないかレビューする
2. 跳躍が見つかった場合、戦術連鎖を修正して解消する
3. 全命題の fc を確認し、N*/NF* が含まれている場合は「はい」命題に置き換える
4. 修正に伴い propositions / questions を更新する

#### 規則

- 修正は戦術連鎖レベルで行い、質問単体の調整ではなく連鎖全体の整合性を保つ
- 跳躍がなくなり、かつ fc に N*/NF* が含まれなくなるまで反復する
- N*/NF* の置き換えは、除去ではなく代替「はい」命題の検討を先に行う

### Phase 8: 連鎖内分岐分析

規則: `rules/連鎖内分岐分析_v2.md`（質問生成時は `rules/質問生成_v4.md` も参照）

#### 目的

真側連鎖の各ステップにおける並列命題を分析し、モデルとの対応を確認した上で、全ての並列命題から NF 命題と QF 質問を導出する。

#### 入力

Phase 1〜5, 7 の全出力

#### 手順

##### Step 1: 連鎖内分岐の分析

真側連鎖の各ステップについて、並列命題とモデルとの対応を分析する。

- 各ステップの真側命題に対して自然に並列に並ぶ命題を特定する
- 各並列命題に対応するモデルを `_phase2.model_propositions` と照合して特定する
- 連鎖の全ステップを対象とする（最初のステップも省略しない）

##### Step 2: カバレッジ確認

全競合モデルが、連鎖内分岐のいずれかの並列命題に対応しているかを確認する。

- Step 1 の対応モデルを集約し、全競合モデルと照合する
- 未カバーモデルがあれば記録する（対応はこの時点では決定しない）

##### Step 3: 質問生成

Step 1 で特定した並列命題から質問を導出する。

- 全ての並列命題について命題（`NF*`）を定義し、formation_conditions を設定する
- 各命題に対応する質問（`QF*`）を生成する
- 命題と質問の生成は `rules/質問生成_v4.md` の規則に従うこと

#### 出力データ構造

```json
{
  "_phase8": {
    "chain_branches": [
      {
        "chain_id": "C1",
        "chain_path": "S → N1a → I1 → P1",
        "branches": [
          {
            "at_step": "N1a",
            "true_side_label": "味の不味さが理由ではない",
            "parallels": [
              {"label": "味の不味さが理由", "models": ["M7"]}
            ]
          }
        ]
      }
    ],
    "coverage": {
      "all_models": ["M1", "M2"],
      "model_coverage": {
        "M1": [{"chain": "C3", "step": "N4a"}]
      },
      "uncovered": [],
      "uncovered_notes": ""
    },
    "generated": {
      "propositions": [
        {"id": "NF1a", "source_chain": "C1", "source_step": "N1a", "parallel_index": 0}
      ],
      "questions": [
        {"id": "QF1a", "reveals_proposition": "NF1a"}
      ]
    }
  }
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `_phase8`: Phase 固有メタデータ
  - `chain_branches`: 各連鎖の分岐分析（Step 1 出力）
  - `coverage`: カバレッジ結果（Step 2 出力）
  - `generated`: 生成した命題・質問のトレーサビリティ（Step 3 出力）
- `propositions` への追加: 分岐命題（`NF*`）。Phase 5 で生成した配列に追加する
- `questions` への追加: 分岐質問（`QF*`）。Phase 5 で生成した配列に追加する

#### 規則

- 分岐の分析は戦術連鎖（命題と戦術）のレベルで行うこと。質問レベルで考えない
- 並列命題は真偽を問わず自然に並ぶものを探すこと
- **各連鎖の全ステップに最低 1 つの分岐を持つこと（分岐 0 個のステップは不合格）**
- カバーされないモデルへの対応を強制しない
- 質問は `rules/質問生成_v4.md` の規則に従うこと

### Phase 10: 調整

規則: `rules/調整.md`

#### 目的

全命題・全質問が確定した状態で、2 つの調整を行い質問数を削減する。

- **Step A（間引き）**: 概念的距離が不十分な冗長命題を間引く（命題と質問を削除）
- **Step B（演繹関係の特定）**: 論理的結合で導出できる命題に entailment_conditions を設定する（質問は残るがエンジンが自動的に非表示にする）

#### 入力

Phase 1〜8 の全出力（全質問・命題が確定した状態）

#### 手順

Step A → Step B の順に実施する。

**Step A（間引き）**:
1. 全導出関係の概念的距離を評価し、冗長ペアを特定する
2. 間引く側を判断する（デフォルト: 導出元を間引く）
3. fc の付け替え・prerequisites 付け替え・命題削除・質問削除・連鎖更新・分岐更新を実施する
4. 再検証する

**Step B（演繹関係の特定）**:
1. 間引き後の全命題について、命題の意味的内容に基づき演繹関係を特定する（複数命題の情報の合成のみで得られ、推論が介在しないもの）
2. 該当命題に entailment_conditions を設定する（formation_conditions はそのまま残す）
3. 再検証する

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `_phase10`: Phase 固有メタデータ
  - `step_a_thinning`: 間引き結果（`thinned`, `summary`）
  - `step_b_entailment`: 演繹関係の設定結果（`entailed`）

#### 規則

- Step A: デフォルトで導出元を間引く。間引き後に量的最低基準を満たさなくてもよい
- Step B: 演繹関係は命題の意味的内容に基づき判定する（fc の構造分析ではない）。推論を含む導出は演繹としない

### Phase 9: 質問分類

規則: `rules/質問分類.md`

#### 目的

全質問にプレイヤー向けのトピックカテゴリを付与する。プレイヤーが自分の思いついた質問を効率的に探せるようにすることを目的とする。ゲームロジックには影響しない後処理である。

#### 入力

Phase 1〜8 の全出力（全質問が確定した状態）

#### 手順

##### Step 1: 戦術ベースのカテゴリ付与

各質問の対応する戦術ステップ（`_phase5.chain_mapping` → `_phase4.tactical_chains` の `tactic`）を参照し、固定カテゴリセットに基づいて `topic_category` を付与する。固定カテゴリセットを `topic_categories` としてトップレベルに追加する。

固定カテゴリセット:

| カテゴリ | 対応する戦術 |
|---------|------------|
| 状況の確認 | 直接同定(0)・観測事実の深掘り(2)・段階的絞り込み(10) |
| 原因と動機 | 因果の逆転(5)・動機の探索(8) |
| 人物と背景 | 時間軸の拡張(3)・登場人物の探索(4)・人物像の探索(9) |
| 前提の確認 | 常識的仮説の排除(1)・矛盾の指摘(6)・ラベルと実体の乖離(7) |

##### Step 2: 棄却質問のカテゴリ付与

棄却質問（QF*）は元の連鎖ステップの戦術、または質問テキストの内容に基づいてカテゴリを決定する。

##### Step 3: 情報漏洩検証

1. カテゴリと命題の偏り検証
2. カテゴリの空白検証（ゲーム開始時に全カテゴリに初期オープン質問があること）

##### Step 4: バランス確認

各カテゴリの質問数を集計し、参考情報として記録する。

#### 出力データ構造

```json
{
  "topic_categories": [
    {"id": "状況の確認", "name": "状況の確認"},
    {"id": "原因と動機", "name": "原因と動機"},
    {"id": "人物と背景", "name": "人物と背景"},
    {"id": "前提の確認", "name": "前提の確認"}
  ],
  "_phase9": {
    "assignments": [
      {"question_id": "Q1a", "tactic": "常識的仮説の排除", "category": "前提の確認"}
    ],
    "initial_open_distribution": {
      "状況の確認": ["Q3a"],
      "原因と動機": ["Q2a"],
      "人物と背景": [],
      "前提の確認": ["Q1a"]
    },
    "balance": {
      "状況の確認": 5,
      "原因と動機": 4,
      "人物と背景": 3,
      "前提の確認": 7
    }
  }
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `topic_categories`: トップレベルキー。固定カテゴリセットの配列
- `_phase9`: Phase 固有メタデータ
  - `assignments`: 各質問のカテゴリ割り当て根拠
  - `initial_open_distribution`: ゲーム開始時のカテゴリ別初期オープン質問
  - `balance`: カテゴリ別質問数
- 各質問の `topic_category` フィールドへの値の設定

#### 規則

- カテゴリは固定セット（パズル非依存）とし、パズルごとに新規カテゴリを追加しない
- カテゴリの付与は質問の内容を変更しない（後処理）
- 情報漏洩検証で FAIL の場合、カテゴリの再割り当てで対処する（質問自体は変更しない）

## フェーズ間の依存

```
工程1: 真側の構築
  Phase 1（緊張構造分析とモデル列挙）
    → Phase 2（モデル具象命題と分割命題）
      → Phase 3（導出構造と最終命題）
        → Phase 4（S 細分化と戦術連鎖）
          → Phase 5（質問生成）
            → Phase 6（検証: 真側）

工程2: ブラッシュアップ
  Phase 7（戦術連鎖のブラッシュアップ、反復）

工程3: 棄却側の構築
  Phase 8（連鎖内分岐分析）
    → Phase 6（検証: 全体）

工程4: 間引き
  Phase 10（冗長命題の間引き）

工程5: 質問分類
  Phase 9（質問分類）
```

Phase 1〜2 が T 側の骨格を構成し、Phase 3 で導出構造と最終命題（FP）を構築してクリア条件を確定する。Phase 4 で S との接続を設計し、Phase 5 で質問を生成する。工程1の末尾で真側の検証を行う。工程2で想起条件の跳躍を磨き上げる。工程3では Phase 8 で真側連鎖の分岐を分析して棄却側の質問を生成し、Phase 6 で全体を検証する。工程4では概念的距離が不十分な命題ペアを間引いて冗長性を排除する。工程5では全質問にトピックカテゴリを付与する。

## 最終データ形式

### data_src.json

全フェーズの中間生成物を含む。各フェーズの出力は `_phase*` キーに格納される。

### data.json

ゲーム実行に必要なデータのみを含む。

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "negation_of": null,
      "formation_conditions": [["S5"]],
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",
      "answer": "はい",
      "reveals": ["P1"],
      "mechanism": "observation",
      "prerequisites": [],
      "topic_category": "状況の確認"
    }
  ],
  "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"],
  "clear_conditions": [["FP"]],
  "topic_categories": [
    {"id": "状況の確認", "name": "状況の確認"},
    {"id": "原因と動機", "name": "原因と動機"},
    {"id": "人物と背景", "name": "人物と背景"},
    {"id": "前提の確認", "name": "前提の確認"}
  ]
}
```

- `propositions`: 分割命題 + 中間命題 + FP。S 命題は含めない（initial_confirmed で管理）
- `questions`: 全質問は `reveals` を持つ。`reveals` は `list[str]` 型
- `clear_conditions`: `[["FP"]]` 形式（FP が confirmed になればクリア）
- フィールド定義は `app/poc_v3/src/models.py` の `Proposition` / `Question` クラスに準拠
- `_` プレフィックスのフィールド（`_truth_value`, `_phase*` 等）はエクスポート時に除去される
