# 統合アルゴリズム v3

## 概要

モデル集合の分割構造を起点に、戦術連鎖・質問生成を構築した後、棄却分析に基づく分岐連鎖を構築するアルゴリズム。

### v2 からの主要な変更

- **抽象の葉・パターン・代表モデルの廃止**: モデル具象命題の横断比較から分割命題を直接導出する。T の抽象化を経由しない
- **中間命題の導出方法の変更**: モデル集合の分割構造から導出する。抽象化は分割後の磨き工程として位置づける
- **質問生成の戦術連鎖への統合**: 戦術連鎖が reveals → fc の連鎖を直接定義し、質問の fc が自然に決まる
- **棄却分析の後置**: 棄却分析を T 方向の連鎖構築後に移動。分析と連鎖構築を分離した 2 Phase 構成
- **Phase 構成の再編**: 棄却分析(Phase 5)→偽側戦術連鎖構築(Phase 6)→偽側質問生成と分岐統合(Phase 7)→検証(Phase 8)の 4 Phase 後段構成

### 維持する原則

- **二段階導出**: entailment_conditions（confirmed→confirmed 不動点）と formation_conditions（confirmed→derived 1回パス）。`structure/仮説導出.md` 参照
- **recall_conditions = reveals される命題の formation_conditions**
- **仮説連鎖の廃止**: formation_conditions は confirmed のみ参照。derived を参照しない
- **1 段階推論の原則**: 各導出条件は単一の推論ステップを表現する
- **命題の独立性・客観性**: 全命題は自己完結的で客観的に検証可能
- **T 忠実性**: 生成する命題は T の情報から逸脱しない

## 入力

- S: 問題文
- T: 真相

## 出力

- data_src.json: 全フェーズの中間生成物を含む
- data.json: ゲームデータ

## 大域目標

**複数の質問選択順において合理的なパスで T に到達できること**

合理的なパスとは、各質問の開放が想起（確認済み命題または導出命題）により説明でき、唐突な飛躍がないパスを指す。

## Phase 構成

### Phase 1: 緊張構造分析とモデル列挙

規則: `rules/緊張構造分析_v3.md`、`rules/モデル列挙.md`

#### 目的

S に含まれる緊張構造を特定し、T および競合モデルを列挙する。

#### 入力

S, T

#### 手順

1. S の緊張構造を抽出する
   - 予測の失敗: S の事象から常識的に予測される結果と実際の乖離
   - 前提の崩壊: S が暗黙に想定させる前提の不成立
   - 説明不能な点: S の事象間で因果関係が見えない箇所
2. T がそれらをどう緩和するか確認する
3. 緊張の部分集合ごとに、S と整合する競合モデルを列挙する
   - 全緊張を緩和するものから、一部のみ緩和するものまで
   - 各モデルは T とは異なる具象的なストーリーを持つ

#### 出力データ構造

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "_phase1": {
    "tensions": [
      {"id": "α", "label": "緊張の記述"}
    ],
    "models": [
      {
        "id": "T",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α", "β", "γ"]
      },
      {
        "id": "M1",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α", "β", "γ"]
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `title`: パズル名（文字列）
- `S`: 問題文（文字列）
- `T`: 真相（文字列）
- `_phase1`: Phase 固有メタデータ
  - `tensions`: 緊張の配列。各要素は `{id, label}`
  - `models`: モデルの配列。各要素は `{id, label, summary, tension_subset}`
    - `id`: モデル ID（T, M1, M2, ...）
    - `label`: モデル名
    - `summary`: 1〜2 文の概要
    - `tension_subset`: 緩和する緊張 ID のリスト

#### 規則

- 競合モデルは最低 10 以上列挙する
- 各モデルは S の全事実と整合すること
- 緊張の部分集合の分類により、モデルの網羅性を担保する
- T も models に含め、他のモデルと同列に扱う

### Phase 2: モデル具象命題と分割構造

規則: `rules/モデル具象命題と分割構造.md`

#### 目的

各モデルの具象命題を記述し、モデル横断で共通命題を抽出して、モデル集合の階層的分割構造を導出する。

#### 入力

Phase 1 の出力

#### 手順

1. T および全競合モデルについて、具象命題を列挙する
   - 各モデルの因果構造を命題として明示する
2. モデル間で共通する命題を抽出する
   - 同じ事実を共有するモデル群を特定する
   - カテゴリは「人物の状態」「他者の認識と行動」「解消メカニズム」等、パズルの構造に応じて自然に定まる
3. 共通命題によるモデル集合の分割を構成する
   - 上位分割（大きく分ける命題）と下位分割（さらに細分する命題）を階層として整理する

#### 出力データ構造

```json
{
  "_phase2": {
    "model_propositions": {
      "T": ["男はしゃっくりが止まらなかった", "..."],
      "M1": ["男は何かが喉に詰まっていた", "..."]
    },
    "division_propositions": [
      {
        "id": "P1",
        "label": "男は何らかの症状・状態に困っていた",
        "truth_value_in_T": true,
        "true_models": ["T", "M1", "M2", "M3", "M4", "M7", "M8"],
        "false_models": ["M5", "M6", "M9", "M10", "M11", "M12", "M13", "M14", "M15"]
      }
    ],
    "division_hierarchy": [
      {
        "proposition_id": "P1",
        "true_set": ["T", "M1", "M2", "M3", "M4", "M7", "M8"],
        "false_set": ["M5", "M6", "M9", "M10", "M11", "M12", "M13", "M14", "M15"],
        "children": [
          {
            "proposition_id": "P2",
            "true_set": ["T", "M1", "M2", "M3", "M7", "M8"],
            "false_set": ["M4"],
            "children": []
          }
        ]
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `_phase2`: Phase 固有メタデータ
  - `model_propositions`: モデル ID → 具象命題ラベルのリスト（辞書）
  - `division_propositions`: 分割命題の配列。各要素:
    - `id`: 分割命題 ID（P*）
    - `label`: 分割命題のラベル
    - `truth_value_in_T`: T における真偽（必ず true）
    - `true_models`: 真であるモデル ID リスト
    - `false_models`: 偽であるモデル ID リスト
  - `division_hierarchy`: 分割の木構造。各ノード:
    - `proposition_id`: 分割命題 ID
    - `true_set`: 真側モデル ID リスト
    - `false_set`: 偽側モデル ID リスト
    - `children`: 子ノードの配列（同じ構造の再帰）

#### 規則

- 分割命題は T において全て真であること
- 具象命題はモデルの因果構造を反映すること（単なる事実の羅列ではなく因果関係を含む）
- 分割命題はモデルの具象命題から抽象化してよいが、T の事実に忠実であること
- 分割の階層は上位（棄却力が大きい）から下位（棄却力が小さい）へ整理する

### Phase 3: S 細分化と戦術連鎖

規則: `rules/S細分化.md`、`rules/戦術連鎖_v3.md`

#### 目的

S を原子命題に分解し、S 命題から分割命題への戦術連鎖を構成する。連鎖の reveals → fc 関係が質問の formation_conditions を定める。

#### 入力

Phase 1〜2 の出力、`structure/汎用戦術.md`

#### 手順

##### Step 1: S 細分化

S を分割不能な原子命題に分解する。論理的含意による導出命題も列挙する。

- 原子命題: S の文から直接取り出せる最小の事実
- 導出命題: 原子命題から論理的に含意されるもののみ。推測・解釈を含めない
- 自明な含意（「バーがあった」等）も見落とさないこと

##### Step 2: S → 上位分割命題への戦術連鎖

S の原子命題を起点とし、上位の分割命題に至る戦術連鎖を構成する。

- 各連鎖は `structure/汎用戦術.md` の戦術を適用する
- S の情報構造から自然に生まれるものであり、T からの逆算ではない

##### Step 3: 上位 → 下位分割命題への戦術連鎖

上位分割命題の確認を起点とし、下位の分割命題に至る戦術連鎖を構成する。

- 前段の連鎖の reveals が次の連鎖の fc になる
- 連鎖間の依存関係が質問の formation_conditions を自然に定める

本 Phase では各命題に対して 1 つの導出経路（fc の単一 AND グループ）を設計する。fc の OR 構造（複数の代替導出経路）はこの時点では使用しない。代替経路の設計は Phase 7（偽側質問生成と分岐統合）で行う。

#### 出力データ構造

```json
{
  "_phase3": {
    "s_propositions": [
      {"id": "S1", "label": "ある男がいた", "type": "atomic"},
      {"id": "S10", "label": "男はバーの外から来た", "type": "derived", "derived_from": ["S3"]}
    ],
    "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"],
    "tactical_chains": [
      {
        "id": "C1",
        "type": "S_to_upper",
        "source": ["S5"],
        "target": "P1",
        "tactic": "動機の探索 → 段階的絞り込み",
        "description": "S5 →[動機の探索]→ 水を求めた理由がある →[段階的絞り込み]→ P1"
      },
      {
        "id": "C5",
        "type": "upper_to_lower",
        "source": ["P1", "S5"],
        "target": "P2",
        "tactic": "観測事実の深掘り → 段階的絞り込み",
        "description": "P1+S5 →[観測事実の深掘り]→ 困りごとと水の関係 →[段階的絞り込み]→ P2",
        "fc_source": "P1 は連鎖 C1 の reveals"
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `initial_confirmed`: ゲーム開始時の確認済み命題 ID リスト（S 原子命題 + 導出命題）
- `_phase3`: Phase 固有メタデータ
  - `s_propositions`: S 命題の配列。各要素:
    - `id`: S 命題 ID（S*）
    - `label`: 命題のラベル
    - `type`: `"atomic"` | `"derived"`
    - `derived_from`: 導出元の命題 ID リスト（derived の場合のみ）
  - `initial_confirmed`: initial_confirmed のコピー（トップレベルと同一）
  - `tactical_chains`: 戦術連鎖の配列。各要素:
    - `id`: 連鎖 ID（C*）
    - `type`: `"S_to_upper"` | `"upper_to_lower"`
    - `source`: 起点命題 ID リスト
    - `target`: 到達目標の分割命題 ID
    - `tactic`: 適用する汎用戦術名
    - `description`: 連鎖の記述
    - `fc_source`: 起点の由来（upper_to_lower の場合）

#### 規則

- S の細分化は分割不能なレベルまで行う
- 導出は論理的含意のみ。「不満なく帰った」のような推測を含めない
- 戦術連鎖は S の情報構造から自然に生まれるものであり、T からの逆算ではない
- 各連鎖ステップは `structure/汎用戦術.md` の戦術を使用する
- 上位→下位連鎖の source には、前段の連鎖で reveal された分割命題を含める

### Phase 4: 質問生成（連鎖ベース）

規則: `rules/質問生成_v3.md`

#### 目的

戦術連鎖に沿って質問を生成する。各質問の fc（形成条件）は前段の reveals から自然に定まる。

#### 入力

Phase 1〜3 の全出力

#### 手順

##### Step 1: 連鎖ごとの質問展開

各戦術連鎖について、連鎖のステップを質問に展開する。

- 排除質問（回答: いいえ）: 常識的な仮説を排除し、探索方向を示す
- 確認質問（回答: はい）: 分割命題を確認する

##### Step 2: fc の設定

各質問の fc を、連鎖の構造から設定する。

- 連鎖の起点が S 命題の場合: fc = {S 命題群}（常に確認済み）
- 連鎖の起点が分割命題の場合: fc = {前段の reveals に含まれる P 命題群}
- 連鎖内の後続質問: fc に前の質問の回答確認（中間命題または prerequisites）を含める

##### Step 3: reveals の設定

- 分割命題を確認する質問: reveals = [確認される P 命題]
- 中間ステップの質問で名前付き命題を reveal しない場合: 中間命題を明示的に導入するか、後続質問の prerequisites で処理する

##### Step 4: 分割命題の formation_conditions 確定

各分割命題の formation_conditions を、それを reveal する質問の fc から確定する。

- P の formation_conditions = P を reveal する質問の fc
- recall_conditions = reveals される命題の formation_conditions（維持原則）

##### Step 5: 探索パスの構成

複数の探索パス（異なる入口から開始）を構成し、合理性を確認する。

- 最低 3 パスを構成する
- 各パスが合理的であること（各質問の開放が想起により説明でき、唐突な飛躍がない）

#### 出力データ構造

```json
{
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "_truth_value": true,
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    },
    {
      "id": "N1a",
      "label": "男が水を求めたのは喉の渇きからではない",
      "_truth_value": true,
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1a",
      "text": "男が水を求めたのは喉が渇いたからですか？",
      "answer": "いいえ",
      "recall_conditions": [["S5"]],
      "reveals": ["N1a"],
      "mechanism": "observation",
      "prerequisites": []
    },
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",
      "answer": "はい",
      "recall_conditions": [["S5"]],
      "reveals": ["P1"],
      "mechanism": "observation",
      "prerequisites": []
    }
  ],
  "_phase4": {
    "chain_mapping": {
      "C1": ["Q1a", "Q1b"]
    },
    "exploration_paths": [
      {
        "name": "パス A: 水の動機から入る",
        "sequence": "Q1a(いいえ) → Q1b(はい:P1) → ..."
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `propositions`: 分割命題 + 中間命題の配列。各要素は `Proposition` 型（`models.py` 参照）
  - `id`: 命題 ID（P* は分割命題、N* は中間命題）
  - `label`: 命題のラベル（文字列）
  - `formation_conditions`: `list[list[str]]`（OR of AND）。Phase 3 の連鎖構造から確定
  - `entailment_conditions`: `list[list[str]] | null`。論理的含意がある場合のみ設定
  - `rejection_conditions`: `list[list[str]] | null`。Phase 5 で設定するため、本 Phase では null
  - `_truth_value`: T における真偽（エクスポート時に除去される）
- `questions`: 質問の配列。各要素は `Question` 型（`models.py` 参照）
  - `id`: 質問 ID
  - `text`: 質問文
  - `answer`: 「はい」or「いいえ」
  - `recall_conditions`: `list[list[str]]`（OR of AND）。reveals される命題の formation_conditions と一致
  - `reveals`: `list[str]`。回答で明らかになる命題 ID 群
  - `mechanism`: `"observation"` | `"link"` | `"anomaly"`
  - `prerequisites`: `list[str]`。質問文の言語的前提となる命題 ID 群
- `_phase4`: Phase 固有メタデータ
  - `chain_mapping`: 各連鎖 ID → 対応する質問 ID リスト
  - `exploration_paths`: 探索パスの記録

#### 規則

- **recall_conditions = reveals される命題の formation_conditions**: 質問の recall_conditions は、その質問が reveal する命題の formation_conditions と一致する
- 回答は T の事実と矛盾しないこと
- 各質問は単一の推論ステップに対応すること
- **全質問が reveals を持つこと**（reveals: [] は原則あり得ない）
- reveals のない中間ステップは、中間命題を導入して reveals を持たせる
- 探索パスは最低 3 つ検証する
- `reveals` は `list[str]` 型であること（文字列ではなくリスト）

### Phase 5: 棄却分析

規則: `rules/棄却条件_v3.md`

#### 目的

T 以外の経路を分析する。各競合モデルの棄却条件を整理し、分割階層の偽側を展開する。

#### 入力

Phase 1〜4 の全出力

#### 手順

##### Step 1: 棄却条件の整理

各競合モデルの棄却条件を整理する。

1. 各モデルについて、偽となる分割命題を特定する
2. 最小棄却条件（最も少ない確認で棄却できる命題）を決定する
3. 棄却力（各命題が棄却するモデル数）を評価する
4. 共通命題では棄却不可能なモデル群を特定する
5. 棄却の進行順序（どの命題を確認すればどのモデルが棄却されるか）を整理する

##### Step 2: 分割階層の偽側展開

Phase 2 で構築した分割階層の偽側（false_set）について、モデル集合の特徴を分析する。

- 偽側のモデル群がどのような共通性を持つか
- 偽側の内部でさらに区別可能な構造があるか
- 各分岐点からの偽側の探索方向を明確にする

Phase 2 の分割階層は T 方向（真側）のみ展開されている。本 Step では偽側の構造を補完する。

#### 出力データ構造

```json
{
  "_phase5": {
    "rejection_map": [
      {
        "model_id": "M5",
        "false_propositions": ["P1"],
        "min_rejection": ["P1"]

    ],
    "rejection_power": [

        "proposition_id": "P1",
        "rejects": ["M5", "M6", "M9", "M10", "M11"],

      }
    ],
    "unrejectable_models": {
      "models": ["M1"],
      "reason": "共通命題では棄却不可。T との区別は具象レベルの命題が必要"
    },
    "rejection_progression": [
      {
        "step": 1,
        "confirmed_proposition": "P1",
        "rejected": ["M5", "M6", "M9", "M10", "M11"],

      }
    ],
    "false_side_analysis": [
      {
        "division_point": "P1",

      }

  }

```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `_phase5`: Phase 固有メタデータ
  - `rejection_map`: モデルごとの棄却条件。各要素:
    - `model_id`: モデル ID
    - `false_propositions`: 偽となる分割命題 ID リスト
    - `min_rejection`: 最小棄却条件（命題 ID リスト）
  - `rejection_power`: 分割命題ごとの棄却力。各要素:
    - `proposition_id`: 分割命題 ID
    - `rejects`: 棄却されるモデル ID リスト
    - `count`: 棄却数
  - `unrejectable_models`: 棄却不可能モデル群
    - `models`: モデル ID リスト
    - `reason`: 棄却不可能の理由
  - `rejection_progression`: 棄却の進行順序。各要素:
    - `step`: 段階番号
    - `confirmed_proposition`: 確認される命題 ID
    - `rejected`: 棄却されるモデル ID リスト
    - `remaining`: 残存モデル ID リスト
  - `false_side_analysis`: 偽側の分析。各要素:
    - `division_point`: 分割命題 ID
    - `false_set`: 偽側モデル ID リスト
    - `characteristics`: 偽側モデル群の共通性
    - `internal_structure`: 偽側内部の区別可能な構造

#### 規則

- 棄却条件は分割命題の真偽に基づくこと
- 全競合モデルに対して棄却条件を定義すること
- 棄却不可能モデルの存在を明示的に記録し、根拠を説明すること
- 偽側の分析は分割階層の各レベルで行うこと

### Phase 6: 偽側戦術連鎖構築

規則: `rules/偽側戦術連鎖構築.md`

#### 目的

Phase 5 の棄却分析に基づき、分割階層の偽側に対する戦術連鎖を構築し、偽側の中間命題（formation_conditions 付き）を生成する。Phase 3（真側の戦術連鎖）の偽側版。

#### 入力

Phase 1〜5 の全出力、`structure/汎用戦術.md`

#### 手順

##### Step 1: 偽側の到達目標の設定

各分割命題の偽側について、探索の方向と到達目標を設定する。

- Phase 5 の false_side_analysis を参照し、偽側モデル群の特徴を把握する
- 到達目標: 棄却命題（行き止まり）または真側への合流点
- 各偽側枝の深さ・方向を設計判断として決定する

##### Step 2: 偽側の戦術連鎖の構成

各偽側枝について、汎用戦術を適用して戦術連鎖を構成する。

- 連鎖の起点: 分岐点での排除質問の reveals（否定命題）または S 原子命題
- 連鎖の終点: 棄却命題または真側への合流点
- 各ステップは `structure/汎用戦術.md` の戦術を使用する
- 連鎖は S の情報構造から自然に生まれるものであり、T からの逆算ではない

##### Step 3: 偽側中間命題の生成

連鎖の各ステップに対応する中間命題を生成する。

- 各命題に formation_conditions を設定する（連鎖の前段の reveals から定まる）
- 命題は独立性・客観性・T 忠実性を満たすこと
- 導入時検証（1 次チェック: T 文脈での真偽の明確さ + 意味の確定性）を適用する

#### 出力データ構造

```json
{
  "_phase6": {
    "false_side_chains": [
      {
        "id": "FC1",
        "division_point": "P1",
        "direction": "false",
        "source": ["N1a"],
        "target": "NF1",
        "tactic": "段階的絞り込み",
        "description": "N1a →[段階的絞り込み]→ NF1",
        "fc_source": "N1a は Q1a の reveals"
      }
    ]
  },
  "propositions": [
    {
      "id": "NF1",
      "label": "偽側中間命題のラベル",
      "formation_conditions": [["N1a", "S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null,
      "_truth_value": false
    }
  ]
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `_phase6.false_side_chains`: 偽側の戦術連鎖の配列
  - `id`: 連鎖 ID（FC*）
  - `division_point`: 分岐元の分割命題 ID
  - `direction`: `"false"`（偽側）
  - `source`: 連鎖の起点命題 ID リスト
  - `target`: 連鎖の到達目標命題 ID
  - `tactic`: 適用する汎用戦術名
  - `description`: 連鎖の記述
  - `fc_source`: 起点命題がどの質問の reveals に由来するか
- `propositions` への追加: 偽側中間命題。Phase 4 で生成した配列に追加する
  - フィールドは Phase 4 の命題と同一（`Proposition` 型）
  - `_truth_value`: T における真偽（偽側命題は false が多い）

#### 規則

- 偽側の連鎖は分割階層の各レベルで構築すること
- 連鎖の起点は分岐点の排除質問の reveals または S 原子命題であること
- 各ステップは `structure/汎用戦術.md` の戦術を使用すること
- 中間命題は導入時検証（T 文脈での真偽の明確さ + 意味の確定性）を通過すること
- 枝の深さは設計判断であり、アルゴリズムは固定しない
- 連鎖は一本鎖（直列）である必要はなく、分岐・合流を持つ構造として表現できる

---

### Phase 7: 偽側質問生成と分岐統合

規則: `rules/偽側質問生成と分岐統合.md`

#### 目的

Phase 6 の偽側戦術連鎖に沿って質問を生成し、既存連鎖との分岐・合流を設計する。Phase 4（真側の質問生成）の偽側版 + 分岐統合。

#### 入力

Phase 1〜6 の全出力

#### 手順

##### Step 1: 偽側連鎖の質問展開

Phase 6 で構築した偽側戦術連鎖の各ステップを質問に展開する。

- 排除質問と確認質問を Phase 4 と同じ基準で生成する
- 各質問に recall_conditions を設定する（reveals される命題の formation_conditions と一致）
- 各質問に reveals を設定する（全質問が reveals を持つこと）
- 導入時検証（1 次チェック）を適用する

##### Step 2: 既存連鎖の内部分岐

Phase 3〜4 で構築した既存連鎖の各ステップで、反対の回答に進む枝を設計する。

- 既存のノードから分岐するため、新たな連鎖の構築は不要
- 各質問の回答が反対だった場合の展開を記録する
- 浅い設計で十分。全ステップの完全な分岐設計は不要

##### Step 3: 合流の検討と fc OR 構造

偽側の枝が既存命題に合流する構造を検討する。

- 合流が生じれば、その命題の formation_conditions に代替 AND グループ（OR 構造）が生まれる
- 該当する命題の formation_conditions を更新する
- 合流は自然に生じる場合に取り入れる。強制しない

#### 出力データ構造

```json
{
  "_phase7": {
    "internal_branches": [
      {
        "source_question": "Q1b",
        "branch_answer": "いいえ",
        "leads_to": ["QF1"],
        "description": "P1 確認質問が「いいえ」の場合の展開"
      }
    ],
    "convergence_points": [
      {
        "proposition_id": "P2",
        "new_fc_group": ["NF1", "S5"],
        "source": "偽側からの合流"
      }
    ]
  },
  "questions": [
    {
      "id": "QF1",
      "text": "偽側の質問文",
      "answer": "はい/いいえ",
      "recall_conditions": [["N1a", "S5"]],
      "reveals": ["NF1"],
      "mechanism": "link",
      "prerequisites": ["N1a"]
    }
  ]
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `_phase7.internal_branches`: 既存連鎖の内部分岐の記録
  - `source_question`: 分岐元の質問 ID
  - `branch_answer`: 分岐を発生させる回答
  - `leads_to`: 分岐先の質問 ID リスト
  - `description`: 分岐の説明
- `_phase7.convergence_points`: 合流点の記録
  - `proposition_id`: 合流先の命題 ID
  - `new_fc_group`: 追加される代替 AND グループ
  - `source`: 合流の由来
- `questions` への追加: 偽側の質問。Phase 4 で生成した配列に追加する
  - フィールドは Phase 4 の質問と同一（`Question` 型）
- `propositions` の更新: 合流により formation_conditions に OR グループを追加

#### 規則

- 偽側の質問は Phase 4 と同じ品質基準を満たすこと
- **recall_conditions = reveals される命題の formation_conditions**
- 全質問が reveals を持つこと（`reveals` は `list[str]` 型）
- 既存連鎖の内部分岐は浅く設計すること
- 合流は自然に生じる場合に取り入れること。強制しない
- 合流により生じる fc の OR 構造を propositions に反映すること

### Phase 8: 検証

規則: `rules/検証_v3.md`

#### 目的

大域目標の達成と各種整合性を検証する。

#### 入力

Phase 1〜7 の全出力

#### 手順

##### Step 1: 構造的整合性

- 全命題の ID が一意であること
- 全質問の reveals が存在する命題 ID を参照していること
- fc / recall_conditions / formation_conditions が存在する命題 ID を参照していること
- initial_confirmed の命題が全て定義されていること

##### Step 2: fc 連鎖の整合性

- 全質問の fc が初期確認済み命題（initial_confirmed）から到達可能であること
  - fc に含まれる P 命題が、先行する質問の reveals に含まれること
  - fc に含まれる S 命題が initial_confirmed に含まれること
- 循環依存がないこと

##### Step 3: reveals の網羅性

- 全分割命題が少なくとも 1 つの質問の reveals に含まれること
- 棄却可能な全モデルが、対応する分割命題の質問回答により棄却されること

##### Step 4: recall = fc の整合性

- 各質問について、recall_conditions が reveals される命題の formation_conditions と一致していること
- reveals が空の質問については、recall_conditions が連鎖の source 命題に基づいていること

##### Step 5: 回答の T 整合性

- 全質問の回答（はい/いいえ）が T の事実と矛盾しないこと
- 分割命題の truth_value が T の内容と整合すること

##### Step 6: 導出条件の 1 段階推論原則

- 全命題の formation_conditions について、各 AND グループが単一の推論ステップとして妥当であること
- 結論から逆算した不自然な条件がないこと

##### Step 7: パストレース

複数の質問選択順でゲームをシミュレーションする。

1. confirmed = initial_confirmed から開始
2. 論理的導出（entailment_conditions 不動点）→ confirmed 拡大
3. 仮説導出（formation_conditions 1回パス）→ derived 計算
4. known = confirmed ∪ derived
5. recall_conditions が known で満たされる質問を列挙（オープン質問）
6. オープン質問から 1 つ選択 → reveals を confirmed に追加
7. 2〜6 を繰り返し、全分割命題が confirmed になるまで

確認事項:
- **到達可能性**: いかなる選択順でも T 近傍に到達可能であること
- **行き詰まり**: オープン質問がなくなり、かつ分割命題が未確認の状態が発生しないこと
- **想起の合理性**: 各質問の開放が唐突な飛躍を含まないこと

最低 3 パスを異なる入口から検証する。

##### Step 8: 棄却の完全性

- 棄却可能な全モデルが、探索パスの終了時に棄却されていること
- 残存するモデルが棄却不可能モデル（Phase 5 で特定済み）のみであること

#### 検証条件

全ステップが PASS であること。FAIL の場合、エラー内容に応じて原因フェーズに差し戻す。

| 検証項目 | FAIL 時の差し戻し先 |
|---|---|
| 構造的整合性 | 該当フェーズ |
| fc 連鎖の整合性 | Phase 3, 4, 6, または 7 |
| reveals の網羅性 | Phase 4 または Phase 7 |
| recall = fc の整合性 | Phase 4 または Phase 7 |
| 回答の T 整合性 | Phase 4 または Phase 7 |
| 1 段階推論原則 | Phase 3, 4, 6, または 7 |
| パストレース | Phase 3, 4, 6, または 7 |
| 棄却の完全性 | Phase 2 または Phase 5 |

## フェーズ間の依存

```
Phase 1（緊張構造分析とモデル列挙）
  → Phase 2（モデル具象命題と分割構造）
    → Phase 3（S 細分化と戦術連鎖）
      → Phase 4（質問生成）
        → Phase 5（棄却分析）
          → Phase 6（偽側戦術連鎖構築）
            → Phase 7（偽側質問生成と分岐統合）
              → Phase 8（検証）
```

Phase 1〜2 が T 側の骨格を構成し、Phase 3 で S との接続を設計し、Phase 4 で質問を生成する。Phase 5 で棄却分析を行い、Phase 6 で偽側の戦術連鎖を構築し、Phase 7 で偽側の質問を生成して分岐を統合し、Phase 8 で全体を検証する。

## 最終データ形式

### data_src.json

全フェーズの中間生成物を含む。各フェーズの出力は `_phase*` キーに格納される。

### data.json

ゲーム実行に必要なデータのみを含む。

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",
      "answer": "はい",
      "recall_conditions": [["S5"]],
      "reveals": ["P1"],
      "mechanism": "observation",
      "prerequisites": []
    }
  ],
  "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"]
```

- `propositions`: 分割命題 + 中間命題。S 命題は含めない（initial_confirmed で管理）
