# 統合アルゴリズム v3

## 概要

モデル集合の分割構造を起点に、戦術連鎖・質問生成を構築し、T事実の到達可能性とクリア条件を確定し、真側連鎖の分岐分析を通じて棄却側を構築するアルゴリズム。工程（真側構築→ブラッシュアップ→棄却側構築）による段階的構築を行う。

### v2 からの主要な変更

- **抽象の葉・パターン・代表モデルの廃止**: モデル具象命題の横断比較から分割命題を直接導出する。T の抽象化を経由しない
- **中間命題の導出方法の変更**: モデル集合の分割構造から導出する。抽象化は分割後の磨き工程として位置づける
- **質問生成の戦術連鎖への統合**: 戦術連鎖が reveals → fc の連鎖を直接定義し、質問の fc が自然に決まる
- **棄却側の構築方法の変更**: 偽側の独立連鎖構築を廃止。真側連鎖の各ステップにおける並列命題（分岐）の分析に変更
- **工程の導入**: 工程1（真側構築）→工程2（ブラッシュアップ）→工程3（棄却側構築）の段階的構築
- **Phase 構成の再編**: 旧 Phase 5（棄却分析）・6（偽側連鎖）・7（偽側質問生成）を Phase 9（連鎖内分岐分析）に統合。検証は工程1末尾と工程3末尾の 2 箇所で実施

### 維持する原則

- **二段階導出**: entailment_conditions（confirmed→confirmed 不動点）と formation_conditions（confirmed→derived 1回パス）。`structure/仮説導出.md` 参照
- **recall_conditions = reveals される命題の formation_conditions**
- **仮説連鎖の廃止**: formation_conditions は confirmed のみ参照。derived を参照しない
- **1 段階推論の原則**: 各導出条件は単一の推論ステップを表現する
- **命題の独立性・客観性**: 全命題は自己完結的で客観的に検証可能
- **T 忠実性**: 生成する命題は T の情報から逸脱しない

## 入力

- S: 問題文
- T: 真相

## 出力

- data_src.json: 全フェーズの中間生成物を含む
- data.json: ゲームデータ

## 大域目標

**複数の質問選択順において合理的なパスで T に到達できること**

合理的なパスとは、各質問の開放が想起（確認済み命題または導出命題）により説明でき、唐突な飛躍がないパスを指す。

## 工程構成

Phase 1-9 を 3 つの工程に分け、段階的に構築する。

### 工程1: 真側の構築

| Phase | 内容 |
|-------|------|
| Phase 1 | 緊張構造分析 + モデル列挙 |
| Phase 2 | モデル具象命題 + 分割構造 |
| Phase 3 | S 細分化 + 戦術連鎖 |
| Phase 4 | 質問生成 |
| Phase 5 | T事実の統合 |
| Phase 6 | クリア条件とユニーク要素の導出 |
| Phase 7 | 検証（真側） |

真側だけで完結する。棄却側は扱わない。検証は真側の範囲で実施する（棄却の完全性チェックは除く）。

### 工程2: ブラッシュアップ

| Phase | 内容 |
|-------|------|
| Phase 8 | 戦術連鎖のブラッシュアップ |

工程1の真側を磨き上げる。各質問の想起条件（recall_conditions）に跳躍がないかをレビューし、跳躍がある場合は戦術連鎖を修正して解消する。反復的に実施する。

### 工程3: 棄却側の構築

| Phase | 内容 |
|-------|------|
| Phase 9 | 連鎖内分岐分析 |
| Phase 7 | 検証（全体） |

真側連鎖の分岐を分析し、棄却側の質問を生成する。反復的に実施する（導出と検証を別ファイルに分離し、問題があれば修正→再検証のサイクルを回す）。検証は全体（真側 + 棄却側）の範囲で実施する。

### 工程4: 質問分類

| Phase | 内容 |
|-------|------|
| Phase 10 | 質問分類 |

全質問にプレイヤー向けのトピックカテゴリを付与する。戦術ベースの固定カテゴリセットを使用し、情報漏洩検証とバランス確認を行う。ゲームロジックには影響しない後処理である。

## Phase 構成

### Phase 1: 緊張構造分析とモデル列挙

規則: `rules/緊張構造分析_v3.md`、`rules/モデル列挙.md`

#### 目的

S に含まれる緊張構造を特定し、T および競合モデルを列挙する。

#### 入力

S, T

#### 手順

1. S の緊張構造を抽出する
   - 予測の失敗: S の事象から常識的に予測される結果と実際の乖離
   - 前提の崩壊: S が暗黙に想定させる前提の不成立
   - 説明不能な点: S の事象間で因果関係が見えない箇所
2. T がそれらをどう緩和するか確認する
3. 緊張の部分集合ごとに、S と整合する競合モデルを列挙する
   - 全緊張を緩和するものから、一部のみ緩和するものまで
   - 各モデルは T とは異なる具象的なストーリーを持つ

#### 出力データ構造

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "_phase1": {
    "tensions": [
      {"id": "α", "label": "緊張の記述"}
    ],
    "models": [
      {
        "id": "T",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α", "β", "γ"]
      },
      {
        "id": "M1",
        "label": "モデル名",
        "summary": "概要",
        "tension_subset": ["α", "β", "γ"]
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `title`: パズル名（文字列）
- `S`: 問題文（文字列）
- `T`: 真相（文字列）
- `_phase1`: Phase 固有メタデータ
  - `tensions`: 緊張の配列。各要素は `{id, label}`
  - `models`: モデルの配列。各要素は `{id, label, summary, tension_subset}`
    - `id`: モデル ID（T, M1, M2, ...）
    - `label`: モデル名
    - `summary`: 1〜2 文の概要
    - `tension_subset`: 緩和する緊張 ID のリスト

#### 規則

- 競合モデルは最低 10 以上列挙する
- 各モデルは S の全事実と整合すること
- 緊張の部分集合の分類により、モデルの網羅性を担保する
- T も models に含め、他のモデルと同列に扱う

### Phase 2: モデル具象命題と分割構造

規則: `rules/モデル具象命題と分割構造_v2.md`

#### 目的

各モデルの具象命題を記述し、モデル横断で共通命題を抽出して、モデル集合の階層的分割構造を導出する。

#### 入力

Phase 1 の出力

#### 手順

1. T および全競合モデルについて、具象命題を列挙する
   - 各モデルの因果構造を命題として明示する
2. モデル間で共通する命題を抽出する
   - 同じ事実を共有するモデル群を特定する
   - カテゴリは「人物の状態」「他者の認識と行動」「解消メカニズム」等、パズルの構造に応じて自然に定まる
3. 共通命題によるモデル集合の分割を構成する
   - 上位分割（大きく分ける命題）と下位分割（さらに細分する命題）を階層として整理する

#### 出力データ構造

```json
{
  "_phase2": {
    "model_propositions": {
      "T": ["男はしゃっくりが止まらなかった", "..."],
      "M1": ["男は何かが喉に詰まっていた", "..."]
    },
    "division_propositions": [
      {
        "id": "P1",
        "label": "男は何らかの症状・状態に困っていた",
        "truth_value_in_T": true,
        "true_models": ["T", "M1", "M2", "M3", "M4", "M7", "M8"],
        "false_models": ["M5", "M6", "M9", "M10", "M11", "M12", "M13", "M14", "M15"]
      }
    ],
    "division_hierarchy": [
      {
        "proposition_id": "P1",
        "true_set": ["T", "M1", "M2", "M3", "M4", "M7", "M8"],
        "false_set": ["M5", "M6", "M9", "M10", "M11", "M12", "M13", "M14", "M15"],
        "children": [
          {
            "proposition_id": "P2",
            "true_set": ["T", "M1", "M2", "M3", "M7", "M8"],
            "false_set": ["M4"],
            "children": []
          }
        ]
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `_phase2`: Phase 固有メタデータ
  - `model_propositions`: モデル ID → 具象命題ラベルのリスト（辞書）
  - `division_propositions`: 分割命題の配列。各要素:
    - `id`: 分割命題 ID（P*）
    - `label`: 分割命題のラベル
    - `truth_value_in_T`: T における真偽（必ず true）
    - `true_models`: 真であるモデル ID リスト
    - `false_models`: 偽であるモデル ID リスト
  - `division_hierarchy`: 分割の木構造。各ノード:
    - `proposition_id`: 分割命題 ID
    - `true_set`: 真側モデル ID リスト
    - `false_set`: 偽側モデル ID リスト
    - `children`: 子ノードの配列（同じ構造の再帰）

#### 規則

- 分割命題は T において全て真であること
- 具象命題はモデルの因果構造を反映すること（単なる事実の羅列ではなく因果関係を含む）
- 分割命題はモデルの具象命題から抽象化してよいが、T の事実に忠実であること
- 分割の階層は上位（抽象的、true_set 大、棄却力小）から下位（具体的、true_set 小、棄却力大）へ整理する

### Phase 3: S 細分化と戦術連鎖

規則: `rules/S細分化.md`、`rules/戦術連鎖_v3.md`

#### 目的

S を原子命題に分解し、S 命題から分割命題への戦術連鎖を構成する。連鎖の reveals → fc 関係が質問の formation_conditions を定める。

#### 入力

Phase 1〜2 の出力、`structure/汎用戦術.md`

#### 手順

##### Step 1: S 細分化

S を分割不能な原子命題に分解する。論理的含意による導出命題も列挙する。

- 原子命題: S の文から直接取り出せる最小の事実
- 導出命題: 原子命題から論理的に含意されるもののみ。推測・解釈を含めない
- 自明な含意（「バーがあった」等）も見落とさないこと

##### Step 2: S → 上位分割命題への戦術連鎖

S の原子命題を起点とし、上位の分割命題に至る戦術連鎖を構成する。

- 各連鎖は `structure/汎用戦術.md` の戦術を適用する
- S の情報構造から自然に生まれるものであり、T からの逆算ではない

##### Step 3: 上位 → 下位分割命題への戦術連鎖

上位分割命題の確認を起点とし、下位の分割命題に至る戦術連鎖を構成する。

- 前段の連鎖の reveals が次の連鎖の fc になる
- 連鎖間の依存関係が質問の formation_conditions を自然に定める

本 Phase では各命題に対して 1 つの導出経路（fc の単一 AND グループ）を設計する。fc の OR 構造（複数の代替導出経路）はこの時点では使用しない。

#### 出力データ構造

```json
{
  "_phase3": {
    "s_propositions": [
      {"id": "S1", "label": "ある男がいた", "type": "atomic"},
      {"id": "S10", "label": "男はバーの外から来た", "type": "derived", "derived_from": ["S3"]}
    ],
    "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"],
    "tactical_chains": [
      {
        "id": "C1",
        "type": "S_to_upper",
        "source": ["S5"],
        "target": "P1",
        "tactic": "動機の探索 → 段階的絞り込み",
        "description": "S5 →[動機の探索]→ 水を求めた理由がある →[段階的絞り込み]→ P1"
      },
      {
        "id": "C5",
        "type": "upper_to_lower",
        "source": ["P1", "S5"],
        "target": "P2",
        "tactic": "観測事実の深掘り → 段階的絞り込み",
        "description": "P1+S5 →[観測事実の深掘り]→ 困りごとと水の関係 →[段階的絞り込み]→ P2",
        "fc_source": "P1 は連鎖 C1 の reveals"
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `initial_confirmed`: ゲーム開始時の確認済み命題 ID リスト（S 原子命題 + 導出命題）
- `_phase3`: Phase 固有メタデータ
  - `s_propositions`: S 命題の配列。各要素:
    - `id`: S 命題 ID（S*）
    - `label`: 命題のラベル
    - `type`: `"atomic"` | `"derived"`
    - `derived_from`: 導出元の命題 ID リスト（derived の場合のみ）
  - `initial_confirmed`: initial_confirmed のコピー（トップレベルと同一）
  - `tactical_chains`: 戦術連鎖の配列。各要素:
    - `id`: 連鎖 ID（C*）
    - `type`: `"S_to_upper"` | `"upper_to_lower"`
    - `source`: 起点命題 ID リスト
    - `target`: 到達目標の分割命題 ID
    - `tactic`: 適用する汎用戦術名
    - `description`: 連鎖の記述
    - `fc_source`: 起点の由来（upper_to_lower の場合）

#### 規則

- S の細分化は分割不能なレベルまで行う
- 導出は論理的含意のみ。「不満なく帰った」のような推測を含めない
- 戦術連鎖は S の情報構造から自然に生まれるものであり、T からの逆算ではない
- 各連鎖ステップは `structure/汎用戦術.md` の戦術を使用する
- 上位→下位連鎖の source には、前段の連鎖で reveal された分割命題を含める

### Phase 4: 質問生成（連鎖ベース）

規則: `rules/質問生成_v3.md`

#### 目的

戦術連鎖に沿って質問を生成する。各質問の fc（形成条件）は前段の reveals から自然に定まる。

#### 入力

Phase 1〜3 の全出力

#### 手順

##### Step 1: 連鎖ごとの質問展開

各戦術連鎖について、連鎖のステップを質問に展開する。

- 排除質問（回答: いいえ）: 常識的な仮説を排除し、探索方向を示す
- 確認質問（回答: はい）: 分割命題を確認する

##### Step 2: fc の設定

各質問の fc を、連鎖の構造から設定する。

- 連鎖の起点が S 命題の場合: fc = {S 命題群}（常に確認済み）
- 連鎖の起点が分割命題の場合: fc = {前段の reveals に含まれる P 命題群}
- 連鎖内の後続質問: fc に前の質問の回答確認（中間命題または prerequisites）を含める

##### Step 3: reveals の設定

- 分割命題を確認する質問: reveals = [確認される P 命題]
- 中間ステップの質問で名前付き命題を reveal しない場合: 中間命題を明示的に導入するか、後続質問の prerequisites で処理する

##### Step 4: 分割命題の formation_conditions 確定

各分割命題の formation_conditions を、それを reveal する質問の fc から確定する。

- P の formation_conditions = P を reveal する質問の fc
- recall_conditions = reveals される命題の formation_conditions（維持原則）

##### Step 5: 探索パスの構成

複数の探索パス（異なる入口から開始）を構成し、合理性を確認する。

- 最低 3 パスを構成する
- 各パスが合理的であること（各質問の開放が想起により説明でき、唐突な飛躍がない）

#### 出力データ構造

```json
{
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "_truth_value": true,
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    },
    {
      "id": "N1a",
      "label": "男が水を求めたのは喉の渇きからではない",
      "_truth_value": true,
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1a",
      "text": "男が水を求めたのは喉が渇いたからですか？",
      "answer": "いいえ",
      "recall_conditions": [["S5"]],
      "reveals": ["N1a"],
      "mechanism": "observation",
      "prerequisites": []
    },
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",
      "answer": "はい",
      "recall_conditions": [["S5"]],
      "reveals": ["P1"],
      "mechanism": "observation",
      "prerequisites": []
    }
  ],
  "_phase4": {
    "chain_mapping": {
      "C1": ["Q1a", "Q1b"]
    },
    "exploration_paths": [
      {
        "name": "パス A: 水の動機から入る",
        "sequence": "Q1a(いいえ) → Q1b(はい:P1) → ..."
      }
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `propositions`: 分割命題 + 中間命題の配列。各要素は `Proposition` 型（`models.py` 参照）
  - `id`: 命題 ID（P* は分割命題、N* は中間命題）
  - `label`: 命題のラベル（文字列）
  - `formation_conditions`: `list[list[str]]`（OR of AND）。Phase 3 の連鎖構造から確定
  - `entailment_conditions`: `list[list[str]] | null`。Phase 5 で設定するため、本 Phase では null
  - `rejection_conditions`: `list[list[str]] | null`。Phase 9 で設定するため、本 Phase では null
  - `_truth_value`: T における真偽（エクスポート時に除去される）
- `questions`: 質問の配列。各要素は `Question` 型（`models.py` 参照）
  - `id`: 質問 ID
  - `text`: 質問文
  - `answer`: 「はい」or「いいえ」
  - `recall_conditions`: `list[list[str]]`（OR of AND）。reveals される命題の formation_conditions と一致
  - `reveals`: `list[str]`。回答で明らかになる命題 ID 群
  - `mechanism`: `"observation"` | `"link"` | `"anomaly"`
  - `prerequisites`: `list[str]`。質問文の言語的前提となる命題 ID 群
- `_phase4`: Phase 固有メタデータ
  - `chain_mapping`: 各連鎖 ID → 対応する質問 ID リスト
  - `exploration_paths`: 探索パスの記録

#### 規則

- **recall_conditions = reveals される命題の formation_conditions**: 質問の recall_conditions は、その質問が reveal する命題の formation_conditions と一致する
- 回答は T の事実と矛盾しないこと
- 各質問は単一の推論ステップに対応すること
- **全質問が reveals を持つこと**（reveals: [] は原則あり得ない）
- reveals のない中間ステップは、中間命題を導入して reveals を持たせる
- 探索パスは最低 3 つ検証する
- `reveals` は `list[str]` 型であること（文字列ではなくリスト）

### Phase 5: T事実の統合

規則: `rules/T事実の統合.md`

#### 目的

T の具体的事実（Phase 2 の model_propositions.T で列挙済み）が、ゲームの進行中にプレイヤーに観測されるかを分析し、必要に応じて連鎖延長を設計する。

#### 入力

Phase 1〜4 の全出力

#### 手順

##### Step 1: 演繹の分析

各T事実について、それを論理的に含意する命題集合族を特定する。

- 各T事実 t に対して、t を導出する命題集合族 {A₁, A₂, ...} を特定する
- 各 Aᵢ は作成済みの命題（S命題・分割命題・中間命題）の部分集合である
- 導出可能な t に entailment_conditions を設定する
- 演繹は**厳密な論理的含意**のみ。「容易に想像できる」「自然に連想される」は formation（連鎖延長）の領域

##### Step 2: C_final の特定

全分割命題が confirmed になった時点で、確実に confirmed な命題をリストアップする。

C_final = initial_confirmed（S命題群）∪ 全分割命題（P1〜P_last）

中間命題（N*）はパスにより confirmed か derived かが異なりうるため、C_final には含めない（控えめな近似）。

##### Step 3: 導出済みT事実の判定

Step 1 の演繹関係と Step 2 の C_final を照合し、P_last 到達時点で自動的に導出されるT事実を特定する。導出されないT事実を以降の Step の対象とする。

##### Step 4: 連鎖延長の設計

Step 3 で導出されないT事実について、既存連鎖の延長として中間命題と質問を設計する。

- 連鎖の末端から、T事実に至る中間命題と質問を設計する
- 質問生成と同じ品質基準を満たすこと（recall = fc の原則、1段階推論、想起の自然さ）
- 強引な導出は行わない。連鎖が不自然であれば設計しない

##### Step 5: 延長の検証

Step 4 で設計した各延長について以下を検証する:

1. **棄却の非発生**: 延長命題が新たなモデルの棄却を引き起こさないこと
2. **跳躍の非存在**: 連鎖が跳躍なく想起可能であること

いずれかに不合格の場合、その延長は除外する。

##### Step 6: ユニーク要素の確定

演繹でも連鎖延長でも到達できないT事実を「ユニーク要素」として記録する。ユニーク要素は正解の核心的な驚きであり、無理に導出する必要はない。

#### 出力データ構造

```json
{
  "_phase5": {
    "t_facts": [
      {"id": "T1", "label": "T事実の記述", "deducible": false, "entailment_conditions": null},
      {"id": "T4", "label": "T事実の記述", "deducible": true, "entailment_conditions": [["P7"]]}
    ],
    "c_final": ["S1", "S2", "P1", "P2"],
    "chain_extensions": [
      {"t_fact_id": "T3", "source_chain": "C7", "propositions": ["NT3"], "questions": ["QT3"]}
    ],
    "unique_elements": ["T1", "T2"]
  }
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `_phase5`: Phase 固有メタデータ
  - `t_facts`: 各T事実の演繹分析結果
  - `c_final`: C_final の命題 ID リスト
  - `chain_extensions`: 連鎖延長のトレーサビリティ
  - `unique_elements`: ユニーク要素の ID リスト
- `propositions` への追加: 連鎖延長で導入した中間命題（`NT*`）
- `questions` への追加: 連鎖延長で生成した質問（`QT*`）
- 既存命題への `entailment_conditions` の設定

#### 規則

- 演繹は厳密な論理的含意のみ。思考の跳躍を一切含めないこと
- 連鎖延長は質問生成と同じ品質基準に従うこと
- 棄却を引き起こす延長、連鎖が不自然な延長は無理に組み込まないこと
- 導出できるものを導出した結果として残るものがユニーク要素であり、それを最終盤の探索対象とすることは適切である

### Phase 6: クリア条件とユニーク要素の導出

規則: `rules/クリア条件とユニーク要素の導出.md`

#### 目的

T事実の統合で特定されたユニーク要素に対して、クリア条件を確定し、必要なユニーク要素への戦術連鎖を設計する。

#### 入力

Phase 5 の全出力

#### 手順

##### Step 1: クリア条件の確定

T事実のうち、どの組み合わせが揃えば「正解に到達した」と言えるかを意味的に分析し、クリア条件を集合族（OR of AND）として定義する。

- 全てのT事実が揃う必要があるとは限らない
- ある事実が別の事実の自明な帰結である場合、帰結側はクリア条件から除外できる
- 正解の核心を構成するT事実の組み合わせを特定する
- **S の全緊張が解消されること**: クリア条件に含まれるT事実（および既に導出済みの命題）により、S の全ての緊張が説明されることを確認する

##### Step 2: 必要なユニーク要素の特定

クリア条件に含まれるT事実のうち、Phase 5 でユニーク要素に分類されたものを特定する。これが戦術連鎖の設計対象となる。クリア条件に含まれないユニーク要素には戦術連鎖を設計しない。

##### Step 3: 戦術連鎖の設計

Step 2 で特定した各ユニーク要素について、C_final を起点とした戦術連鎖を設計する。

- 必要に応じて中間命題を導入する
- ユニーク要素は最終盤の探索対象であり、全ての抽象的命題が確認された後に到達する設計とする

##### Step 4: 質問生成

Step 3 で設計した戦術連鎖に沿って質問を生成する。質問生成と同じ品質基準を満たすこと。

#### 出力データ構造

```json
{
  "_phase6": {
    "clear_conditions": [["T1", "T6", "T4"]],
    "required_unique_elements": ["T1"],
    "chain_designs": [
      {"target": "T1", "source": "C_final", "propositions": ["NU1"], "questions": ["QU1"]}
    ]
  }
}
```

#### 生成するデータ

本 Phase は以下を生成する:

- `clear_conditions`: クリア条件（トップレベルキー）
- `_phase6`: Phase 固有メタデータ
  - `clear_conditions`: クリア条件のコピー
  - `required_unique_elements`: 戦術連鎖の設計対象ユニーク要素
  - `chain_designs`: 戦術連鎖のトレーサビリティ
- `propositions` への追加: ユニーク要素関連の中間命題（`NU*`）
- `questions` への追加: ユニーク要素関連の質問（`QU*`）

#### 規則

- クリア条件の確定は意味的な分析であり、機械的に決定するものではない
- 全てのユニーク要素にクリア到達可能な連鎖が必要なわけではない。クリア条件に含まれるもののみが対象
- 連鎖の設計は質問生成と同じ品質基準に従うこと

### Phase 7: 検証

規則: 工程1では `rules/検証_工程1.md`、工程3では `rules/検証_工程3.md`

#### 目的

大域目標の達成と各種整合性を検証する。工程1では真側のみ、工程3では棄却側を含む全体を検証する。

#### 入力

工程1: Phase 1〜6 の全出力
工程3: Phase 1〜6, 8, 9 の全出力

#### 手順

##### Step 1: 構造的整合性

- 全命題の ID が一意であること
- 全質問の reveals が存在する命題 ID を参照していること
- fc / recall_conditions / formation_conditions が存在する命題 ID を参照していること
- initial_confirmed の命題が全て定義されていること

##### Step 2: fc 連鎖の整合性

- 全質問の fc が初期確認済み命題（initial_confirmed）から到達可能であること
  - fc に含まれる P 命題が、先行する質問の reveals に含まれること
  - fc に含まれる S 命題が initial_confirmed に含まれること
- 循環依存がないこと

##### Step 3: reveals の網羅性

- 全分割命題が少なくとも 1 つの質問の reveals に含まれること
- 棄却可能な全モデルが、対応する分割命題の質問回答により棄却されること

##### Step 4: recall = fc の整合性

- 各質問について、recall_conditions が reveals される命題の formation_conditions と一致していること
- reveals が空の質問については、recall_conditions が連鎖の source 命題に基づいていること

##### Step 5: 回答の T 整合性

- 全質問の回答（はい/いいえ）が T の事実と矛盾しないこと
- 分割命題の truth_value が T の内容と整合すること

##### Step 6: 導出条件の 1 段階推論原則

- 全命題の formation_conditions について、各 AND グループが単一の推論ステップとして妥当であること
- 結論から逆算した不自然な条件がないこと

##### Step 7: パストレース

複数の質問選択順でゲームをシミュレーションする。

1. confirmed = initial_confirmed から開始
2. 論理的導出（entailment_conditions 不動点）→ confirmed 拡大
3. 仮説導出（formation_conditions 1回パス）→ derived 計算
4. known = confirmed ∪ derived
5. recall_conditions が known で満たされる質問を列挙（オープン質問）
6. オープン質問から 1 つ選択 → reveals を confirmed に追加
7. 2〜6 を繰り返し、全分割命題が confirmed になるまで

確認事項:
- **到達可能性**: いかなる選択順でも T 近傍に到達可能であること
- **行き詰まり**: オープン質問がなくなり、かつ分割命題が未確認の状態が発生しないこと
- **想起の合理性**: 各質問の開放が唐突な飛躍を含まないこと

最低 3 パスを異なる入口から検証する。

##### Step 8: 棄却の完全性

- 棄却可能な全モデルが、探索パスの終了時に棄却されていること
- 残存するモデルが棄却不可能モデルのみであること

#### 検証条件

全ステップが PASS であること。FAIL の場合、エラー内容に応じて原因フェーズに差し戻す。

| 検証項目 | FAIL 時の差し戻し先 |
|---|---|
| 構造的整合性 | 該当フェーズ |
| fc 連鎖の整合性 | Phase 3, 4, 5, または 9 |
| reveals の網羅性 | Phase 4 または Phase 9 |
| recall = fc の整合性 | Phase 4, 5, または 9 |
| 回答の T 整合性 | Phase 4, 5, または 9 |
| 1 段階推論原則 | Phase 3, 4, 5, または 9 |
| パストレース | Phase 3, 4, 5, または 9 |
| 棄却の完全性 | Phase 2 または Phase 9 |

### Phase 8: ブラッシュアップ

規則: `rules/戦術連鎖のブラッシュアップ.md`

#### 目的

工程1の真側を磨き上げる。各質問の想起条件（recall_conditions）に跳躍がないかをレビューし、跳躍がある場合は戦術連鎖を修正して解消する。

工程1（Phase 1〜7）の全出力

- 跳躍がなくなるまで反復する

### Phase 9: 連鎖内分岐分析

#### 目的

#### 入力

Phase 1〜6, 8 の全出力

##### Step 1: 連鎖内分岐の分析

- `topic_categories`: トップレベルキー。固定カテゴリセットの配列
- `_phase10`: Phase 固有メタデータ
  - `assignments`: 各質問のカテゴリ割り当て根拠
  - `initial_open_distribution`: ゲーム開始時のカテゴリ別初期オープン質問
  - `balance`: カテゴリ別質問数
- 各質問の `topic_category` フィールドへの値の設定

#### 規則

- カテゴリは固定セット（パズル非依存）とし、パズルごとに新規カテゴリを追加しない
- カテゴリの付与は質問の内容を変更しない（後処理）
- 情報漏洩検証で FAIL の場合、カテゴリの再割り当てで対処する（質問自体は変更しない）

## フェーズ間の依存

```
工程1: 真側の構築
  Phase 1（緊張構造分析とモデル列挙）
    → Phase 2（モデル具象命題と分割構造）
      → Phase 3（S 細分化と戦術連鎖）
        → Phase 4（質問生成）
          → Phase 5（T事実の統合）
            → Phase 6（クリア条件とユニーク要素の導出）
              → Phase 7（検証: 真側）

工程2: ブラッシュアップ
  Phase 8（戦術連鎖のブラッシュアップ、反復）

工程3: 棄却側の構築
  Phase 9（連鎖内分岐分析）
    → Phase 7（検証: 全体）

工程4: 質問分類
  Phase 10（質問分類）
```

Phase 1〜2 が T 側の骨格を構成し、Phase 3 で S との接続を設計し、Phase 4 で質問を生成する。Phase 5〜6 で T事実の到達可能性を分析しクリア条件を確定する。工程1の末尾で真側の検証を行う。工程2で想起条件の跳躍を磨き上げる。工程3では Phase 9 で真側連鎖の分岐を分析して棄却側の質問を生成し、Phase 7 で全体を検証する。工程4では全質問にトピックカテゴリを付与する。

## 最終データ形式

### data_src.json

全フェーズの中間生成物を含む。各フェーズの出力は `_phase*` キーに格納される。

### data.json

ゲーム実行に必要なデータのみを含む。

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "propositions": [
    {
      "id": "P1",
      "label": "男は何らかの症状・状態に困っていた",
      "formation_conditions": [["S5"]],
      "entailment_conditions": null,
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1b",
      "text": "男は何かに困っていましたか？",
      "answer": "はい",
      "recall_conditions": [["S5"]],
      "reveals": ["P1"],
      "mechanism": "observation",
      "prerequisites": [],
      "topic_category": "状況の確認"
    }
  ],
  "initial_confirmed": ["S1", "S2", "S3", "S4", "S5", "S6", "S7", "S8", "S9"],
  "topic_categories": [
    {"id": "状況の確認", "name": "状況の確認"},
    {"id": "原因と動機", "name": "原因と動機"},
    {"id": "人物と背景", "name": "人物と背景"},
    {"id": "前提の確認", "name": "前提の確認"}
  ]
}
```

- `propositions`: 分割命題 + 中間命題。S 命題は含めない（initial_confirmed で管理）
- `questions`: 全質問は `reveals` を持つ。`reveals` は `list[str]` 型
- フィールド定義は `app/poc_v3/src/models.py` の `Proposition` / `Question` クラスに準拠
- `_` プレフィックスのフィールド（`_truth_value`, `_phase*` 等）はエクスポート時に除去される
