# 統合アルゴリズム v6

## 概要

v5の構造を基盤としつつ、「パズルの面白さ」分析シリーズ（分析1〜16）から得た主要概念（2要素特性・隠れアノマリー特性）を反映する。Phase 1を構造分解として再設計し、パズルの特性に応じた規則を適用する。

### v5 からの主要な変更

- **Phase 1 の再設計**: 緊張構造分析を「構造分解」に置き換え。パズルの特性（2要素・隠れアノマリー）に応じた規則を適用
- **FPC（FP Component）の導入**: FPを要素ごとのFPC（FPC1, FPC2）から構成する
- **競合モデルの要素単位化**: パズル全体ではなく各要素に対する競合モデルを分析

### 維持する原則

- **仮説導出**: formation_conditions（confirmed→derived 1回パス）。`structure/仮説導出.md` 参照
- **質問と命題の1対1対応**: 各質問は1つの命題を reveals する（`reveals` は単一の命題 ID）
- **質問の利用可能条件は命題構造から導出**: 質問に独立した条件フィールドを持たない。回答が「はい」なら reveals 先命題の fc、「いいえ」なら reveals 先命題の negation_of が指す命題の fc が利用可能条件となる
- **否定関係（negation_of）**: 命題間の対称的な否定関係。P.negation_of = Q ⟺ Q.negation_of = P。nullable（全命題に必須ではない）
- **仮説連鎖の廃止**: formation_conditions は confirmed のみ参照。derived を参照しない
- **1 段階推論の原則**: 各導出条件は単一の推論ステップを表現する
- **命題の独立性・客観性**: 全命題は自己完結的で客観的に検証可能
- **T 忠実性**: 生成する命題は T の情報から逸脱しない
- **YES 進行原則**: 「はい」回答のみが進行を駆動する。fc は「はい」で confirm される命題（P*, I*, S*）のみを含む。ゲームは「いいえ」質問を一度も聞かずにクリア可能であること

#### fc とゲーム進行サイクル

ゲームは質問回答サイクルで進行する。各命題の formation_conditions（fc）は、initial_confirmed または先行する質問の reveals で confirmed される命題のみを参照する。戦術連鎖の各ステップはこのサイクルの 1 単位に対応し、ステップの出力命題は質問で confirm され、後続ステップの fc を充足する。

Phase 1 は導出グラフ（骨組み）を提供する。fc は Phase 2 で戦術連鎖の各ステップの入力構造から導出する。Phase 3 で fc に基づいて質問を生成する。全命題の fc を遡ると、末端は必ず initial_confirmed（S 命題）に到達する。

entailment_conditions（EC）は Phase 7（調整）で初めて導入される後処理であり、Phase 1〜6 では使用しない。EC は厳密な論理的演繹（情報の合成・要約・縮約のみで推論を含まない導出）に限定される。判断・推論・解釈が介在する導出は EC ではなく fc で表現する。FP には EC を設定しない（FP の confirm はプレイヤーが質問で明示的に確認する体験であり、自動 confirm にしない）。基準の詳細は `rules/調整.md` Step B-2 を参照。

- T: 真相
- 補足（任意）: クイズファイルの `## 補足` セクション。特性・要素・鍵が事前に記載されている場合、Phase 1 の特性判定・要素特定・鍵特定の入力として使用する

### Phase 3: 質問生成

規則: `rules/質問生成_v4.md`

#### 目的

戦術連鎖に沿って質問を生成する。各質問の fc（形成条件）は前段の reveals から自然に定まる。

#### 入力

Phase 1〜2 の全出力

#### 手順

##### Step 1: 連鎖ごとの質問展開

各戦術連鎖について、連鎖のステップを質問に展開する。

- 排除質問（回答: いいえ）: 常識的な仮説を排除し、探索方向を示す
- 確認質問（回答: はい）: 命題を確認する

##### Step 2: fc の設定

各質問の fc を、連鎖内の位置に基づいて設定する。fc は段階的オープンを実現するための順序制約であり、各質問は直前のステップの reveals を fc に含めなければならない。

- **連鎖の入口質問**（各連鎖の最初の質問のみ）:
  - S 起点の連鎖: fc = {起点の S 命題群}（常にオープン）
  - 分割命題起点の連鎖: fc = {前段の連鎖で reveals された命題群}
- **連鎖内の後続質問**（入口以外の全質問）: fc に直前の質問で reveals された中間命題を含める。連鎖 source → I1 → target の場合:
  - I1 の質問: fc = {source}（入口）
  - target の質問: fc に I1 を含める（I1 確認後にオープン）

**target の fc が source のみを参照し intermediates をスキップしてはならない。** fc が intermediate を参照しないと段階的オープンが機能しない（全質問が初手からオープンになる）。

##### Step 3: reveals の設定

- **各質問は1つの命題を reveals する**（質問と命題の1対1対応）
- 中間ステップの質問で名前付き命題を reveal しない場合: 中間命題を明示的に導入して reveals を持たせる

##### Step 4: 分割命題の formation_conditions 確定

各分割命題の formation_conditions を、連鎖内の直前ステップから確定する。

- 連鎖 source → I1 → ... → In → target において:
  - target.fc = [[In]]（最後の intermediate）
  - Ik.fc = [[Ik-1]]（直前の intermediate。k=1 の場合は source）
- **fc は連鎖の source ではなく、直前のステップを参照すること。** source を直接参照すると intermediate がゲートとして機能しない

##### Step 5: 否定命題の導入

回答が「いいえ」の質問について、reveals される命題の否定対象（negation_of が指す命題）を定義する。

- 「いいえ」質問が reveal する命題（N*）に対して、その肯定版の仮説命題（H*）を作成する
- N*.negation_of = H*.id, H*.negation_of = N*.id（対称関係）
- H* の formation_conditions を設定する（質問の利用可能条件として使用される）

##### Step 5.5: prerequisites の設定

全質問について、質問文が言語的に前提とする情報を特定し、prerequisites フィールドに設定する。initial_confirmed に含まれない前提概念に対応する命題を prerequisites に追加する。判定基準は `precedents/質問文の含意の判例.md` に従う。

##### Step 6: 探索パスの構成

複数の探索パス（異なる入口から開始）を構成し、合理性を確認する。

- 最低 3 パスを構成する
- 各パスが合理的であること（各質問の開放が想起により説明でき、唐突な飛躍がない）

#### 出力データ構造

```json
{
  "propositions": [
    {
      "id": "P1", "label": "命題ラベル",
      "_truth_value": true,
      "formation_conditions": [["S5"]]
    },
    {
      "id": "N1a", "label": "否定命題ラベル",
      "_truth_value": true, "negation_of": "H1a",
      "formation_conditions": [["S5"]]
    },
    {
      "id": "H1a", "label": "仮説命題ラベル",
      "_truth_value": false, "negation_of": "N1a",
      "formation_conditions": [["S5"]]
    }
  ],
  "questions": [
    {
      "id": "Q1a", "text": "質問文", "answer": "いいえ",
      "reveals": "N1a",
      "mechanism": "observation", "prerequisites": []
    },
    {
      "id": "Q1b", "text": "質問文", "answer": "はい",
      "reveals": "P1",
      "mechanism": "observation", "prerequisites": []
    }
  ]
}
```

#### 生成するデータ

本 Phase は以下のトップレベルキーを**新規作成**する:

- `propositions`: 命題の配列。各要素は `Proposition` 型（`models.py` 参照）
- `questions`: 質問の配列。各要素は `Question` 型（`models.py` 参照）
  - `reveals`: `str` 型。回答で明らかになる命題 ID（1つ）
  - `prerequisites`: `list[str]`。質問文の言語的前提となる命題 ID 群
- `_phase3`: Phase 固有メタデータ（chain_mapping、exploration_paths）

#### 規則

- 回答は T の事実と矛盾しないこと
- 各質問は単一の推論ステップに対応すること
- **各質問は1つの命題を reveals すること**（`reveals` は `str` 型）
- **全質問が reveals を持つこと**（空文字列は不可）
- reveals のない中間ステップは、中間命題を導入して reveals を持たせる
- 探索パスは最低 3 つ検証する

### Phase 4: 検証

規則: 工程1では `rules/検証_工程1_v3.md`、工程3では `rules/検証_工程3_v3.md`

#### 目的

大域目標の達成と各種整合性を検証する。工程1では真側のみ、工程3では棄却側を含む全体を検証する。

#### 入力

工程1: Phase 1〜3 の全出力
工程3: Phase 1〜3, 5, 6 の全出力

#### 手順

##### Step 1: 構造的整合性

- 全命題の ID が一意であること
- 全質問の reveals が存在する命題 ID を参照していること
- formation_conditions / negation_of が存在する命題 ID を参照していること
- initial_confirmed の命題が全て定義されていること

##### Step 2: 利用可能条件の到達可能性

- 全質問の利用可能条件が initial_confirmed から到達可能であること
  - 利用可能条件は reveals 先命題の fc（「いいえ」回答の場合は negation_of 先命題の fc）
  - fc に含まれる P/I 命題が、先行する質問の reveals に含まれること
  - fc に含まれる S 命題が initial_confirmed に含まれること
  - **fc に N*/NF* が含まれていないこと**（YES 進行原則）
- 循環依存がないこと

##### Step 3: reveals の網羅性

- 全分割命題が少なくとも 1 つの質問の reveals に含まれること

##### Step 4: 否定関係の整合性

- 「いいえ」回答の質問が reveal する命題に negation_of が設定されていること
- negation_of が指す命題が存在し、formation_conditions を持つこと
- negation_of 関係が対称的であること

##### Step 5: 回答の T 整合性と命題品質

- 全質問の回答が T の事実と矛盾しないこと
- 全命題の T 文脈での真偽が明確であること
- 全命題の意味が確定的であること

##### Step 6: reveals の厳密性

- 各質問の reveals が質問テキストと回答から厳密に論理的に導かれること

##### Step 7: prerequisites の対話上確立可能性

- prerequisites の全命題が initial_confirmed またはいずれかの質問の reveals に含まれること

##### Step 7.5: 質問文の含意検証

- 質問文の前提情報が利用可能時点で保証されていること。判定基準は `precedents/質問文の含意の判例.md` に従う

##### Step 8: 1 段階推論原則

- 全命題の fc について、各 AND グループが単一の推論ステップとして妥当であること

##### Step 8.5: 連鎖構造と fc の整合性

- target.fc が直前の intermediate を参照していること
- 各 Ik.fc が前のステップを参照していること
- target.fc が source のみを参照し intermediates をスキップしている場合は FAIL

##### Step 9: パストレース

複数の質問選択順でゲームをシミュレーションする。最低 3 パスを異なる入口から検証する。

確認事項:
- **到達可能性**: いかなる選択順でも全分割命題が confirmed になること
- **行き詰まり**: オープン質問がなくなり未確認の状態が発生しないこと
- **想起の合理性**: 各質問の開放が唐突な飛躍を含まないこと

#### 検証条件

全ステップが PASS であること。FAIL の場合、エラー内容に応じて原因フェーズに差し戻す。

### Phase 5: ブラッシュアップ

規則: `rules/戦術連鎖のブラッシュアップ_v2.md`

#### 目的

工程1の真側を磨き上げる。(1) 各質問の利用可能条件に跳躍がないかレビューし解消する。(2) fc に含まれる N*/NF* を「はい」命題に置き換える。

#### 入力

工程1（Phase 1〜4）の全出力

#### 手順

反復的に実施する。各反復で:

1. 全質問の利用可能条件を確認し、跳躍（想起の不自然さ）がないかレビューする
2. 跳躍が見つかった場合、戦術連鎖を修正して解消する
3. 全命題の fc を確認し、N*/NF* が含まれている場合は「はい」命題に置き換える
4. 全質問の prerequisites が質問文の言語的前提をカバーしているかレビューする
5. 修正に伴い propositions / questions を更新する

#### 規則

- 修正は戦術連鎖レベルで行い、質問単体の調整ではなく連鎖全体の整合性を保つ
- 跳躍がなくなり、fc に N*/NF* が含まれなくなり、prerequisites が妥当になるまで反復する

### Phase 6: 連鎖内分岐分析

規則: `rules/連鎖内分岐分析_v2.md`（質問生成時は `rules/質問生成_v4.md` も参照）

#### 目的

真側連鎖の各ステップにおける並列命題を分析し、モデルとの対応を確認した上で、全ての並列命題から NF 命題と QF 質問を導出する。

#### 入力

Phase 1〜3, 5 の全出力

#### 手順

##### Step 1: 連鎖内分岐の分析

真側連鎖の各ステップについて、並列命題とモデルとの対応を分析する。

- 各ステップの真側命題に対して自然に並列に並ぶ命題を特定する
- 各並列命題に対応するモデルを `_phase1.competing_models` と照合して特定する
- **各連鎖の全ステップに最低 1 つの分岐を持つこと**

##### Step 2: カバレッジ確認

全競合モデルが連鎖内分岐のいずれかの並列命題に対応しているかを確認する。

##### Step 3: 質問生成

Step 1 で特定した並列命題から質問を導出する。`rules/質問生成_v4.md` の規則に従うこと。

- 全ての並列命題について命題（`NF*`）を定義し、formation_conditions を設定する
- 各命題に対応する質問（`QF*`）を生成する（**1質問1命題**）
- NF ラベルが正解情報を含まないこと（情報漏洩チェック）

#### 規則

- 分岐の分析は戦術連鎖（命題と戦術）のレベルで行うこと
- 並列命題は真偽を問わず自然に並ぶものを探すこと
- **各連鎖の全ステップに最低 1 つの分岐を持つこと（分岐 0 個のステップは不合格）**
- カバーされないモデルへの対応を強制しない
- 質問は `rules/質問生成_v4.md` の規則に従うこと

---

### Phase 7: 調整

規則: `rules/調整.md`

#### 目的

全命題・全質問が確定した状態で、2 つの調整を行い質問数を削減する。

- **Step A（間引き）**: 概念的距離が不十分な冗長命題を間引く（命題と質問を削除）
- **Step B（演繹関係の特定）**: 論理的結合で導出できる命題に entailment_conditions を設定する（質問は残るがエンジンが自動的に非表示にする）

#### 入力

Phase 1〜6 の全出力（全質問・命題が確定した状態）

#### 手順

Step A → Step B の順に実施する。

**Step A（間引き）**:
1. 全導出関係の概念的距離を評価し、冗長ペアを特定する
2. 間引く側を判断する（デフォルト: 導出元を間引く）
3. fc の付け替え・prerequisites 付け替え・命題削除・質問削除・連鎖更新・分岐更新を実施する
4. 再検証する

**Step B（演繹関係の特定）**:
1. 間引き後の全命題について、命題の意味的内容に基づき演繹関係を特定する（複数命題の情報の合成のみで得られ、推論が介在しないもの）
2. 該当命題に entailment_conditions を設定する（formation_conditions はそのまま残す）
3. 再検証する

#### 規則

- Step A: デフォルトで導出元を間引く。間引き後に量的最低基準を満たさなくてもよい
- Step B: 演繹関係は命題の意味的内容に基づき判定する（fc の構造分析ではない）。推論を含む導出は演繹としない
- **FP に entailment_conditions を設定しない**

---

### Phase 8: 質問分類

規則: `rules/質問分類.md`

#### 目的

全質問にプレイヤー向けのトピックカテゴリを付与する。ゲームロジックには影響しない後処理である。

#### 入力

Phase 1〜7 の全出力（全質問が確定した状態）

#### 手順

##### Step 1: 戦術ベースのカテゴリ付与

各質問の対応する戦術ステップを参照し、固定カテゴリセットに基づいて `topic_category` を付与する。固定カテゴリセットを `topic_categories` としてトップレベルに追加する。

| カテゴリ | 対応する戦術 |
|---------|------------|
| 状況の確認 | 直接同定(0)・観測事実の深掘り(2)・段階的絞り込み(10) |
| 原因と動機 | 因果の逆転(5)・動機の探索(8) |
| 人物と背景 | 時間軸の拡張(3)・登場人物の探索(4)・人物像の探索(9) |
| 前提の確認 | 常識的仮説の排除(1)・矛盾の指摘(6)・ラベルと実体の乖離(7) |

##### Step 2: 棄却質問のカテゴリ付与

棄却質問（QF*）は元の連鎖ステップの戦術、または質問テキストの内容に基づいてカテゴリを決定する。

##### Step 3: 情報漏洩検証

1. カテゴリと命題の偏り検証
2. カテゴリの空白検証（ゲーム開始時に全カテゴリに初期オープン質問があること）

##### Step 4: バランス確認

各カテゴリの質問数を集計し、参考情報として記録する。

#### 規則

- カテゴリは固定セット（パズル非依存）とし、パズルごとに新規カテゴリを追加しない
- カテゴリの付与は質問の内容を変更しない（後処理）
- 情報漏洩検証で FAIL の場合、カテゴリの再割り当てで対処する（質問自体は変更しない）

## 最終データ形式

### data_src.json

全フェーズの中間生成物を含む。各フェーズの出力は `_phase*` キーに格納される。

### data.json

ゲーム実行に必要なデータのみを含む。

```json
{
  "title": "パズル名",
  "S": "問題文",
  "T": "真相",
  "propositions": [
    {
      "id": "P1",
      "label": "命題ラベル",
      "negation_of": null,
      "formation_conditions": [["S5"]],
      "rejection_conditions": null
    }
  ],
  "questions": [
    {
      "id": "Q1",
      "text": "質問文",
      "answer": "はい",
      "reveals": "P1",
      "mechanism": "observation",
      "prerequisites": [],
      "topic_category": "状況の確認"
    }
  ],
  "initial_confirmed": ["S1", "S2", "S3"],
  "clear_conditions": [["FP"]],
  "topic_categories": [
    {"id": "状況の確認", "name": "状況の確認"},
    {"id": "原因と動機", "name": "原因と動機"},
    {"id": "人物と背景", "name": "人物と背景"},
    {"id": "前提の確認", "name": "前提の確認"}
  ]
}
```

- `propositions`: 命題の配列。S 命題は含めない（initial_confirmed で管理）
- `questions`: 全質問は `reveals` を持つ。**`reveals` は `str` 型**（1質問1命題）
- `clear_conditions`: `[["FP"]]` 形式（FP が confirmed になればクリア）
- フィールド定義は `app/poc_v3/src/models.py` の `Proposition` / `Question` クラスに準拠
- `_` プレフィックスのフィールド（`_truth_value`, `_phase*` 等）はエクスポート時に除去される
