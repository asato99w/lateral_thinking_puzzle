# 戦術連鎖

## 概要

統合アルゴリズム v3 Phase 3 の規則。S の命題を抽出し、戦術連鎖の構成を通じて**中間命題**を導入する。加えて、導入した中間命題の導出条件を設定し、Phase 2 で暫定設定した既存命題の導出条件を確定する。

**本フェーズの意義は、適切な中間命題の導入と、全命題の導出条件の確定にある。** 戦術連鎖は「どこに命題が必要か」を発見し「何が何を導出するか」を構築するための分析手法であり、出力は戦術ラベルではなく命題群とその導出関係である。

このフェーズで初めて S が登場する。

v2 からの主要な変更:
- 末端命題の概念を廃止。到達目標を一般化
- 導出条件の構築・確定を本フェーズの責務に追加
- 仮説連鎖の非末端命題への戦術連鎖（Step 2b）を廃止

## 入力

- S: 問題文
- Phase 2 の出力（ピース群、トリガー族、暫定的な導出条件）
- 汎用戦術リスト（`structure/汎用戦術.md`）

## 出力

- S の命題群
- **Phase 3 で新たに導入された中間命題群**（`_phase3.generated_descriptors` に記録）
- 各トリガーに対する戦術連鎖
- 各ピースに対するピース内部の戦術連鎖
- **全命題の導出条件（確定版）**

## 中間命題の導入原則

### 本フェーズの核心

戦術連鎖の分析過程で、起点と終点の間のギャップを埋める中間命題を発見し、形式的な命題として導入する。

連鎖の各ステップの出力は**命題**でなければならない。非形式的なテキスト記述（「銃を向けたことが有益だった」等）で終わらせてはならない。テキスト記述が出てきた場合、それは形式化すべき命題の候補である。

### 中間命題が必要な場合

- **段階的絞り込み**: 起点と終点の抽象度の差が大きい場合、中間の抽象度の命題を導入する
- **具体化パターン**: 抽象版と具体版の命題ペアが必要な場合（例: 「乗り物」→「熱気球」）
- **複合命題の分解**: T の命題が複数情報を含む場合、構成要素を中間命題として導入する
- **矛盾の解消**: 矛盾の指摘の結果（解消された状態）が非自明な場合
- **因果の中間ステップ**: 因果の逆転や動機の探索で、直接的な接続が跳躍を含む場合

### 量的最低基準

各戦術連鎖（トリガーへの連鎖・各ピース内部連鎖）は、**最低 1 つの中間命題を導入しなければならない**。

- **1 つは最低評価である**: 1 つの中間命題は「基準をかろうじて満たしている」状態であり、望ましい水準ではない。連鎖の複雑さに応じて 2 つ以上を目指すべきである
- **0 は不合格**: いかなる理由であれ、中間命題が 0 個の連鎖は不完全であり、再検討が必要である

### 導入後の適切性レビュー

中間命題を導入した後、以下の観点でレビューする:

- **T 忠実性**: T の情報から逸脱していないか
- **プレイヤー視点の自然さ**: その中間ステップはプレイヤーの思考の流れとして自然か
- **質問への展開可能性**: その中間命題から自然な質問を生成できるか

レビューで問題が見つかった場合は、中間命題を**削除するのではなく、より適切な中間命題に置き換える**。量を減らすのではなく質を上げる方向で対処する。

### 中間命題の制約

- **T 忠実性**: T の命題の抽象化・一般化、T から論理的に導出可能な中間ステップ、または T に明示的に含まれる情報の再表現に限定する。T に含まれない具体的事実を新たに創作してはならない
- **独立性と客観性**: Phase 1 の命題と同じ品質基準を満たすこと

## 戦術ステップの形式

各ステップは以下を明示する:

- 入力命題集合（命題 ID）
- 適用する汎用戦術
- 出力命題（**命題 ID**。非形式テキストは不可）

```
{命題ID群} →[戦術]→ 命題ID
```

## 手順

### Step 1: S の命題の抽出

S を意味単位に分解し、S の命題として列挙する。ID は D-S* を使用する。

- S と T の命題間に重複がないこと
- 各命題は独立性と客観性を満たすこと

### Step 2: トリガーへの戦術連鎖

各ピースのトリガー命題に至る戦術連鎖を構成する。

```
起点命題ID群 →[戦術1]→ 中間命題ID →[戦術2]→ ...→ トリガー命題（または fc 前提命題）
```

起点はピースの種類により異なる:
- **独立ピース**: S の命題集合
- **依存ピース**: S の命題集合 + 依存先ピースの命題集合（推移的）

命題自体が起点の命題集合に含まれる場合は、直接同定を適用する。

**中間命題の導入**: 各ステップの出力が非形式テキストになった場合、それを正式な命題として定義する（ID を付与し、descriptors に追加する）。

設計にあたっては**設計方針**（後述）に従い、**プレイヤー視点の検証**を各連鎖に適用する。

### Step 3: ピース内部の戦術連鎖

各ピースに対して、トリガーを起点としてピース内の命題に到達する**命題の連鎖**を構成する。

```
トリガーID(+既知命題ID) →[戦術1]→ 中間命題ID →[戦術2]→ メンバー命題ID →...
```

- 前ステップの出力は次ステップの入力に利用可能（順序依存の連鎖）
- 全ての構成命題がいずれかのステップの出力に含まれること
- **中間命題の導入**: Phase 2 のメンバー命題に直接到達できない場合、中間命題を導入して連鎖を構成する

### Step 4: 中間命題の形式化と導出条件設定

Step 2-3 で導入した中間命題を整理する:

1. 全ての中間命題に ID を付与し、descriptors 配列に追加する
2. **各中間命題の導出条件（formation_conditions）を設定する**。戦術連鎖のステップ構造を参考に、各中間命題がどの命題群から導出されるかを定義する
3. `_phase3.generated_descriptors` に記録する
4. 中間命題がピースのメンバーに追加されるべきかを検討する

### Step 5: 導出条件の確定

Phase 2 で暫定設定した導出条件を、中間命題の導入を踏まえて確定する。

- **暫定 fc の調整**: Phase 2 の暫定的な導出条件が、中間命題によりより適切に表現できる場合は調整する
- **T 命題の fc 確定**: T 命題（ピースメンバー）の導出条件をここで確定する。戦術連鎖のステップ構造に基づき、各 T 命題の導出元を定義する
- **横展開の充実**: 中間命題を活用して、既存命題の OR グループを追加する
- **想起スコープの確認**: 導出条件が想起スコープ制約（構造設計規則）を満たすことを確認する

### Step 6: 重複整理

異なる戦術連鎖が同じ命題群を共有するケースを確認し、重複を整理する。

重複が存在する場合:
- 共有される命題群を整理し、不要な重複を排除する
- 質問生成（Phase 4）で同一の質問が複数の連鎖に対応する形で処理できるよう記録する

### Step 7: 保留の確認

保留状態の連鎖を確認し、最終的な判断を行う。

- 自然な連鎖が見つからなかった連鎖を列挙する
- 他の連鎖との合流が可能か再検討する
- 不要と判断されれば破棄する
- 依然として解決できない場合は保留のまま記録し、Phase 5（検証）で合理的なパスが存在するか確認する

### Step 8: 出力の検証

- 全ての連鎖が命題 ID の連鎖で構成されていること（非形式テキストが残っていないこと）
- Phase 3 で導入された中間命題が `_phase3.generated_descriptors` に記録されていること
- 中間命題が T 忠実性・独立性・客観性を満たすこと
- **全命題の導出条件が確定していること（暫定のまま残っていないこと）**
- **各導出条件が 1 段階推論の原則を満たすこと**

## 設計方針

### 主軸の設定

ピース内部の戦術連鎖では、トリガーから次のピースのトリガー（とその導出に必要な命題）への到達パスを**主軸**とする。

主軸は次のピースへの遷移を駆動する最重要パスであり、設計の優先度が最も高い。

### 独立鎖の扱い

主軸に直接関与しない命題は**独立鎖**とし、深さを抑える。

### 具体化パターン

抽象度の異なる命題ペア（例: 「乗り物」vs「熱気球」）が存在する場合、抽象版 + 具体化命題の組み合わせで導出する。

### 複合的な T 命題の分解

T の命題が複数の情報を含む場合、導出仮説として分解する。

## 失敗パターンと対策

### 誤り: 中間命題を導入しない連鎖

- 既存命題間に戦術ラベルを貼るだけで、中間命題を導入しなかった
- 連鎖の中間結果を非形式テキストで記述し、正式な命題にしなかった
- **対策**: 連鎖の各ステップの出力は命題 ID であること。テキスト記述が出てきたら、形式化すべき命題の候補として扱う

### 誤り: 領域命題の無視

- Phase 2 で定義した領域命題を無視し、トリガーから直接 T メンバーへ至る連鎖を提案した
- **対策**: Phase 2 の成果物（領域命題・生成命題）を Phase 3 で必ず参照する

### 誤り: 不自然な連想（結論から逆算した設計）

- 結論から逆算して「この導出が必要だから作った」連鎖を設計した
- **対策**: 全ての連鎖に対して「プレイヤーの自然な思考か」を一貫して検証する

### 誤り: 段階的絞り込みの具体例省略

- 段階的絞り込みを戦術として使う際、中間命題の具体例を挙げずに一足飛びに書いた
- **対策**: 段階的絞り込みを使う場合、少なくとも 1 つの具体的な中間命題を正式に導入する

### 誤り: 安易な保留判断

- 検討もせずに保留を提案した
- **対策**: 保留は複数の戦術を検討した上での判断であり、検討過程を記録する

### 誤り: 戦術選択の誤り（why vs how）

- 「どうやって死んだか」（how）に因果の逆転（why）を適用した
- **対策**: 「なぜ」（原因）と「どうやって」（手段）を区別する

## 保留判断

### 保留の条件

- 複数の汎用戦術を具体的に検討し、全て不自然と判断した場合
- 検討過程（どの戦術を試し、なぜ不自然と判断したか）を履歴に記録する
- 検討なしの保留は禁止

### 保留の処理

- Step 7（保留の確認）で最終判断を行う
- 保留のまま残った連鎖がある場合、Phase 5（検証）で他の経路を通じた到達性を確認する

## プレイヤー視点の検証

全ての戦術連鎖に対して、以下の観点で検証する:

1. **入力からの連想は自然か**: プレイヤーがその入力情報を持っている時、その方向に考えることが自然か
2. **結論から逆算していないか**: 「この導出が必要だから作った」のではなく、プレイヤーの思考の流れとして成立するか
3. **戦術の適用が適切か**: why/how/what/who/when の区別が正しいか、入れ子の構造が自然か
4. **跳躍が大きすぎないか**: 入力と出力の間に埋めるべき中間命題がないか

検証は全ての連鎖に一貫して適用する。

## 戦術の適用単位と組み合わせ

### 命題集合への適用

戦術は命題集合に対して適用される。単一命題への適用は、要素数 1 の集合への適用にすぎない。

### 入れ子構造（戦術の階層的組み合わせ）

戦術は入れ子に組み合わせられる。上位の方針として一つの戦術を取り、その実行過程のサブ戦術として別の戦術を使う構造が可能である。

## 質問生成との紐付け

戦術連鎖の各戦術ステップは、Phase 4（質問生成）で 1 つ以上の具体的な質問に展開される（1:多の対応）。この対応関係を明確に保つために:

- 各戦術ステップは、その戦術が「何を入力として」「何を導くか」を明示する
- 戦術の出力命題は、対応する質問の reveals の基礎となる
- 質問の recall_conditions は reveals される命題の導出条件から機械的に導出される

## 規則

- 全てのトリガー命題に戦術連鎖が定義されること（保留を除く）
- 全てのピースにピース内部の戦術連鎖が定義されること
- 使用する戦術は汎用戦術リストに含まれること
- **各戦術ステップの出力は命題 ID であること（非形式テキストは不可）**
- **中間命題を導入しない連鎖出力は不完全であり、再検討すること**（量的最低基準: 各連鎖に最低 1 つ）
- ピース内部の戦術連鎖で、全構成命題がいずれかのステップの出力に含まれること
- 戦術の入力は命題集合であること（単一命題に限定しないこと）
- 段階的絞り込みを使う場合、少なくとも 1 つの具体的な中間命題を正式に導入すること
- 全ての連鎖についてプレイヤー視点の検証を実施すること
- 保留は複数の戦術を検討した上での判断であり、検討過程を記録すること
- S の命題は独立性と客観性を満たすこと
- S と T の命題間に重複がないこと
- **Phase 3 で導入した中間命題は `_phase3.generated_descriptors` に記録すること**
- **中間命題は T 忠実性・独立性・客観性を満たすこと**
- **全命題の導出条件が Phase 3 完了時に確定していること**
- **各導出条件は想起スコープ制約を満たすこと**
