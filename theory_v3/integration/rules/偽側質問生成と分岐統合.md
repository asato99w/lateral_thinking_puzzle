# 偽側質問生成と分岐統合

## 概要

統合アルゴリズム v3 Phase 7 の規則。Phase 6 の偽側戦術連鎖に沿って質問を生成し、既存連鎖との分岐・合流を設計する。

Phase 4（真側の質問生成）の偽側版 + 分岐統合。偽側の質問は Phase 4 と同じ品質基準を満たす。

## 入力

- Phase 1〜6 の全出力

## 出力

- 偽側の質問群（recall_conditions, reveals 付き）
- 既存連鎖の内部分岐の記録
- 合流点と fc の OR 構造の更新

## 出力データ構造

```json
{
  "_phase7": {
    "internal_branches": [
      {
        "source_question": "Q1b",
        "branch_answer": "いいえ",
        "leads_to": ["QF1"],
        "description": "P1 確認質問が「いいえ」の場合の展開"
      }
    ],
    "convergence_points": [
      {
        "proposition_id": "P2",
        "new_fc_group": ["NF1", "S5"],
        "source": "偽側からの合流"
      }
    ]
  },
  "questions": [
    {
      "id": "QF1",
      "text": "偽側の質問文",
      "answer": "はい/いいえ",
      "recall_conditions": [["N1a", "S5"]],
      "reveals": ["NF1"],
      "mechanism": "link",
      "prerequisites": ["N1a"]
    }
  ]
}
```

偽側の質問は Phase 4 で生成した `questions` 配列に追加する。合流により更新された formation_conditions は `propositions` 配列を直接更新する。

### フィールド定義

#### internal_branches

| フィールド | 型 | 説明 |
|---|---|---|
| source_question | string | 分岐元の質問 ID |
| branch_answer | string | 分岐を発生させる回答（「はい」or「いいえ」） |
| leads_to | list[string] | 分岐先の質問 ID リスト |
| description | string | 分岐の説明 |

#### convergence_points

| フィールド | 型 | 説明 |
|---|---|---|
| proposition_id | string | 合流先の命題 ID |
| new_fc_group | list[string] | 追加される代替 AND グループ |
| source | string | 合流の由来の説明 |

#### 偽側の質問（questions に追加）

Question 型（`models.py` 参照）。全フィールドは Phase 4 の質問と同一:
- `id`: 質問 ID（QF* 形式）
- `text`: 質問文
- `answer`: 「はい」or「いいえ」
- `recall_conditions`: `list[list[str]]`（OR of AND）
- `reveals`: `list[str]`
- `mechanism`: `"observation"` | `"link"` | `"anomaly"`
- `prerequisites`: `list[str]`

## 手順

### Step 1: 偽側連鎖の質問展開

Phase 6 で構築した偽側戦術連鎖の各ステップを質問に展開する。

質問の種類:
- **排除質問**（回答: いいえ）: 偽側の仮説を排除し、方向を示す。否定文を中間命題として reveals に含める
- **確認質問**（回答: はい）: 偽側の中間命題を確認する。reveals に命題を含む

展開の方針:
- Phase 4 と同じ基準で生成する
- 各質問に recall_conditions を設定する（reveals される命題の formation_conditions と一致）
- 各質問に reveals を設定する（全質問が reveals を持つこと）
- 導入時検証（1 次チェック）を適用する

### Step 2: 既存連鎖の内部分岐

Phase 3〜4 で構築した既存連鎖の各ステップで、反対の回答に進む枝を設計する。

- 既存のノードから分岐するため、新たな連鎖の構築は不要
- 各質問の回答が反対だった場合の展開を記録する
- 浅い設計で十分。全ステップの完全な分岐設計は不要

### Step 3: 合流の検討と fc OR 構造

偽側の枝が既存命題に合流する構造を検討する。

- 合流が生じれば、その命題の formation_conditions に代替 AND グループ（OR 構造）が生まれる
- 該当する命題の formation_conditions を更新する
- 合流は自然に生じる場合に取り入れる。強制しない

合流の判例は `precedents/戦術連鎖の構造の判例.md` を参照。

## 質問の導入時検証（1次チェック）

### 回答の T 文脈での真偽の明確さ

- 各質問の回答（はい/いいえ）が T の事実と矛盾しないことを確認する
- 質問文の内容が T の事実と照合して、回答が一意に定まることを確認する

### reveals と回答の整合性

- 「はい」の回答から reveals される命題が、質問文の肯定から厳密に導かれることを確認する
- 「いいえ」の回答から reveals される命題が、質問文の否定から厳密に導かれることを確認する

### 意味の確定性

- 質問文が、その質問の直接の前提命題と合わせて読んだとき、意味が一意に確定することを確認する

## 規則

- 偽側の質問は Phase 4 と同じ品質基準を満たすこと
- **recall_conditions = reveals される命題の formation_conditions**
- 全質問が reveals を持つこと（`reveals` は `list[str]` 型）
- 既存連鎖の内部分岐は浅く設計すること
- 合流は自然に生じる場合に取り入れること。強制しない
- 合流により生じる fc の OR 構造を propositions に反映すること
- prerequisites に設定する命題は initial_confirmed またはいずれかの質問の reveals に含まれていること
- 質問は客観的に回答可能であること
