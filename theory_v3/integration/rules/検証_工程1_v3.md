# 検証（工程1: 真側）v3

## 概要

工程1（真側の構築）の末尾で実施する検証。Phase 1〜5 の出力を対象とし、真側の構造が正しく構築されていることを確認する。棄却側（NF/QF）は工程3で構築されるため、本検証の対象外である。

v2 からの変更:
- Step 7.5（質問文の含意検証）を追加
- Step 2 に fc の N*/NF* 不在チェックを追加

**本検証は 2 次チェック（安全網）として機能する。** 命題の品質保証は、各命題が導入される Phase での 1 次チェック（`rules/質問生成_v4.md` の「質問・回答の導入時検証」等）が本来の防御線である。

## 入力

- Phase 1〜5 の全出力

## 出力

- 検証結果（PASS/FAIL）
- 各パスのトレース
- FAIL 時の差し戻し先

## 手順

### Step 1: 構造的整合性

- 全命題（P*, N*, FP）の ID が一意であること
- 全質問（Q*）の ID が一意であること
- 全質問の reveals が存在する命題 ID を参照していること
- formation_conditions / negation_of が存在する命題 ID を参照していること
- initial_confirmed の命題が全て `_phase4.s_propositions` に定義されていること

### Step 2: 利用可能条件の到達可能性

全質問の利用可能条件が初期確認済み命題から到達可能であることを検証する。

- 利用可能条件は reveals 先命題の fc（「いいえ」回答の場合は negation_of 先命題の fc）
- fc に含まれる S 命題が initial_confirmed に含まれること
- fc に含まれる P/I 命題が、先行する「はい」質問の reveals に含まれること
- **fc に N*/NF* が含まれていないこと**（YES 進行原則）
- 循環依存がないこと

### Step 3: 否定関係の整合性

- 「いいえ」回答の質問が reveal する命題に negation_of が設定されていること
- negation_of が指す命題が存在し、formation_conditions を持つこと
- negation_of 関係が対称的であること（P.negation_of = Q ⇒ Q.negation_of = P）

### Step 4: reveals の網羅性と完全性

- 全分割命題（P*）が少なくとも 1 つの質問の reveals に含まれること
- **全質問が reveals を持つこと**（reveals: [] は原則あり得ない）

### Step 5: T 整合性と命題品質（2次チェック）

#### 回答の T 整合性

- 全質問の回答（はい/いいえ）が T の事実と矛盾しないこと
- 分割命題の truth_value が T の内容と整合すること

#### 全命題の T 文脈での真偽の明確さ

- 全ての分割命題・中間命題について、T の事実から真偽が一意に定まることを確認する

#### 全命題の意味の確定性

- 各命題が、その直接の前提命題と合わせて読んだとき、意味が一意に確定することを確認する

### Step 6: reveals の厳密性

各質問について、reveals に含まれる命題が質問テキストと回答（Yes/No）から厳密に論理的に導かれるかを確認する。

### Step 7: prerequisites の対話上確立可能性

各質問について、prerequisites に設定された命題が以下のいずれかに含まれているかを確認する:
- initial_confirmed
- いずれかの質問の reveals

formation_conditions でのみ導出される命題が prerequisites に含まれている場合は FAIL。

### Step 7.5: 質問文の含意検証

各質問について、質問文が前提とする情報が利用可能時点で保証されているかを検証する。

判定基準と具体例は `precedents/質問文の含意の判例.md` に従う。各質問を判例の NG/OK パターンと照合し、同種の問題がないかを確認すること。

### Step 8: 導出条件の 1 段階推論原則

全命題の formation_conditions について、各 AND グループが単一の推論ステップとして妥当であるかを確認する。

### Step 8.5: 連鎖構造と fc の整合性

全連鎖（`_phase4.tactical_chains`）について、以下を検証する。

#### 8.5a: fc の直前ステップ参照

target および intermediates の fc が連鎖内の直前ステップを参照しているかを検証する。

- 連鎖 source → I1 → ... → In → target において:
  - target.fc が In を参照していること
  - 各 Ik.fc が Ik-1（k=1 の場合は source）を参照していること
- target.fc が source のみを参照し intermediates をスキップしている場合は FAIL

#### 8.5b: 戦術ステップ数と中間命題数の整合性

各連鎖の中間命題数が戦術ステップ数以上であることを検証する。

- 連鎖の tactic フィールドに含まれる戦術ステップ数を N とする（「→」で区切られた個数）
- intermediates の要素数が N 未満の場合は FAIL
- 理由: 各戦術ステップは新たな中間命題を生成すべきであり、命題を生成しない戦術ラベルは実質的に機能していない

検証: `check_chain_consistency.py`

### Step 9: パストレース

複数の質問選択順でゲームをシミュレーションする。

1. confirmed = initial_confirmed から開始
2. 仮説導出（formation_conditions 1回パス）→ derived 計算
3. known = confirmed ∪ derived
4. 利用可能条件が known で満たされる質問を列挙（オープン質問）
5. オープン質問から 1 つ選択 → reveals を confirmed に追加
6. 2〜5 を繰り返し、全分割命題が confirmed になるまで

確認事項:
- **到達可能性**: いかなる選択順でも全分割命題が confirmed になること
- **行き詰まり**: オープン質問がなくなり、かつ分割命題が未確認の状態が発生しないこと
- **想起の合理性**: 各質問の開放が唐突な飛躍を含まないこと

最低 3 パスを異なる入口から検証する。

## 検証条件

全ステップが PASS であること。
