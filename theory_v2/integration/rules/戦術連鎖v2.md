# 戦術連鎖

## 概要

統合アルゴリズム v2 Phase 3 の規則。S の記述素を抽出し、戦術連鎖の分析を通じて必要な**中間記述素**を導入する。

**本フェーズの意義は適切な中間記述素の導入に尽きる。** 戦術連鎖は「どこに記述素が必要か」を発見するための分析手法であり、出力は戦術ラベルではなく記述素群である。戦術ラベルを既存記述素間に貼るだけでは本フェーズの目的を果たしていない。

このフェーズで初めて S が登場する。

v1 からの主要な変更:
- S の記述素抽出を Phase 1 から移動
- **中間記述素の導入をフェーズの主目的として明文化**
- 設計方針（主軸・独立鎖・具体化パターン）の明文化
- 保留判断の明示的組み込み
- 末端チェーンの重複整理ステップの追加
- プレイヤー視点の検証の義務化
- 失敗パターンの具体例による規則の補強

## 入力

- S: 問題文
- 末端記述素群（Phase 2 の出力）
- 各ピースのトリガー族と構成記述素（Phase 2 の出力）
- 汎用戦術リスト（`structure/汎用戦術.md`）

## 出力

- S の記述素群
- **Phase 3 で新たに導入された中間記述素群**（`_phase3.generated_descriptors` に記録）
- 各末端記述素に対する戦術連鎖（中間記述素を含む記述素の連鎖）
- 各ピースに対するピース内部の戦術連鎖（中間記述素を含む記述素の連鎖）

## 中間記述素の導入原則

### 本フェーズの核心

戦術連鎖の分析過程で、起点と終点の間のギャップを埋める中間記述素を発見し、形式的な記述素として導入する。

連鎖の各ステップの出力は**記述素**でなければならない。非形式的なテキスト記述（「銃を向けたことが有益だった」等）で終わらせてはならない。テキスト記述が出てきた場合、それは形式化すべき記述素の候補である。

### 中間記述素が必要な場合

- **段階的絞り込み**: 起点と終点の抽象度の差が大きい場合、中間の抽象度の記述素を導入する
- **具体化パターン**: 抽象版と具体版の記述素ペアが必要な場合（例: 「乗り物」→「熱気球」）
- **複合記述素の分解**: T の記述素が複数情報を含む場合、構成要素を中間記述素として導入する
- **矛盾の解消**: 矛盾の指摘の結果（解消された状態）が非自明な場合
- **因果の中間ステップ**: 因果の逆転や動機の探索で、直接的な接続が跳躍を含む場合

### 量的最低基準

各チェーン（末端チェーン・各ピース内部チェーン）は、**最低 1 つの中間記述素を導入しなければならない**。

- **1 つは最低評価である**: 1 つの中間記述素は「基準をかろうじて満たしている」状態であり、望ましい水準ではない。連鎖の複雑さに応じて 2 つ以上を目指すべきである
- **0 は不合格**: いかなる理由であれ、中間記述素が 0 個のチェーンは不完全であり、再検討が必要である

この基準は「不必要に増やさない」という判断が免罪符となり中間記述素を 0 個にする傾向を防ぐためのものである。中間記述素の数を減らす方向の判断は適切に行うことが困難であり、まず導入した上で適切性を改善する方が良い結果を生む。

### 導入後の適切性レビュー

中間記述素を導入した後、以下の観点でレビューする:

- **T 忠実性**: T の情報から逸脱していないか
- **プレイヤー視点の自然さ**: その中間ステップはプレイヤーの思考の流れとして自然か
- **質問への展開可能性**: その中間記述素から自然な質問を生成できるか

レビューで問題が見つかった場合は、中間記述素を**削除するのではなく、より適切な中間記述素に置き換える**。量を減らすのではなく質を上げる方向で対処する。

### 具体例: 砂漠の男 P-BAL

Phase 2 の出力: トリガー D-11「上空にいた」、メンバー D-1「数人で熱気球に乗っていた」、D-2「高度が下がり始めた」

Phase 3 で導入された中間記述素:
```
D-11「上空にいた」
  ├→ D-15「空を移動していた」         ← 段階的絞り込みの中間
  │    └→ D-34「熱気球に乗っていた」   ← 具体化
  ├→ D-16「一人ではなかった」          ← 登場人物の探索
  └→ D-1 (= D-34 + D-16 の導出)
       └→ D-32, D-33 → D-35 → D-17 → D-2  ← 因果の中間ステップ群
```

D-15, D-16, D-17, D-32, D-33, D-34, D-35 が全て Phase 3 で導入された中間記述素。これらの記述素が連鎖を「連鎖」にしている。

### 反例: 記述素を導入しない連鎖

```
{D-S3, D-S4} →[矛盾の指摘]→ 「銃を向けたことが有益だった」→ D-7
```

「銃を向けたことが有益だった」がテキスト記述のまま放置されている。中間記述素が導入されておらず、記述素の連鎖になっていない。

### 中間記述素の制約

- **T 忠実性**: T の記述素の抽象化・一般化、T から論理的に導出可能な中間ステップ、または T に明示的に含まれる情報の再表現に限定する。T に含まれない具体的事実を新たに創作してはならない（Phase 2 の生成記述素と同じ制約）
- **独立性と客観性**: Phase 1 の記述素と同じ品質基準を満たすこと
- **formation_conditions**: 中間記述素に導出条件が必要な場合は設定する。特に複数の記述素の組み合わせから導出される中間記述素は formation_conditions を持つ

## 戦術ステップの形式

各ステップは以下を明示する:

- 入力記述素集合（記述素 ID）
- 適用する汎用戦術
- 出力記述素（**記述素 ID**。非形式テキストは不可）

```
{記述素ID群} →[戦術]→ 記述素ID
```

## 手順

### Step 1: S の記述素の抽出

S を意味単位に分解し、S の記述素として列挙する。ID は D-S* を使用する。

- S と T の記述素間に重複がないこと
- 各記述素は独立性と客観性を満たすこと

### Step 2: 末端記述素への戦術連鎖

末端記述素に対して、起点記述素集合からその記述素に至るまでの**記述素の連鎖**を構成する。

```
記述素ID群 →[戦術1]→ 中間記述素ID →[戦術2]→ ...→ 末端記述素ID
```

起点はピースの種類により異なる:
- **独立ピース**: S の記述素集合
- **依存ピース**: S の記述素集合 + 依存先ピースの記述素集合（推移的）

記述素自体が起点の記述素集合に含まれる場合は、直接同定を適用する。

**中間記述素の導入**: 各ステップの出力が非形式テキストになった場合、それを正式な記述素として定義する（ID を付与し、descriptors に追加する）。

設計にあたっては**設計方針**（後述）に従い、**プレイヤー視点の検証**を各連鎖に適用する。

### Step 3: ピース内部の戦術連鎖

各ピースに対して、トリガーを起点としてピース内の記述素に到達する**記述素の連鎖**を構成する。

```
トリガーID(+既知記述素ID) →[戦術1]→ 中間記述素ID →[戦術2]→ メンバー記述素ID →...
```

- 前ステップの出力は次ステップの入力に利用可能（順序依存の連鎖）
- 全ての構成記述素がいずれかのステップの出力に含まれること
- **中間記述素の導入**: Phase 2 のメンバー記述素に直接到達できない場合、中間記述素を導入して連鎖を構成する

### Step 4: 中間記述素の形式化

Step 2-3 で導入した中間記述素を整理する:

1. 全ての中間記述素に ID を付与し、descriptors 配列に追加する
2. 必要に応じて formation_conditions を設定する（特に複数記述素の組み合わせから導出されるもの）
3. `_phase3.generated_descriptors` に記録する
4. 中間記述素がピースのメンバーに追加されるべきかを検討する

### Step 5: 末端チェーンの重複整理

異なる末端記述素への連鎖が同じ観測事実群を共有するケースを確認し、重複を整理する。

具体例（砂漠の男）:
- D-14「乗り物に乗っていた」への連鎖と D-27「地上の乗り物ではない」への連鎖が、同じ観測事実群（足跡がない、タイヤの跡がない等）を共有していた
- D-14 と D-27 は同じ「乗り物」に関する記述素であり、同じ観測事実から同時に導かれるのは自然

重複が存在する場合:
- 共有される観測事実群を整理し、不要な重複を排除する
- 質問生成（Phase 4）で同一の質問が複数の連鎖に対応する形で処理できるよう記録する

### Step 6: 保留の確認

保留状態の連鎖を確認し、最終的な判断を行う。

- 自然な連鎖が見つからなかった連鎖（`"status": "pending"`）を列挙する
- 他の連鎖との合流が可能か再検討する
- 不要と判断されれば破棄する
- 依然として解決できない場合は保留のまま記録し、Phase 5（検証）で合理的なパスが存在するか確認する

### Step 7: 出力の検証

- 全ての連鎖が記述素 ID の連鎖で構成されていること（非形式テキストが残っていないこと）
- Phase 3 で導入された中間記述素が `_phase3.generated_descriptors` に記録されていること
- 中間記述素が T 忠実性・独立性・客観性を満たすこと

## 設計方針

### 主軸の設定

ピース内部の戦術連鎖では、トリガーから次のピースのトリガー（とその導出仮説）への到達パスを**主軸**とする。

```
P-BAL トリガー → ... → D-2（P-WGT トリガー）
P-WGT トリガー → ... → D-5（P-LOT トリガーの導出に必要）
```

主軸は次のピースへの遷移を駆動する最重要パスであり、設計の優先度が最も高い。

### 独立鎖の扱い

主軸に直接関与しない記述素は**独立鎖**とし、深さを抑える。

具体例（砂漠の男）:
- D-16「一人ではなかった」は主軸（P-BAL → P-WGT 遷移）に直接関与しないため、D-11 からの独立鎖として深さ 1 で設計

### 具体化パターン

抽象度の異なる記述素ペア（例: 「乗り物」vs「熱気球」）が存在する場合、抽象版 + 具体化記述素の組み合わせで導出する。

具体例（砂漠の男）:
- D-2「熱気球の高度が下がり始めた」← D-17「乗り物の高度が低下した」+ D-34「熱気球に乗っていた」
- D-13「熱気球にトラブルが発生した」← D-35「乗り物にトラブルが発生した」+ D-34「熱気球に乗っていた」
- D-34 が乗り物→熱気球の具体化に繰り返し使われる構造

### 複合的な T 記述素の分解

T の記述素が複数の情報を含む場合、導出仮説として分解する。

具体例（砂漠の男）:
- D-1「数人で熱気球に乗っていた」← D-34「熱気球に乗っていた」+ D-16「一人ではなかった」
- D-1 を直接導出するのではなく、構成要素に分解して組み合わせる

## 失敗パターンと対策

### 誤り: 中間記述素を導入しない連鎖

- 既存記述素間に戦術ラベルを貼るだけで、中間記述素を導入しなかった
- 連鎖の中間結果を非形式テキスト（「銃を向けたことが有益だった」等）で記述し、正式な記述素にしなかった
- **対策**: 連鎖の各ステップの出力は記述素 ID であること。テキスト記述が出てきたら、形式化すべき記述素の候補として扱う

### 誤り: 領域記述素の無視

- P-BAL の戦術連鎖を提案した際、Phase 2 で定義した領域記述素（D-15「空を移動していた」, D-16「一人ではなかった」, D-17「高度が低下した」）を無視し、トリガー D-11 から直接 T メンバー D-1, D-2 へ至る連鎖を提案した
- 誤った提案: `D-11 →[段階的絞り込み]→ D-1` `D-1 →[観測事実の深掘り]→ D-2`
- 正しくは中間記述素（D-15, D-16, D-17）を経由すべき
- **対策**: Phase 2 の成果物（領域記述素・中間記述素）を Phase 3 で必ず参照する。データの `piece_domain` や生成記述素を明示的に確認してから連鎖設計に入る

### 誤り: 不自然な連想（結論から逆算した設計）

- {D-1「数人で熱気球に乗っていた」} →[時間軸の拡張]→ D-32「死ぬ間際に男は焦っていた」を「素直な適用」と判断した
- 実際の問題:
  1. 連想が不自然: 「熱気球に乗っていた」からプレイヤーが「死ぬ間際に焦っていたか」を問う動機がない
  2. 結論から逆算した設計: D-32 + D-33 → D-35 の導出に必要だから作っただけで、プレイヤーの自然な思考の流れではない
  3. 客観性の問題: 「焦っていた」は内面の状態であり、客観的に確認しにくい
- **対策**: 全ての連鎖に対して「プレイヤーの自然な思考か」を一貫して検証する（**プレイヤー視点の検証**を参照）

### 誤り: 段階的絞り込みの具体例省略

- 段階的絞り込みを戦術として使う際、中間記述素の具体例を挙げずに「段階的絞り込み → 終点」と一足飛びに書いた
- 段階的絞り込みの本質は中間ステップの存在であり、少なくとも1つの具体的な中間記述素を示さなければ戦術として機能していない
- **対策**: 段階的絞り込みを使う場合、少なくとも 1 つの具体的な中間記述素を正式に導入する

### 誤り: 安易な保留判断

- D-33 の循環問題を指摘された際、検討もせずに「保留が妥当でしょうか？」と提案した
- 保留は十分に検討した上で自然な連鎖が見つからない場合の判断であり、考えてもいないものを保留とすべきではない
- **対策**: 保留は複数の内部戦術を具体的に検討し、全て不自然と判断した場合のみ。検討過程を履歴に記録する

### 誤り: 戦術選択の誤り（why vs how）

- {D-S4「死んでいた」} →[因果の逆転]→ D-28「転落死だった」と設計した
- 「どうやって死んだか」は how であり、因果の逆転（why）ではなく手段の探索（how）が適切
- 再設計:
  ```
  {D-S4「死んでいた」}
    → [手段の探索 > 常識的仮説の排除]
      → D-X「脱水や熱中症で死んだのではない」
    → [手段の探索 > 段階的絞り込み]
      → D-Y「外傷で死んだ」
    → [手段の探索 > 段階的絞り込み]
      → D-28「転落死だった」
  ```
  D-X, D-Y が中間記述素として導入される。
- **対策**: 「なぜ」（原因）と「どうやって」（手段）を区別する。状態や結果の実現方法を問う場合は手段の探索を使う

## 保留判断

戦術連鎖の設計中、自然な連鎖が見つからない場合に無理に不適切な戦術を当てはめるべきではない。

### 保留の条件

- 複数の汎用戦術を具体的に検討し、全て不自然と判断した場合
- 検討過程（どの戦術を試し、なぜ不自然と判断したか）を履歴に記録する
- 検討なしの保留は禁止

### 保留の処理

- data_src.json 上で `"status": "pending"` マーカーを付与する
- Step 6（保留の確認）で最終判断を行う
- 保留のまま残った連鎖がある場合、Phase 5（検証）で他の経路を通じた到達性を確認する

### 保留が有効なケース

具体例（砂漠の男）:
- D-32「死ぬ間際に男は焦っていた」への連鎖を設計する際、起点 D-28「転落死だった」から:
  - 常識的仮説の排除「自殺ではなかった」→ 有効だが「焦っていた」への接続が不自然
  - 因果の逆転、観測事実の深掘り → いずれも「焦っていた」への接続が不自然
- 複数の戦術を検討した上で保留とした。他の経路との合流や、不要として破棄される可能性を残す

## プレイヤー視点の検証

全ての戦術連鎖に対して、以下の観点で検証する:

1. **入力からの連想は自然か**: プレイヤーがその入力情報を持っている時、その方向に考えることが自然か
2. **結論から逆算していないか**: 「この導出が必要だから作った」のではなく、プレイヤーの思考の流れとして成立するか
3. **戦術の適用が適切か**: why/how/what/who/when の区別が正しいか、入れ子の構造が自然か
4. **跳躍が大きすぎないか**: 入力と出力の間に埋めるべき中間記述素がないか

検証は全ての連鎖に一貫して適用する。一部の連鎖でのみ批判的視点を適用し、他では省略するという不均一な検証は、見逃しの原因となる。

## 戦術の適用単位と組み合わせ

### 記述素集合への適用

戦術は記述素集合に対して適用される。単一記述素への適用は、要素数 1 の集合への適用にすぎない。

入力が複数記述素の場合、それらの組み合わせから認知的経路が生まれる。単一記述素では見えない方向が、集合として見ると自然に見えることがある。

```
{D-1, D-2} →[常識的仮説の排除]→ D-19
```

### 入れ子構造（戦術の階層的組み合わせ）

戦術は入れ子に組み合わせられる。上位の方針として一つの戦術を取り、その実行過程のサブ戦術として別の戦術を使う構造が可能である。

```
手段の探索（到着方法について）:
  └─ 常識的仮説の排除:
       └─ D-X「歩いてここに来たのではない」
  └─ 段階的絞り込み:
       └─ D-14「乗り物に乗っていた」
```

上位戦術が「問いの枠」を設定し、サブ戦術がその枠内での探索方法を決める。入れ子構造の各段階でも中間記述素を導入する。

## 質問生成との紐付け

戦術連鎖の各戦術ステップは、Phase 4（質問生成）で 1 つ以上の具体的な質問に展開される（1:多の対応）。この対応関係を明確に保つために:

- 各戦術ステップは、その戦術が「何を入力として」「何を導くか」を明示する
- 戦術の入力記述素集合は、対応する質問の recall_conditions の基礎となる
- 戦術の出力記述素（中間記述素を含む）は、対応する質問の reveals の基礎となる

## 規則

- 全ての末端記述素に戦術連鎖が定義されること（保留を除く）
- 全てのピースにピース内部の戦術連鎖が定義されること
- 使用する戦術は汎用戦術リストに含まれること
- **各戦術ステップの出力は記述素 ID であること（非形式テキストは不可）**
- **中間記述素を導入しない連鎖出力は不完全であり、再検討すること**（量的最低基準: 各チェーンに最低 1 つ）
- ピース内部の戦術連鎖で、全構成記述素がいずれかのステップの出力に含まれること
- 戦術の入力は記述素集合であること（単一記述素に限定しないこと）
- 段階的絞り込みを使う場合、少なくとも 1 つの具体的な中間記述素を正式に導入すること
- 全ての連鎖についてプレイヤー視点の検証を実施すること
- 保留は複数の戦術を検討した上での判断であり、検討過程を記録すること
- S の記述素は独立性と客観性を満たすこと
- S と T の記述素間に重複がないこと
- **Phase 3 で導入した中間記述素は `_phase3.generated_descriptors` に記録すること**
- **中間記述素は T 忠実性・独立性・客観性を満たすこと**
