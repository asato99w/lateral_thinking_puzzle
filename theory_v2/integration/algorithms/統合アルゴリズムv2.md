# 統合アルゴリズム v2

## 概要

T からの逆算連鎖を起点に、4 構成要素（ピース・トリガー・仮説連鎖・戦術連鎖）を段階的に構築し、その骨格に基づいて質問を生成するアルゴリズム。工程1（コア生成）、工程2（調整）、工程3（仕掛けと分類）の 3 工程から成る。

### v1 からの主要な変更

- Phase 2（ピース構成）と Phase 3（仮説連鎖）を「構造設計」として統合（トリガーと導出条件は不可分）
- S の記述素抽出を Phase 1 から Phase 3（戦術連鎖）に移動
- 仮説導出の原則と意味検証を構造設計に組み込み
- 戦術連鎖の目的を「中間記述素の導入」に再定義（戦術は発見手法、出力は記述素群）
- 戦術連鎖に設計方針・保留判断・重複整理・プレイヤー視点検証を明文化
- 質問の前提条件（prerequisites）を質問生成に導入
- clear_conditions の確定を質問生成に移動（構造設計では意味的設計のみ）
- 検証に意味検証・eval スクリプト活用を追加

## 入力

- S: 問題文
- T: 真相

## 出力

- data.json: パズルデータ（記述素・ピース・トリガー・仮説連鎖・戦術連鎖・質問の全定義）

## 大域目標

**複数の質問選択順において合理的なパスで T に到達できること**

合理的なパスとは、各質問のオープンが想起（確認済み記述素または導出記述素による）により説明でき、唐突な飛躍がないパスを指す。

## 工程構成

```
工程1: コア生成（Phase 1〜5）
  → 構造的な到達性を優先し、パズルデータの骨格を構築する
工程2: 調整（未設計）
  → プレイヤー視点の認知的自然さを検査し、骨格を調整する
工程3: 仕掛けと分類（未設計）
  → 認知的断層を分析し、対立仮説と棄却メカニズムを構成し、質問を分類する
```

各工程は前の工程の出力を入力とする。後の工程は前の工程の出力に対して追記のみ行い、既存の構造を変更しない。

## S/T 分離原則

Phase 1〜2 は **T の構造のみ**を扱う。S が登場するのは Phase 3（戦術連鎖）以降である。

- Phase 1: T の因果構造を分解する（S は扱わない）
- Phase 2: T の記述素からピースを構成し、トリガー族・依存関係・導出条件を設定する
- Phase 3: ここで初めて S の記述素集合を抽出し、戦術連鎖を構成する

v1 では Phase 1 で S の記述素も抽出していたが、S の記述素は S → T への接続を設計する時点で初めて必要であり、Phase 1 での抽出は不要である。

## 工程1: コア生成

### Phase 1: 逆算連鎖

規則「逆算連鎖」を適用し、T から因果の逆算連鎖を構成する。T の記述素を統一的な ID（D-*）で抽出する。S は扱わない。

### Phase 2: 構造設計

規則「構造設計」を適用し、以下を一体的に設計する:

1. ピース群の構成（キーピース特定 → 支持ピース設計 → 記述素の再構成）
2. トリガー族の設計と依存関係の導出
3. 導出条件（formation_conditions）の設定と意味検証
4. clear_conditions の意味的設計（暫定版）

v1 ではピース構成（Phase 2）と仮説連鎖（Phase 3）を分離していたが、トリガー設計には中間記述素の生成とその導出条件の設定が不可分であるため、統合した。

### Phase 3: 戦術連鎖

規則「戦術連鎖」を適用し、**戦術連鎖の分析を通じて必要な中間記述素を導入する**。

戦術連鎖の主目的は中間記述素の導入である。戦術（矛盾の指摘・段階的絞り込み等）は「どこに記述素が必要か」を発見するための分析手法であり、出力は戦術ラベルではなく記述素群である。Phase 2 がトリガーレベルの抽象記述素を導入するのに対し、Phase 3 は連鎖内部の通過点（プレイヤーの推論ステップ）となる中間記述素を導入する。

1. S の記述素の抽出（ここで初めて S を扱う）
2. 末端記述素への戦術連鎖（S → 末端記述素）— 中間記述素の導入を含む
3. ピース内部の戦術連鎖（トリガー → ピースメンバー）— 中間記述素の導入を含む
4. 中間記述素の形式化（全中間ステップに正式な記述素 ID を付与）
5. 末端チェーンの重複整理
6. 保留の確認

### Phase 4: 質問生成

規則「質問生成」を適用し、戦術連鎖の各戦術ステップを 1 つ以上の具体的な質問に展開する。同時に:

- 前提条件（prerequisites）の設定
- clear_conditions の確定（revealed 記述素のみで構成）

### Phase 5: 検証

規則「検証」を適用し、大域目標の達成を検証する。FAIL の場合は原因フェーズに差し戻す。eval スクリプト（check_integrity.py 等）を活用する。

### フェーズ間の依存

```
Phase 1（逆算連鎖: T 分解）
  → Phase 2（構造設計: ピース + トリガー + 導出条件 + 意味検証）
    → Phase 3（戦術連鎖: S 抽出 + 中間記述素導入 + 連鎖構成）← ここで S が登場
      → Phase 4（質問生成: prerequisites + clear_conditions 確定）
        → Phase 5（検証: 意味検証 + eval 活用）
```

Phase 1〜2 が T 側の骨格を構成し、Phase 3 で S との接続を設計しつつ中間記述素を導入し、Phase 4 で質問を生成し、Phase 5 で検証する。

## サンプルの構成

```
app/poc_v2/samples/{puzzle}/
└── {nn}_{YYYY-MM-DD}/
    ├── data.json
    ├── data_src.json
    └── history/
        ├── 01_逆算連鎖.md
        ├── 02_構造設計.md
        ├── 03_戦術連鎖.md
        ├── 04_質問生成.md
        └── 05_検証.md
```

- `nn` は連番（同一パズルの反復）、`YYYY-MM-DD` は実施日
- data_src.json はアルゴリズムの各ステップに従って段階的に構築される中間データ
- data.json は data_src.json から `_` プレフィックスのメタ情報を除去したもの
- history/ には各ステップの実施記録を追記する
