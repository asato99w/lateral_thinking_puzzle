# 統合アルゴリズム

## 概要

T からの逆算連鎖を起点に、4 構成要素（ピース・トリガー・仮説連鎖・戦術連鎖）を段階的に構築し、その骨格に基づいて質問を生成するアルゴリズム。工程1（コア生成）と工程2（調整）の 2 工程から成る。

## 入力

- S: 問題文
- T: 真相

## 出力

- data.json: パズルデータ（記述素・ピース・トリガー・仮説連鎖・戦術連鎖・質問の全定義）

## 大域目標

**複数の質問選択順において合理的なパスで T に到達できること**

合理的なパスとは、各質問のオープンが想起（確認済み記述素または導出記述素による）により説明でき、唐突な飛躍がないパスを指す。

## 工程構成

```
工程1: コア生成（Phase 1〜6）
  → 構造的な到達性を優先し、パズルデータの骨格を構築する
工程2: 調整（Phase 7〜9）
  → プレイヤー視点の認知的自然さを検査し、骨格を調整する
```

工程2 は工程1 の出力を入力とし、構造を保ったまま認知的自然さの観点で調整する。工程2 で変更できる範囲は限定されており、工程1 で設計された reveals・記述素定義・ピース構成は変更しない。

## S/T 分離原則

Phase 1〜3 は **T の構造のみ**を扱う。S が登場するのは Phase 4（戦術連鎖）以降である。

- Phase 1: T の因果構造を分解する（S は記述素の抽出元としてのみ扱う）
- Phase 2: T の記述素からピースを構成し、トリガー族と依存関係を定義する
- Phase 3: トリガー記述素の横展開とピースメンバーの導出条件を設定する
- Phase 4: ここで初めて S の記述素集合を起点とした戦術連鎖を構成する

## 工程1: コア生成

### Phase 1: 逆算連鎖

規則「逆算連鎖」を適用し、T から因果の逆算連鎖を構成する。S と T の記述素を統一的な ID（D-*）で抽出する。

### Phase 2: ピース構成

規則「ピース構成」を適用し、キーピースの特定を起点にピース群を構成する。各ピースのトリガー族と依存関係を同時に定義する。必要に応じて新しい記述素を生成する。

### Phase 3: 仮説連鎖（導出条件設定）

規則「仮説連鎖（導出条件設定）」を適用し、トリガー記述素の横展開とピースメンバーの導出条件を設定する。

### Phase 4: 戦術連鎖

規則「戦術連鎖」を適用し、2 種類の戦術連鎖を構成する:

1. **末端記述素への戦術連鎖**: 起点記述素集合から末端記述素に至るパス。起点はピースの種類により異なる:
   - **独立ピース**: S の記述素集合
   - **依存ピース**: S の記述素集合 + 依存先ピースの記述素集合（推移的）
2. **ピース内部の戦術連鎖**: トリガーを起点としてピース内の記述素に至るパス

### Phase 5: 質問生成

規則「質問生成」を適用し、戦術連鎖の各戦術ステップを 1 つ以上の具体的な質問に展開する。各質問は対応する戦術ステップとの紐付けを維持する。

### Phase 6: 検証

規則「検証」を適用し、大域目標の達成を検証する。FAIL の場合は原因フェーズに差し戻す。

### フェーズ間の依存

```
Phase 1（逆算連鎖）
  → Phase 2（ピース構成）
    → Phase 3（仮説連鎖・導出条件設定）
      → Phase 4（戦術連鎖の構成）← ここで S が登場
        → Phase 5（質問生成）
          → Phase 6（検証）
```

Phase 1〜3 が T 側の骨格を構成し、Phase 4 で S との接続を設計し、Phase 5 で質問を生成し、Phase 6 で検証する。

## 工程2: 調整

### エージェント分離

工程2 では、生成と検査を**別エージェント**として実行する。同一エージェントが生成と評価を担うと、自己出力への擁護バイアスが生じ批判的評価が機能しない。

- **生成エージェント**: Phase 8（調整適用）を担当
- **検査エージェント**: Phase 7（検査）および Phase 8 後の再検査を担当

同じ LLM であっても、別エージェントとして呼び出すことで生成物を初見で評価する認知的距離が生まれる。

必須フロー:
```
検査エージェント: Phase 7 → 違和感リスト
生成エージェント: Phase 8 → 調整済み出力
検査エージェント: Phase 7 再適用 → 調整結果の検証
```

Phase 8 後の再検査を省略しないこと。

### Phase 7: 検査

規則「検査」を適用し、工程1 の出力をプレイヤー視点でシミュレーションして認知的な違和感を列挙する。違和感は 3 つのレイヤー（戦術レベル・戦術連鎖レベル・想起の認知的妥当性）に分類する。**検査エージェント**が担当する。

### Phase 8: 調整適用

規則「調整適用」を適用し、Phase 7 で特定した違和感に対して限定された範囲内で修正を適用する。**生成エージェント**が担当する。

### Phase 9: 調整検証

規則「調整検証」を適用し、変更禁止事項の遵守と調整後の大域目標達成を検証する。FAIL の場合は Phase 8 に戻る。

Phase 8 の完了後、**検査エージェント**が Phase 7 を再適用して調整結果を検証する。

## サンプルの構成

```
app/poc_v2/samples/{puzzle}/
└── {nn}_{YYYY-MM-DD}/
    ├── data.json
    ├── data_src.json
    └── history/
        ├── 01_逆算連鎖.md
        ├── 02_ピース構成.md
        ├── ...
```

- `nn` は連番（同一パズルの反復）、`YYYY-MM-DD` は実施日
- data_src.json はアルゴリズムの各ステップに従って段階的に構築される中間データ
- data.json は data_src.json から `_` プレフィックスのメタ情報を除去したもの
- history/ には各ステップの実施記録を追記する
