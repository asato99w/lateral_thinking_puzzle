# アプリ設計の核心

## 設計原則

プレイヤーの内的体験を逆算して設計する。

1. プレイヤーにどのような認知的体験をさせたいか
2. その体験が起こるにはどのような情報の流れが必要か
3. その流れを実現するためにアプリはどう振る舞うべきか

## 理論とプレイヤーの認知の対応

| 理論的概念 | プレイヤーの認知における位置づけ |
|-----------|---------------------------|
| 記述素空間 | 客観的な構造。パズルに内在し、アプリ側が設計するデータ |
| H | プレイヤーの認知状態。各記述素に対する確信度の分布 |
| パラダイムにいる | 認知バイアスが働いている状態。特定の関係構造を通じて全てを解釈している |
| パラダイムシフト | プレイヤーの内部で起こる認知的な出来事。アプリはこれを起こすのではなく、条件を整え、起きたことを支持する |

## ゲーム進行の流れ

1. プレイヤーが利用可能な質問から一つ選ぶ
2. 回答が返る（はい / いいえ / 関係ない）
3. O または R が更新される
4. アプリが O と R から H を推定する
5. 推定された H に基づいて、新たな質問のオープン判定を行う

## アプリの中核機能

### 1. H の推定

プレイヤーの行動から認知状態を推定する。

- **入力**: O（真偽が確定した記述素）と R（不関与と確定した記述素）
- **出力**: プレイヤーがどのパラダイムに集中しているかの推定
- アプリは H を直接観測できない。O と R の蓄積状態から推定する。

### 2. 質問のオープン判定

推定された H に基づいて、新たな質問をオープンするかを判定する。

- 判定基準は推定された H の状態のみ
- プレイヤーの内的なパラダイムシフトが先行して起こるよう、十分な「遅れ」を設ける
- 「遅れ」とは、O と R の蓄積からプレイヤーが内的にパラダイムシフトを起こしたであろうと十分に推定できるまでの期間

### 3. 体験の最適化

1 と 2 をプレイヤーの内的体験の質を最大化するように調整する。

- パラダイムシフトはプレイヤーの内部で先行して起こるべき
- 新たな質問のオープンは、プレイヤーの気づきを確認・支持するものとして機能する
- アプリが「導く」のではなく、プレイヤーが「発見する」体験を実現する

## 留意事項

### 遅れとフラストレーションのバランス

パラダイムシフトの先行体験を確保するための「遅れ」は、既にシフトを起こしたプレイヤーに詰まりを感じさせるリスクがある。遅すぎればフラストレーション、早すぎればアプリによる誘導になる。この問題はUI的な回避ではなく、理論的に洗練された構造によって解決すべきである。具体的な解決策は今後の検討課題。
